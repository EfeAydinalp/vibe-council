# Plan: v0.5 — execution payload bridge (design only)

A **design/safety document**, not an implementation. PR #74 added real, bounded execution for
`write_file` / `edit_file` via `execute_action(dry_run=False, payload=...)`, but the runtime `Action`
model (`backend/workbench_runtime.py`) has no durable place to store that payload — the caller must
supply `content` / `old_text` / `new_text` explicitly, every time, from outside the runtime store. That
was deliberate (see [bounded file executor decision](../decisions/2026-07-01-workbench-bounded-file-executor.md),
"payload is explicit"), but it means **the panel cannot execute a file action yet**: it has no durable
payload source to read from between "approve" and "execute". This plan designs that bridge before any
panel execute button or CLI `execute` command is built. **No panel wiring, no CLI execution command,
and no `run_command` execution are added in this task.**

Review date: **2026-07-02**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local.

---

## 1. Purpose

Define how an approved `write_file` / `edit_file` action's execution payload (file content or patch
text) is stored so a later caller — panel button or CLI command — can execute it without the payload
being passed in from an ephemeral, out-of-band source. The bridge must preserve every invariant the
executor already enforces (see [v0.5 guarded executor plan](./v0.5-guarded-executor.md) §4): approval
must be `approved`, the action must be `pending` and linked, the target/scope must match what was
approved, and a fresh trust re-check must pass — **and now also**: the payload actually executed must
be the same payload that was present (or hashed) at approval time.

## 2. Problem restated

`execute_action` today takes `payload: Optional[Dict]` as a caller-supplied argument
(`backend/workbench_executor.py:348`). That is safe for tests and any future CLI that reads a local
file argument, but it gives the **panel** nothing to execute against — the panel only has the runtime
`Action`/`ApprovalRequest` records, and those don't carry `content`/`old_text`/`new_text`. Two problems
follow if the payload were added to the `Action` model directly instead of a separate artifact:

- `Action` records are loaded, listed, and displayed broadly (panel task view, `runtime_status_summary`,
  future MCP/CLI reads); embedding potentially large or sensitive file content in every `Action` read
  widens the surface that has to be redaction-checked and enlarges every runtime read.
- The `Action.command_or_path` field already carries the *target*; overloading it (or adding untyped
  extra fields) to also carry a patch invites exactly the "ambiguous/unsafe string overloading" PR #74
  already rejected.

## 3. Decision: separate, local, gitignored payload artifact

Store execution payload in its own runtime JSON file, keyed by action id, under the same gitignored
`.council/runtime/` tree the rest of the Workbench state already lives in:

```
.council/runtime/payloads/<action_id>.json
```

This answers question 1 vs. 2 from the brief: **payload lives outside the `Action` record**, referenced
by `action_id`, not inside it. `Action` gains no new fields in this PR (or, if implemented, only an
optional `has_payload: bool` — see §9). This keeps every existing `Action` read (panel list, index,
`runtime_status_summary`) small and payload-free by default; a payload is only read when a caller is
about to execute that specific action.

### 3.1 Payload artifact shape

```jsonc
{
  "version": 1,
  "action_id": "act-...",
  "task_id": "task-...",
  "approval_id": "appr-...",
  "kind": "write_file",            // or "edit_file"
  "target": "docs/plans/x.md",     // must equal action.command_or_path
  "payload": {
    // write_file:
    "content": "...",
    "overwrite": false
    // edit_file:
    // "old_text": "...", "new_text": "...", "max_replacements": 1
  },
  "created_at": "2026-07-02T00:00:00Z",
  "payload_hash": "sha256:...",          // hash of the canonical payload dict (see §5)
  "approved_scope_hash": "sha256:...",   // hash of approval.scope at approval time, if present
  "redaction_summary": {                 // metadata only, never raw findings text with secrets
    "checked": true,
    "critical_findings": 0
  }
}
```

Same conservative field cleaning as the rest of the runtime store (`_clean`, `_safe_id`, no control
characters, bounded lengths) — this artifact reuses `backend/workbench_runtime.py`'s existing
helpers/conventions, not a parallel ad-hoc format.

## 4. Why local/gitignored, never committed

`.council/runtime/` is already gitignored and documented as **local, ephemeral runtime state, never
committed** (`backend/workbench_runtime.py` module docstring). Payload artifacts follow the same rule
without exception: they can contain full file content (write) or both sides of a diff (edit), which is
exactly the kind of raw material CLAUDE.md already forbids committing. `payloads/` is a subdirectory of
the existing gitignored root, so no new `.gitignore` entry is needed — but this plan calls it out
explicitly (question 3) so a future PR doesn't accidentally relocate it under `docs/` or otherwise widen
its reach.

## 5. Hashing: what, and when

Two independent hashes, both **sha256 of a canonical (sorted-key, UTF-8) JSON encoding**, matching the
stable-serialization style `workbench_runtime._dump` already uses for the rest of the store:

- **`payload_hash`** — hash of `{"kind", "target", "payload"}` (the fields that actually drive
  execution). Computed **when the payload artifact is written** (at or before approval — see §6), and
  re-computed by the executor immediately before real execution. If they don't match, execution is
  refused (see §7).
- **`approved_scope_hash`** — hash of `approval.scope` (when present) as it existed at approval time,
  stored alongside the payload so a later "did the approved scope change" check does not have to trust
  the live `ApprovalRequest` record alone.

Hashing answers question 6 directly: **hash the execution-relevant payload fields (kind/target/payload)
and the approved scope**, not the whole artifact (timestamps and redaction metadata are informational
and must not affect the hash, or every re-read would "tamper" it).

## 6. Immutability after approval

Question 4/5 (keep payload+scope immutable after approval; detect tampering):

- The payload artifact is written **once**, before or at the moment the linked `ApprovalRequest` moves
  to `approved`. After that point it is treated as **write-once**: no code path updates an existing
  `payloads/<action_id>.json` in place once its action has an `approved` approval linked to it.
- Because the store already writes atomically (temp-file-then-replace) and files are content-addressed
  by `payload_hash`, "modified after approval" is detectable without extra locking: at execution time,
  recompute `payload_hash` over the artifact's current `{kind, target, payload}` and compare to the
  stored `payload_hash`. A mismatch means the file was rewritten after approval (or corrupted) and
  execution **must refuse** — this is a new, explicit case in the execution invariant (§4 of the
  guarded-executor plan gains a bullet: "payload hash matches the hash recorded at/before approval").
- This composes with the existing "action target/scope matches the approved scope" check
  (`workbench_executor.validate_execution_invariant`) rather than replacing it: that check still
  compares `action.command_or_path` against the approval's `requested_action`; the new check compares
  the payload artifact's hash against the hash recorded when the artifact was created.

## 7. Executor re-check at execution time

Extends [v0.5 guarded executor plan](./v0.5-guarded-executor.md) §4/§5 rather than replacing it. Before
any real write/edit, in addition to the existing invariant, the executor must:

1. Load `.council/runtime/payloads/<action_id>.json`; refuse (`blocked`) if missing.
2. Verify `kind`/`target` in the artifact match the live `Action` (`kind`, `command_or_path`).
3. Recompute `payload_hash` over the artifact's `{kind, target, payload}` and compare to the stored
   value; refuse on mismatch ("payload modified after approval").
4. Re-run the deterministic trust boundary (unchanged from today) against the **actual** target from
   the artifact, not a caller-supplied one — this is what "trust boundary re-runs using actual
   target/payload metadata" (question 4 of the brief / this section) means concretely.
5. Only then pass `payload["payload"]` into the existing `_do_write` / `_do_edit` path.

An old `AuditResult` remains advisory-only, unchanged from the existing plan. This does not change
`execute_action`'s public shape in a breaking way: a `payload=` argument can remain accepted for tests
and direct callers, but panel/CLI callers would stop passing it directly and instead just pass
`action_id`, letting the executor load the artifact itself. (Left as an implementation choice for the
PR that actually wires this — see §11.)

## 8. What the panel displays before execution

Question 7. Before showing an "Execute" affordance, the panel must show, sourced from the payload
artifact (never guessed or re-derived):

- action kind and target path (`kind`, `target`);
- for `write_file`: whether it's a new file or an overwrite, and byte size (not raw content by
  default — content should be an opt-in "show diff/content" expansion, not shown unconditionally);
- for `edit_file`: `old_text` / `new_text` presented as a diff-style preview, with a size/line-delta
  summary;
- the trust re-check verdict (risk level, blocked/allowed) from a **fresh** evaluation, not a cached
  one;
- `payload_hash` (short form) so a user could, in principle, correlate what's about to run with what
  was approved.

This is a display requirement for the future panel PR, not something this PR builds.

## 9. Logging / redaction (question 8)

Never logged or written to any `result_summary`/log/audit trail:

- raw `content`, `old_text`, `new_text`;
- the full payload artifact.

Only logged: `action_id`, `kind`, `target`, `payload_hash` (safe — a hash, not content), byte/line
councounts, and pass/fail verdicts — consistent with the existing executor's `result_summary` style
(`backend/workbench_executor.py:_do_write` / `_do_edit` already log "wrote N bytes to <basename>", never
content). The redaction guard (`backend/redaction.py`) should be run over payload content **before** it
is written to the artifact (not after), so a payload that would trip a critical secret finding can be
refused at creation time rather than discovered at execution time; `redaction_summary` in §3.1 records
only the boolean/count outcome of that check, never the finding text itself.

## 10. Testing tamper scenarios (question 9)

A later implementation PR must cover, at minimum:

- payload artifact missing for an approved action → execution refuses;
- payload artifact present but `payload_hash` does not match its own `{kind, target, payload}` (content
  edited on disk after approval) → execution refuses;
- payload artifact `target`/`kind` disagrees with the live `Action` → execution refuses (this is the
  same "modified after approval" class the existing invariant already tests for the `Action` itself,
  extended to the payload);
- payload artifact for a *different* `action_id` referenced/copied onto another action's file name →
  refused (the artifact's own `action_id`/`approval_id`/`task_id` fields must match the ones being
  executed, not just the filename);
- payload containing a secret-pattern match is refused at write time (redaction check), not silently
  written;
- re-execution after a first successful execution (action no longer `pending`) still refuses, per the
  existing invariant — unchanged, just re-confirmed with a payload present.

## 11. Invariant preserved: approved scope == executed scope

Question 10. The chain of custody is: **approval.requested_action / approval.scope** (what a human
approved) → **payload artifact** (`target`, `kind`, `payload`, `approved_scope_hash`, `payload_hash`,
written at/before approval) → **live `Action`** (`command_or_path`, `kind`) → **executor** (re-derives
and re-checks all of the above at execution time, using the artifact's own values, not values passed in
by whichever caller happens to invoke execution). Because every step re-checks against the previous
step's recorded hash/value rather than trusting the caller, no caller (panel, CLI, or a bug) can widen
what actually executes beyond what was approved without tripping a `blocked` refusal.

## 12. Non-goals (this PR)

No panel execute button · no CLI `execute` command · no change to `execute_action`'s existing
`payload=` calling convention for direct/test callers · no `run_command` payload support (file kinds
only, matching the current executor) · no encryption of the payload artifact (it is local-only runtime
state, same trust level as the rest of `.council/runtime/`) · no network/provider/model calls.

## 13. Possible PR sequence

- **PR #75 (this plan).** Docs-only design for the payload bridge. No code.
- **PR #76 ✅ done.** `backend/workbench_payloads.py` — the payload artifact store:
  `canonical_payload_hash` / `build_payload_artifact` / `save_payload_artifact` (write-once by
  default) / `load_payload_artifact` / `verify_payload_artifact` / `verify_payload_against_action` /
  `summarize_payload_artifact`, written under `.council/runtime/payloads/<action_id>.json`.
  `backend/workbench_executor.py` loads and verifies the artifact when `execute_action(...,
  payload=None)` (the default) — hash + kind/target/approval/task agreement, checked **before** real
  `write_file`/`edit_file` execution and **in addition to** (never instead of) the fresh deterministic
  trust re-check. The explicit `payload=...` path from PR #74 is unchanged for direct/test callers.
  Full tamper-scenario tests (§10). Still no panel/CLI wiring.
- **PR #77.** Panel "Execute approved action" button reads the payload artifact (via PR #76's loader),
  shows the §8 preview, and calls the executor. Still no `run_command`.
- **PR #78 (optional).** Exact allowlisted `run_command` execution, only after another review — unrelated
  to the file-payload bridge and out of scope here.

(Matches the sequence in [v0.5 guarded executor plan](./v0.5-guarded-executor.md) §11, renumbered now
that PR #75 is the payload bridge rather than command execution — see that plan's "Recommendation" in
§3, which already deferred `run_command` past file execution.)

## 14. Stop conditions

Stop and request review if any of these arise while implementing PR #76/#77:

- pressure to store payload content directly on the `Action` record "for convenience";
- pressure to skip the payload-hash re-check at execution time ("the panel already showed it, so...");
- pressure to let the panel write and immediately execute a payload in one step (still violates
  approve/execute separation — see [guarded executor plan](./v0.5-guarded-executor.md) §8);
- any proposal to commit or sync payload artifacts off the local machine;
- scope creep into `run_command` payloads before file-kind payloads are proven.

## 15. Acceptance criteria for this plan

- Payload storage location and shape are specified (§3) and reuse existing runtime store conventions.
- Hashing scope and timing are specified (§5) and detect post-approval tampering (§6/§10).
- The executor re-check sequence is specified as an extension of the existing invariant, not a
  replacement (§7).
- Panel display and logging/redaction requirements are specified without requiring any panel/log code
  in this PR (§8/§9).
- The approved-scope-equals-executed-scope chain of custody is stated end-to-end (§11).
- No execution, panel, or CLI code changes in this PR.
