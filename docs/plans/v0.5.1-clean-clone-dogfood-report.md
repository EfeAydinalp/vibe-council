## 3. Issues found and docs fixes made

1. **README Quick Start's post-`uv sync` example used a bare `python -m backend.cli ...` command,
   which fails without an active `.venv`/`uv run` prefix** — reproduced directly (see §2b, bare-`python`
   test-suite failure is the same class of issue). **Fixed:** the example now uses
   `uv run python -m backend.cli ...`, with a one-line explanation of why, plus a note that the manual
   `venv`+`activate` path (shown just above it) doesn't need the prefix since that shell already has
   the venv active.
2. **The "Using the CLI directly (no global command)" section had the same bare-`python` examples**
   with no mention of `uv run`. **Fixed:** added a one-line pointer back to the Quick Start note.
3. **`vibe workbench serve --help`'s `help=`/`epilog=` text was stale** — it said "no action execution"
   / "executes nothing" unconditionally, which predates PR #77/#80/#81 adding real, explicitly-gated
   execution for bounded file actions and allowlisted commands. **Fixed** (`backend/cli.py`): the text
   now says approve/reject/hold never auto-execute, and that a verified file action or exact
   allowlisted command can be explicitly executed as a separate step. This is a help-string text change
   only — no argparse structure, executor, or panel behavior changed; `tests/test_cli_smoke.py` and
   `tests/test_cli_ux.py` (17 tests) still pass unchanged.
4. **The v0.5.1 plan's fresh-install checklist (`docs/plans/v0.5.1-dogfood-hardening.md` §3) didn't
   specify the venv interpreter for the test-suite step**, which is exactly what produced the bare-
   `python` failure in §2b. **Fixed:** the checklist item now names the `.venv` interpreter explicitly
   for both OSes and calls out the `ModuleNotFoundError` failure mode as an environment mismatch, not
   a real bug, referencing this report.
5. **`docs/agent-quickstart.md` was checked and needs no change** — it's scoped to reviewing a
   *target* project's own tests/diffs when using vibe-council as a dependency, not to vibe-council's
   own install/dogfood instructions; it doesn't mention `workbench`/`uv run` because those aren't in
   its scope.

## 4. Browser automation

Not attempted in this pass — the Workbench panel smoke test (§2c) used direct HTTP requests
(`curl`) against `GET /`, `GET /api/state`, and `POST /api/tasks/demo` instead, per the plan's
fallback allowance. This confirmed binding, page content, state shape, and token gating without the
risk of a hanging browser-automation session (a documented failure mode in an earlier session of this
same track). A follow-up manual browser check (§6) is still worth doing at some point, but is not a
blocker — the HTTP-level checks already exercise the same server code paths a browser would.
