The **AI Council Workbench MVP** release. A task now moves through **visible stages**, an AI proposes
a change, an **audited approval** step gates it (deterministic guards are the boundary; the Approval
Auditor is advisory only), and — as a **separate, explicit** step — an approved, narrowly-bounded
action can actually run: a file write/edit behind a verified local payload artifact, or an exact
allowlisted command behind a fixed-argv resolver. Everything is logged. Still local-first and
stdlib-only: no new runtime dependency, no provider/model/network call from the executor, no LAN/
mobile/remote surface. Based on and crediting
[`karpathy/llm-council`](https://github.com/karpathy/llm-council).

## Highlights

- **Runtime store + state machine** — a gitignored `.council/runtime/` JSON tree
  (`Task`/`Stage`/`ApprovalRequest`/`ApprovalDecision`/`Action`/`AuditResult`) and a deterministic task
  lifecycle (plan → request approval → approve/reject/hold → executing → complete/fail/hold).
- **Deterministic trust boundary** — the real security gate: path allow/deny, command allowlist,
  secret patterns, change-size limits, cloud-egress consent. Unknown kinds and non-allowlisted/
  metachar commands are blocked by default; this boundary is **re-run at execution time**, not just at
  approval time.
- **Advisory Approval Auditor** — a human-readable approval summary (risk label + rewritten prompt)
  that copies risk/blocked/findings verbatim from the trust boundary, so it can **never** relax a
  block.
- **Localhost Workbench panel** (`vibe workbench serve`) — task/approval cards, approve/reject/hold,
  and a **"Create demo task"** button for a safe, non-executing first look (see Known limitations).
- **Payload artifacts + hash/scope verification** — bounded file actions carry their content in a
  local, gitignored, write-once `.council/runtime/payloads/<action_id>.json`, hashed at creation and
  re-verified (hash + kind/target/approval/task agreement) before every real execution.
- **Bounded file write/edit execution** — atomic writes, size/line-delta limits, path/symlink guard,
  explicit-overwrite requirement, no content ever written to logs.
- **Exact allowlisted command execution** — a fixed label→argv resolver (no shell, no string parsing,
  `sys.executable`-based, no OS-specific launcher dependency) and real execution via
  `subprocess.run(shell=False)` with a sanitized environment, project-root cwd, a timeout, and bounded/
  redaction-checked output capture.
- **Panel execute + result display** — a separate, explicit "Execute" step (with a confirm-dialog UX
  nudge, not a security boundary) for both bounded file actions and allowlisted commands.

## Security model

- **Approval is separate from execution.** Approving only records a decision; it never triggers a
  file write, edit, or command run.
- **The deterministic trust boundary re-runs at execution time** — a stored/stale `AuditResult` (or a
  cached preview) cannot authorize anything; the advisory Auditor is never the gate.
- **The browser sends only an action id** (+ a startup token) to
  `POST /api/actions/<action_id>/execute` — never file content, patch text, a command string, argv,
  cwd, env, or a timeout. The executor resolves everything itself, server-side.
- **Payload artifacts are local and gitignored**, write-once, and hash/scope-verified before use.
- **Command execution is fixed argv + `shell=False`, always** — a label resolves to a hardcoded,
  pre-built argv; no `shlex`/string parsing, no dynamic arguments, ever.
- **The panel is localhost-only (`127.0.0.1`) and token-gated** — no `0.0.0.0`/LAN binding, no CORS
  wildcard.

## Explicit non-goals

No arbitrary shell · no dynamic command arguments · no package install/deploy/`git push` (or any git
mutation) · no provider/model/API calls from the executor · no LAN/mobile/remote access · no voice ·
no autonomous/background loops. None of these are partially built — they are out of scope by design
(see the guarded-executor and command-execution plans' stop conditions).

## How to dogfood locally

```sh
git pull
.venv\Scripts\python.exe -m unittest discover -s tests -t .     # Windows
.venv/bin/python -m unittest discover -s tests -t .             # macOS/Linux
python -m backend.cli workbench serve                           # or: vibe workbench serve
```

1. Open the printed `http://127.0.0.1:<port>/?token=<token>` URL in a browser.
2. Click **"Create demo task"** if the panel is empty — this seeds one task + one pending approval
   (a proposed `write_file:docs/example.md`) with a saved advisory audit.
3. **Approve / reject / hold** the demo approval and confirm the card updates.
4. The demo **intentionally creates no `Action` and no payload artifact** (a PR #77 safety decision:
   there's no path in an arbitrary real project root that's both trust-boundary-writable and
   guaranteed harmless to a reflex click), so **no "Execute" button appears for the demo task** — this
   is expected, not a bug. To exercise real execution, seed an approved `Action` (+ payload artifact
   for file kinds) against a **throwaway temp project directory** yourself — see
   `tests/test_workbench_panel.py`'s `_seed_action` helper for exactly how, or rely on the test suite,
   which already exercises this path extensively.
5. **Inspect the result** — a completed/failed/blocked action shows a safe, content-free result
   summary (bytes written, exit code, etc.), never raw file content or unbounded command output.
6. **Stop the server** with `Ctrl+C`.

## Known limitations

- **Intentionally local-only** — no LAN/mobile/remote access; the panel binds `127.0.0.1` only.
- **Intentionally tiny command allowlist** (7 entries — verification/status commands only); it is not
  meant to grow casually.
- **No hosted/team/mobile/voice** yet — deferred to v0.6+ at the earliest, and only after real-repo
  dogfood proves the current loop.
- **The panel is intentionally simple** — no framework, no rich UI, no persistence beyond the runtime
  JSON store.
- **Windows/Linux dogfood is still needed before a wider release** — the command executor's
  `sys.executable`-based resolution and environment sanitization were designed for cross-platform
  parity but should be confirmed green on the full CI OS matrix before tagging.

## Next after v0.5.0

Do **not** jump into hosted/team/cloud work. In order: dogfood on a small real (non-`vibe-council`)
repo → fix rough edges found there → maybe a v0.5.1 hardening pass (mirroring the v0.3.1 precedent) →
only then consider LAN/mobile access or a richer operator workflow.
