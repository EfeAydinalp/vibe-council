  ([`v0.7-personalization-and-project-profile-plan.md`](v0.7-personalization-and-project-profile-plan.md))

  safe-to-commit boundary and never-store list. Root `AGENTS.md` deliberately **not** the canonical

## 2. v0.7.1 purpose — hardening, not feature expansion

v0.7.0 added *surface* (a scaffold and pointers to it). v0.7.1 makes that surface **safe to build
on** before any behavior application exists. Focus:

- **Profile safety** — lock in, with rules and tests, that the local `.council/profile.*` class can
  never leak into commits, exports, or guides.
- **Redaction / no-stage discipline** — the redaction guard and doctor should *name* the new private
  path class instead of covering it only by prefix accident.
- **Doctor consistency** — distinguish "scaffold absent" (fine) from "likely-confused state" (root
  `AGENTS.md` present while `AGENT-ROLES.md` is missing).
- **Export/guide safety invariants** — keep pointers-only / never-read-local / graceful-degradation
  true under future edits by encoding them as tests and stated invariants, not habits.
- **Avoiding `AGENTS.md` confusion** — one write-target (root, guide output), one preference source
  (vault `AGENT-ROLES.md`), stated consistently everywhere.
- **Making future personalization safer** — when a later phase adds a profile store or a preference
  reader, these guardrails must already be waiting for it.

v0.7.1 should be **small**. Most of the safety work is *tests + one narrow rule + doc consistency*,
because grounding shows the important protections largely already exist (see §4A).

## 3. Non-goals (explicit — v0.7.1 does NOT do any of these)

- **No preference application engine.** Nothing reads a preference and changes behavior.
- **No parser that turns preferences into commands.** Preferences remain prose an agent reads.
- **No local `.council/profile.*` implementation** — not created, not read, not written — unless a
  later phase explicitly scopes it.
- **No vector DB.** **No PostgreSQL / Redis / Docker / Next.js / LangGraph.** **No hosted/SaaS.**
- **No Workbench trust-boundary change.** **No command-allowlist expansion.** **No auto-execution.**
- **No `/council` real command.** **No model/provider/API calls** from anything added here.
- **No secrets storage** anywhere, in any form.

## 4. Hardening areas — evaluation and decisions

### A. Redaction hardening

**Grounding (verified against `backend/redaction.py` on master):**

- The lint's default targets are **tracked `docs/` files + tracked top-level markdown** (via
  `git ls-files`); gitignored `.council/` is never scanned, and `tests/` are not scanned.
- The **secret rules already cover the scaffold**: `openrouter-key`, `private-key-block`,
  `api-key-assignment`, `secret-assignment` (all CRITICAL) run over every tracked doc — including
  `PROFILE.md`/`PREFERENCES.md`/`AGENT-ROLES.md`. "Should public profile files be scanned for
  secret markers?" — **they already are**; what's missing is a *test* that locks this in.
- The `council-artifact-path` CRITICAL rule matches only date-stamped
  `.council/(reviews|decisions|context|runs|stages|usage|diffs)/…` paths — **it does not know
  `.council/profile.*`**.
- The doctor's dangerous-staged check uses prefix matching on `.council/` — a staged
  `.council/profile.json` **already FAILs** today. This is coverage by prefix, not by name; a test
  should pin it so a future refactor of the prefix list can't silently drop it.
- Two committed design docs legitimately reference the concrete filename `.council/profile.json`
  ([`10-personalization-layer.md`](10-personalization-layer.md) and the v0.7 brief); all
  operational docs use the glob form `.council/profile.*`.

**Decision — implement now (PR 1), deliberately narrow:**

1. Add a **WARNING** rule (id: `local-profile-path`) matching a **concrete** local-profile filename —
   `\.council[\\/]profile\.(json|md|toml|ya?ml)` — message: "Reference to a local/private profile
   artifact (never commit; `.council/profile.*` is machine-local)". WARNING, not CRITICAL, follows
   the existing `private-plan-filename` precedent: legitimate design-doc references exist and must
   not break the 0-critical gate. **Expected effect: the tracked-docs warning count rises by the
   known design/plan-doc references** — `10-personalization-layer.md`, the v0.7 brief, and **this
   plan itself** (its grounding/prompts quote concrete fixture names) — roughly 22 → ~29. The PR
   must enumerate the exact expected findings in its tests so the count change is intentional, and
   note the discriminator: operational/policy text prefers the glob form `.council/profile.*`, which
   the rule deliberately does not match.
2. Update [`docs/redaction-policy.md`](../redaction-policy.md) to name `.council/profile.*` as a
   private-path class alongside runtime/payload artifacts.
3. **Tests that lock in existing protections** (no behavior change): a planted secret-shaped value in
   a scaffold-like fixture trips the existing CRITICAL rules; a staged `.council/profile.json` in a
   temp git repo makes the doctor FAIL (pinning the prefix behavior by name).

**Defense-layer map (which layer catches which failure):**

| Failure case | Catching layer |
|---|---|
| Concrete `.council/profile.<ext>` path committed in a public doc | new `local-profile-path` lint WARNING (PR 1) |
| `.council/profile.*` file **staged** for commit | doctor dangerous-staged FAIL — `.council/` prefix (existing; test-pinned in PR 1) |
| Profile-like **content** (secret-shaped value) pasted into a committed doc | existing CRITICAL secret rules (test-pinned in PR 1) |
| The local private profile file itself | gitignore — never tracked, never scanned (existing) |
| Export/guide reading the local profile | never-read-local tests (existing, PR C/D) |

**Severity decision (WARNING now, with a stated promotion path):** the rule ships as WARNING for
v0.7.1, matching the `private-plan-filename` precedent (legitimate design-doc references must not
break the 0-critical gate). **Promotion criterion:** if/when a later phase actually implements a
local `.council/profile.*` store, a concrete profile-path reference in public docs stops being a
design mention and becomes a live-artifact leak — at that point the rule is promoted to CRITICAL and
the two design docs are edited to the glob form as part of that phase. This is a documented decision,
not deferred noise.

**Test-fixture requirement (PR 1):** planted secrets in fixtures must use obviously-fake values
(e.g. `sk-test-FAKE…` / clearly synthetic base64-shaped strings) with a comment marking them
test-only — never patterns resembling real keys. The test must assert the finding severity is
**CRITICAL**, pinning that a secret-shaped value in a scaffold file always blocks.

**Not now (overbuilding):** scanning gitignored files by default (the lint scans *committed*
surface — scanning local private files would be a different tool with different privacy questions);
CRITICAL severity today (see the promotion path above); content-schema validation of scaffold files.

### B. Project doctor polish

**Grounding:** the doctor currently warns on *any* root `AGENTS.md` (pointing to the vault
convention) and warns per-missing scaffold file with one next-step line. Advisory vs. fail semantics
are correct and must not change.

**Decision — implement now (PR 2), small:**

1. **Differentiate the root-`AGENTS.md` warning** by scaffold state:
   - root `AGENTS.md` present **and** `AGENT-ROLES.md` present → keep today's informational warn
     (guide-output target, not a preference source).
   - root `AGENTS.md` present **and** `AGENT-ROLES.md` missing → a **stronger-worded warn** naming
     the likely-confused state. **Exact wording (use this, matter-of-fact not scolding):**
     `[warn] root AGENTS.md exists but docs/context/project/AGENT-ROLES.md does not — this suggests
     a configuration mismatch. Root AGENTS.md is a guide-output target only; keep agent-role
     preferences in docs/context/project/AGENT-ROLES.md.` Still a warn, **never** a failure.
2. **Partial-scaffold note:** when *some but not all* scaffold files exist, add one advisory line
   naming it a partial scaffold (the per-file `[ok ]`/`[warn]` lines already show which).
3. Doctor's guide/export pointers: the doctor already lists the `vibe guide` commands; add
   `vibe context export --for <agent> --role <role>` to that informational block so both onboarding
   surfaces are discoverable from one place.

**Keep unchanged:** READY/NOT-READY semantics (required docs + dangerous-staged only), advisory
context health, git-unavailable → warn, exit codes.

### C. Context-export safety

**Grounding:** PR C already enforces pointers-only, no `.council/profile.*` read (tested with a
planted secret), graceful degradation, determinism, and a <12 000-char lean-export ceiling.

**Decision — tests/docs only (fold into PR 3), no behavior change:**

1. Add a **profile-section size guard test**: the "Project profile & preferences" section stays
   under a small fixed budget (≈1 200 chars), so future edits can't quietly grow it into a dump.
2. Add a **no-full-vault-dump test**: distinctive body content from *every* scaffold file (not just
   one) is absent from the export — extending the existing needle set.
3. Docs: state the export's profile invariants (pointers-only / never-read-local / graceful /
   deterministic) as **named invariants** in the vault `WORKFLOWS.md` line, so they read as contract
   rather than description.
4. **"Graceful degradation," defined concretely** (the contract PR 3's tests assert): the profile
   section is **static** — it renders **identically** whether the scaffold files exist or not: the
   same pointer paths, the same tighten-only and root-`AGENTS.md` notes, and the `vibe project
   doctor` recommendation (which is how a reader discovers the files are absent). The section is
   never empty, never errors, and never switches to inlining; absence of the scaffold must not
   change a single byte of export/guide output.

### D. Guide safety

**Grounding:** `_GUIDE_PROFILE` is static text — the guide never reads the filesystem, so
missing-scaffold and never-reads-local hold by construction; the tighten-only wording and
never-loosen tests exist.

**Decision — tests/docs only (fold into PR 3):**

1. Add one **wording-invariant test**: guide output must contain "tighten" + "never loosen" *and
   must not* contain phrases implying preference authority over safety (assert the section keeps
   "advice to read, not commands" or an equivalent stated marker). This is a guard against a future
   copy-edit softening the safety text.
2. Keep root `AGENTS.md` as a guide-output target **only**; the guide already says it is not the
   preference source — no change needed beyond the test.

### E. Vault consistency

**Grounding:** the never-store/no-stage rules are stated in WORKFLOWS (canonical), PREFERENCES
(restates, explicitly deferring to WORKFLOWS), PROFILE/AGENT-ROLES (per-file lists), RISKS
(`AGENTS.md` collision entry), README (boundary note). PREFERENCES already names WORKFLOWS as the
source of truth for the no-stage list — the right pattern.

**Decision — light consistency pass (fold into PR 3):**

1. One read-through pass confirming the four statements of never-store/tighten-only don't drift;
   where a file restates a rule, it must *point at* the canonical statement (WORKFLOWS for no-stage,
   invariant #17 / the v0.7 brief for tighten-only). Fix only actual contradictions; do not rewrite
   for style.
2. **Do not** wire vault files into the context pack. The pack remains a budgeted projection
   (STATUS + decision index, 21/21); the vault is read directly via pointers. Reopening the pack
   budget is a stop condition, not a task.

### F. Release and review policy (restated, binding for v0.7.1)

- **cheap** — the small implementation PRs below (they follow this reviewed plan).
- **balanced** — this plan itself (phase/hardening architecture), and any PR that unexpectedly
  broadens redaction/security semantics.
- **full council** — only if a critical trust/executor/hosted/network/payload/secrets/payment/
  security-model question appears. None is expected in v0.7.1.

## 6. Risk analysis

- **Preference files treated as commands.** The core future risk. v0.7.1's defense: the guide/export
  wording tests (PR 3) pin "advice to read, not commands," and no parser exists. Any future PR that
  proposes *reading* preferences into behavior must re-derive tighten-only from invariant #17 and
  take a balanced review minimum.
- **Root `AGENTS.md` confusion.** A user keeping role preferences in root `AGENTS.md` gets silently
  out of sync with the vault source. PR 2's state-differentiated warning targets exactly this; the
  write-target/read-source split (RISKS.md entry) remains the rule.
- **Local/private profile leaks.** Layered defenses after PR 1: gitignore (existing), doctor
  staged-FAIL by prefix (existing, newly test-pinned), the `local-profile-path` lint rule (new),
  never-read-local export/guide tests (existing). Residual risk: a user pasting profile *content*
  (not path) into a doc — the existing secret rules partially cover this; full coverage is
  impossible without content schemas (deliberately out of scope).
- **Context pack bloat.** The vault keeps growing; the pack must not. Defense: §4E's hard rule (the
  pack never ingests the vault) + PR 3's export size-guard. Touching the 21/21 budget is a stop
  condition everywhere in this phase.
- **Redaction false positives/negatives.** The new rule's concrete-extension discriminator keeps
  false positives to the two known design docs (accepted, stated); false negatives (e.g. a profile
  path with an unlisted extension) are acceptable because the rule is one layer of four, not the
  boundary.
- **Overbuilding before real use.** The standing v0.7 critique. This plan's answer: PRs 1–3 are
  mostly tests and one rule; anything shaped like a schema, parser, store, or generator is a
  non-goal. If an area's "hardening" starts growing production surface, stop.
- **Accidentally loosening Workbench security.** No PR here touches trust/executor/panel/importer
  paths. Any diff that drifts into those files violates its own scope — re-read
  [`03-security-invariants.md`](03-security-invariants.md) and stop.
- **Fable overuse / broad scans.** This pass read the curated set + two targeted code files
  (`redaction.py` grounding). Future Fable involvement in v0.7.1: none needed; PRs 1–4 are routine
  Opus/Sonnet work against this plan.
