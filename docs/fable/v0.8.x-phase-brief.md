# v0.8.x — Phase brief (council-backed)

> **Planning document. No code, no behavior change, no version bump, no tag/release.** This is the
> cleaned phase brief produced from the council debate in
> [`v0.8.x-council-debate.md`](v0.8.x-council-debate.md) (a `vibe review --preset balanced`
> multi-model critique of the candidate directions, plus a second balanced pass over this brief). The
> **Fable architecture step is next and expected** — this brief is its input; see
> [`v0.8.x-fable-input.md`](v0.8.x-fable-input.md). Fable plans the version line; it does **not**
> implement routine PRs.

## Chosen theme: "Solidify the core, local-first"

**v0.8.x makes the existing local-first product markedly better and more adoptable — it does not add
new trust surface and does not turn personalization into behavior.** Concretely:

- **Headline — Agent launcher / session workflows** (candidate B): reduce per-session/per-project
  re-onboarding via `vibe init-agent`-style bootstrap and richer, confirm-before-overwrite onboarding
  generation, reusing the existing guide/vault/export machinery.
- **Supporting — Project vault polish** (candidate C): `vibe summarize-history`, a capped
  `RELEASES.md`, tighter context-pack budget discipline, small curation UX. Keeps the pack lean
  (a real pressure) and the vault the source of truth.
- **Supporting — Workbench agent-proposal UX** (candidate D): clearer proposal cards, better authoring/
  validation errors, easier review — **strictly within the unchanged trust boundary**, every PR
  balanced-or-full reviewed with explicit invariant tests.
- **Personalization precondition (docs + read-only validation, NOT behavior):** a formal, testable
  **tighten-only preference-control model** and an auditor that *reads and validates* `PREFERENCES.md`
  against it (a `vibe config doctor`-style, read-only check) — **auditable before executable**. This
  is the safe on-ramp that lets v0.9.x apply preferences later; it changes **no** command's behavior.

## Why this theme won

- **The council explicitly rejected "preference behavior as the headline"** as premature — the
  highest-risk direction proposed with no mechanical safety guarantee. Making personalization *real*
  is valuable, but the safe path is **model-and-validate first, apply later**.
- **B is the highest-leverage, lowest-risk value:** it targets the stated #1 adoption pain, reuses
  tested machinery, and needs architectural focus rather than security research.
- **It preserves local-first credibility** — zero security-surface expansion — which every recent
  release has advertised and which is the product's spine.
- **It sets up v0.9.x cleanly:** the preference-control model + validator is the documented,
  tested foundation the council demanded before any behavior ships.

## Alternatives rejected or deferred

- **A — Preference application / preference-aware guidance (as a theme): DEFERRED to v0.9.x**, behind
  the hard prerequisite of a formal, tested preference-control model. v0.8.x builds only the
  *model + read-only validator*, never the applier. Guide/context-export stay **pointer-only**.
- **A′ — "Ultra-Safe named profiles" (Alt-2): OPTIONAL/STRETCH only.** A fixed, hand-audited set
  (`default`/`strict`/`paranoid`) selectable without any parser. Included only if the maintainer wants
  a tangible personalization taste in v0.8.x; otherwise dropped. Never natural-language parsing.
- **F — Skills / `SKILL.md`: SIDE POC, constrained.** If done, only as a **generated pure projection**
  of the vault/roadmap/invariants that introduces no new policy and is never hand-edited — not a
  headline.
- **H — Mobile / LAN / Voice: DEFERRED to its own gated security line** (pre-v0.9 threat-model doc
  required). A **local, same-machine notification** subset (no LAN/mobile) is the only part that could
  be safe now and is noted as a candidate, not committed.
- **E — Website / commercialization: DEFERRED behind license/provenance "Question 0."**
- **G — MCP polish: not a headline;** opportunistic read-only-only improvements at most.

## Committed scope (council-informed; Fable finalizes numbering)

The **committed core** of v0.8.x is three slices. Two "candidate follow-ons" are listed **separately**
below — they are *not* numbered optional releases (the council flagged that "false optionality"); Fable
decides at the architecture step whether each is **in** this line or **deferred to v0.9.x**.

- **v0.8.0 — Agent launcher / session workflows (headline).** `vibe init-agent`-style onboarding
  bootstrap + confirm-before-overwrite generation, reusing guide/export. Read-only generators; writes
  only user-chosen files, with confirmation and **filesystem path safety** (refuse writes outside the
  project root, no symlink traversal, guard against path injection — reuse the executor's path-guard
  posture). Sketch of what it writes: user-chosen onboarding files (e.g. `CLAUDE.md`/`AGENTS.md`-style)
  containing **vault pointers**, small and bounded — never a vault dump.
- **v0.8.1 — Vault polish + localhost-only CI assertion.** `vibe summarize-history` (local digest),
  capped `RELEASES.md`, context-pack budget discipline (pack stays 21/21; vault never dumped into the
  pack); **plus a mandatory CI/test assertion that nothing binds beyond `127.0.0.1`** (no inbound
  listeners) — a concrete deliverable of this slice, cheap review.
- **v0.8.2 — Preference-control model + read-only validator.** A documented **tighten-only schema** —
  a *formally specified, enumerated set of audited, add-friction-only* preference types (see the
  worked example below) — plus a `config doctor`-style validator that **reads and validates**
  `PREFERENCES.md` against it and reports conflicts/risks. **No behavior application.** The validator
  treats file content as **untrusted input**: a strict, memory-safe format (prefer JSON/TOML over
  YAML), bounded size, fail-closed on malformed input, no code execution. This slice defines trust
  semantics → **full review** (see policy table).

### Candidate follow-ons (Fable decides: in v0.8.x or deferred)

- **Workbench proposal UX** (candidate D): clearer cards / authoring / validation errors **within the
  unchanged trust boundary**; every PR balanced-or-full with invariant re-verification.
- **Ultra-Safe named profiles** (candidate A′): selection among a fixed, hand-audited profile set
  (`default`/`strict`/`paranoid`), **no parser**. Open question for Fable: is this an *instance of the
  v0.8.2 model* or a separate mechanism, and does exercising it count as "application" (v0.9.x) or a
  safe pre-application pilot?

Exact numbering/bundling is Fable's call; this is the shape, not a contract.

## Worked example — one tighten-only preference (anchors v0.8.2)

To make "tighten-only" concrete (the council's single most-requested addition):

- **Preference:** `default-review-preset` in `PREFERENCES.md`, allowed values `cheap|balanced|full`.
- **Semantics (tighten-only):** it may only *raise* the default review strictness a command suggests
  (e.g. make `balanced` the floor). It **cannot** lower strictness below the project default, and it
  **cannot** authorize `premium` or `--allow-premium` (that always needs explicit human approval).
- **Validation rule (v0.8.2, read-only):** the validator confirms the value is in the enum and is
  `>=` the project baseline; a value that would *weaken* the default is reported as a **conflict/risk**
  (never silently applied — and in v0.8.x, *never applied at all*).
- **Why it's safe:** the only effect a preference of this type could ever have is *more* review, never
  less. This is the template every enumerated preference type must satisfy: **prove, per type, that
  every legal value can only add friction/checks, never remove them.**

This example is illustrative; Fable defines the full enumerated schema from 2–3 concrete v0.9.x
target behaviors so the model is anchored in real use, minimal, and explicitly evolvable.

## Safety invariants (bind every v0.8.x PR)

All of [`03-security-invariants.md`](03-security-invariants.md) hold unchanged. Specifically for
this phase:

- **Personalization stays tighten-only and, in v0.8.x, non-applying** — v0.8.x may *validate* a
  preference but must not let it change any command's behavior; the deterministic trust boundary
  ignores preferences.
- **Guide / context-export stay pointer-only.** *Pointer-only* means: the output contains a **file
  path or vault-section reference**, and the command **reads that pointer but never inlines the
  content it references** (and never reads a local/private profile, `.council/profile.*`).
- **The v0.8.2 validator treats `PREFERENCES.md` as untrusted input** — a strict, memory-safe format
  (prefer JSON/TOML over YAML), bounded size, fail-closed on malformed content, no code execution, no
  denial-of-service exposure. Parsing must not be able to do anything but read + report.
- **Approval stays separate from execution; no auto-execution; localhost-only; no arbitrary shell; no
  command-allowlist expansion; no CORS/LAN/mobile surface.**
- **New behavior that touches the trust boundary (if any) is feature-flagged default-off** until a
  security review clears it (council process ask).
- **A CI/test-level "localhost-only" assertion is MANDATORY** for the phase (a concrete v0.8.1
  deliverable): tests assert nothing binds/listens beyond `127.0.0.1`, and the launcher/validator open
  no arbitrary ports and start no local HTTP server without dedicated review.
- **Any new third-party dependency requires explicit security review + version pinning** — but the
  default remains **stdlib-only, no new dependency** (a non-goal below).

## Non-goals (v0.8.x)

No preference **parser/application engine** (v0.9.x, behind the model); no natural-language
text-to-behavior; no `.council/profile.*` **local store** (unless Fable later designs a narrow, safe,
read-only approach); no mobile/LAN/voice or any non-localhost surface; no Workbench trust-boundary
loosening or allowlist growth; no arbitrary shell; no auto-execution; no hosted/SaaS; no vector DB; no
PostgreSQL/Redis/Docker/Next.js/LangGraph; no secrets storage; no `/council` real command; no model/
provider/API calls introduced by planning; **no new runtime dependency** (stdlib-only default — any
exception needs a stated reason, security review, and version pinning).

## Provisional PR sequence & review policy

| Slice | Direction | Review level | Why |
|---|---|---|---|
| v0.8.0 launcher | B | **balanced** | new command surface + file emission (confirm-before-overwrite, path safety) |
| v0.8.1 vault polish + localhost CI | C | **cheap**, balanced for the `context build` change | low risk; the budget change is architecture-relevant |
| v0.8.2 pref model + validator | A-precondition | **full** (by default) | it *defines trust semantics* (what preferences are safe) — even though read-only |
| Workbench UX (candidate) | D | **balanced or full per PR** | trust-boundary adjacency; invariant re-verification required |
| named profiles (candidate) | A′ | **balanced** (full if it drifts toward application) | proves preference wiring; must stay non-loosening |
| each release prep | — | **cheap** | mechanical |

- **full council** is reserved for anything that could touch the trust boundary / executor / network /
  secrets / payload — including **v0.8.2 by default** (per the council: a validator that defines what
  preferences are safe is defining trust semantics) and any Workbench-UX PR.

## Stop conditions (halt and re-scope)

- A "launcher" grows into an interactive runtime, a `vibe council start`, or auto-writes files without
  confirmation.
- Any v0.8.x work starts **applying** a preference to behavior (that is v0.9.x, behind the model).
- Vault polish reopens the 21/21 pack budget or ingests the vault into the pack.
- A Workbench-UX diff weakens any invariant in [`03`](03-security-invariants.md), adds a network
  surface, or lets the client send payload/argv.
- Mobile/LAN/voice, a `.council/profile.*` store, or a preference parser appears without an explicit,
  separately-approved design.
- Dependency churn, a version bump implied without instruction, or any private artifact staged.

The second `vibe review --preset balanced` pass over this brief **conditionally approved** it —
"exceptionally well-structured… the decision to separate the preference *model* from its *application*
is the single best decision in the plan" — with **no serious security/trust flaw**, but asked for
concrete revisions to make it actionable before Fable. Applied (in-scope, planning-level):

- **A worked tighten-only preference example** (`default-review-preset`) — see the dedicated section
  above; the council's single most-requested addition.
- **A one-sentence "pointer-only" definition** in the invariants.
- **v0.8.2 upgraded to full review by default** (it defines trust semantics).
- **Parser hardening for v0.8.2**: `PREFERENCES.md` treated as untrusted input, strict memory-safe
  format (JSON/TOML over YAML), bounded size, fail-closed, no code execution.
- **The localhost-only CI assertion made mandatory** and a concrete v0.8.1 deliverable.
- **Launcher filesystem path-safety** (no writes outside project root, no symlink traversal,
  path-injection guards) + a sketch of what it writes (bounded vault-pointer files, never a dump).
- **A dependency-security constraint** (review + pinning; stdlib-only default) and "no new runtime
  dependency" added to non-goals.
- **The false-optionality resolved**: v0.8.3/v0.8.4 are no longer numbered "optional" releases; they
  are **candidate follow-ons** Fable decides in-or-out.
- **"Provably add-friction-only" reframed** to "a formally specified, enumerated set of audited,
  add-friction-only preference types."

Deliberately **left to Fable, not pre-decided here** (recorded as open questions in
