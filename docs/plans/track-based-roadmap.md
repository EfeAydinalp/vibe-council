# Track-based roadmap

A **public-safe, track-based roadmap** for vibe-council. It captures the full product direction
without flattening it into a simple version list: work is organized into parallel **tracks**, and the
**version sections** say which tracks advance, which are intentionally deferred, and why.

**Docs-only.** This is direction, not a commitment to ship on a date; nothing here changes code,
dependencies, provider behavior, or releases.

## Core product statement

> **vibe-council = local-first council workflow + linked project memory + context-pack builder +
> skill/council packs + (later) an optional hosted commercial layer.**

## Framing: AI-first Markdown project memory, not an Obsidian clone

- We are **not** building an Obsidian clone. We are building an **AI-first Markdown project memory +
  decision graph + context-pack engine**.
- **vibe-council is not an Obsidian clone. It stores curated project memory as plain Markdown so
  humans can browse it in Obsidian / VS Code / GitHub and agents can consume it through context packs
  / MCP.**
- **Obsidian is an optional viewer/editor, not a dependency.** No `.obsidian/` is committed; no
  Obsidian plugin is required or vendored.
- Canonical data is **plain Markdown files** that work in GitHub, VS Code, Obsidian, and similar tools.
  A future GUI may *visualize* decisions/links/graphs, but the **canonical data stays Markdown**.
- Agent access comes from **context packs / MCP read-only export**, never from direct coupling to the
  Obsidian app.

## Tracks

Ten parallel tracks. Each version section below advances some and defers others on purpose.

### Track 1 — Core trust / release
The stable foundation everything else rides on.
- Provider abstraction (the `Provider` seam under the council).
- OpenRouter (default) + local Ollama; `VIBE_OLLAMA_MODEL`.
- `vibe doctor` (no-token diagnostics).
- Tests + release discipline (stdlib `unittest`, CI on Linux/macOS/Windows, tagged releases).
- License/provenance resolution (the gating item — see Track 10).
- Install, security, and docs hygiene.
- **Status:** largely shipped at v0.2.0; this track is "keep it stable," not "keep adding."

### Track 2 — Decision memory / context pack
The likely moat: durable, curated, linked project memory an agent can consume.
- `docs/decisions/` dogfood (curated, committed ADR-style records).
- Project-memory folder convention (Track 3).
- **Context-pack builder** — assemble a low-token, agent-ready pack from curated records.
- **Agent-brief generation** — distilled project memory for Claude Code / agents.
- **Rolling summaries every 5–10 decisions** + multi-level summaries (record → topic → project).
- **"Do not re-litigate" rejected alternatives** — rejected options are first-class so agents don't
  re-propose them.
- **Token-reduction waves without quality collapse** — compress context in stages, preserving signal.
- **Extract / promote workflow:**
  1. raw council / chat / PR output (local, gitignored),
  2. **extract** candidate decisions / facts / constraints / rejected options,
  3. **human approve / promote** (curation gate),
  4. curated decision records / `STATUS` / summaries (committed, redacted),
  5. **context-pack build** from the curated set only.

### Track 3 — Project memory / Obsidian-compatible docs
Editor-agnostic plain-Markdown project memory.
- `docs/context/project/README.md` — stable identity/constraints (thin; points to `CLAUDE.md` + brief).
- `STATUS.md` — short current snapshot, next action, blockers, needs-review.
- `PROGRESS.md` — **local/generated by default** (under gitignored `.council/`); committed only as an
  opt-in milestone digest (git history already records most of it).
- `DECISIONS.md` — **only as an index/summary if ever needed, never the source of truth.**
- **`docs/decisions/*.md` remains the source of truth.**
- **Standard Markdown links are canonical;** wikilinks are **optional/future only**.
- **No Obsidian dependency; no `.obsidian/` commit.** Obsidian is an **optional viewer/editor only.**
- **Editor-agnostic:** GitHub, VS Code, Obsidian, and similar Markdown tools.
- A **future GUI can visualize graph/links**, but the canonical data stays Markdown.

### Track 4 — MCP / Claude Code / Remote Control
Agent access to project memory. **Not immediate, but explicitly on the roadmap.**
- **First:** context export for Claude Code (read a context pack / agent brief).
- **Then:** **MCP read-only** — `list` / `show` / `export` decisions and context.
- **Writes behind approval only** (create/link/accept a decision requires explicit human approval).
- **Design around official Claude Remote Control** for phone/remote approval — run cleanly *inside* a
  Remote-Control'd Claude Code session and emit clear, push-worthy decision points.
- **Do not build our own mobile/remote transport** unless later strongly justified.

### Track 5 — Operator control loop / approval inbox
A practical bridge for "Claude is waiting for approval/choice and the user missed it." **Not a full
dashboard.**
- **Early shape:**
  - a local status/events log,
  - terminal/desktop notification,
  - simple event types: `approval_required`, `choice_required`, `pr_ready`, `tests_failed`,
    `merge_ready`, `blocked`,
  - **no secrets / private diffs in notifications.**
- **Later shape:**
  - integrates with Claude Code context export / MCP (Track 4),
  - Remote-Control-friendly approval points,
  - phone-friendly workflow **through official primitives where possible**.
- **Risk:** can drift into a custom, insecure mobile transport if overbuilt; must stay **minimal and
  local-first early**.

### Track 6 — Skill / council packs
Packaged, curated review personas (curated content is the asset).
- Product Review Pack, Code Review Pack, Commercial Direction Pack, **Defensive Security Council Pack**
  (defensive-only — see Track 10), Docs/README Review Pack.
- **Source-fed `SKILL.md` drafts later** (e.g. a NotebookLM-style distillation from curated sources).
- **Generated packs are drafts only; human review required; source citation required; no invented
  facts.**

### Track 7 — Multi-agent orchestration / dashboard
- An Octogent-style dashboard is **relevant but later**.
- **First foundation:** context pack + task-specific council runs.
- **Later:** multiple council/Claude sessions over a **shared context**.
- A dashboard comes **only after the memory/context layer works**.
- A dashboard should **visualize/control workflow, not become the product**.

### Track 8 — Commercial / open-core
Public-safe direction only; detailed strategy stays private/local.
- Current core is **public / local-first**.
- The **full commercial feasibility plan is private/local**; only a public-safe **commercial direction
  decision** is committed ([open-core commercial direction](../decisions/2026-06-29-open-core-commercial-direction.md)).
- **BYOK-first.** No early token resale.
- **Support / consulting + Product/Code Review pack first.**
- **Individual Pro / sync later; team sync later.**
- **Private hosted repo only after validation.**
- *(Out of scope here: cost tables, pricing, hosted/billing implementation, private strategy.)*

### Track 9 — Retrieval / vector
- **Plain retrieval first:** tags, links, backlinks, recency, keyword, **pinned decisions**, summaries,
  **rejected alternatives**.
- **Later:** hybrid / vector retrieval; a **time-boxed local-embeddings prototype** to test whether
  semantic search materially helps.
- **Vector should not become the product** — it's an optional accelerator, never a hard core dependency.

### Track 10 — Security / provenance
- Raw `.council/` stays **local/gitignored**; only **curated, redacted** docs are public.
- **No API keys / local absolute paths / raw outputs** in committed docs.
- **Redaction / secrets guard** (path containment, allowlist sanitization, injection delimiters; treat
  redaction as defense-in-depth, never a guarantee).
- **License/provenance is "Question 0"** before serious commercialization.
- **Security Council is defensive-only** — no exploit-generation product; guardrails operationalized,
  not just stated.

## Version sections

Each version notes: **tracks advancing**, **tracks deferred (and why)**, **prerequisite**, **expected
shape**, **risk**, **later milestone**. Wording is deliberate: deferred items are **not now** with a
**prerequisite** and a **later milestone** — not "never."

### v0.2.x — public-safe docs + conventions
- **Tracks advancing:** T1 (stability), T3 (project-memory convention), T8 (public commercial
  direction), T10 (safety/redaction spec).
- **Tracks deferred:** T2 tooling, T4, T5, T6, T7, T9 — **not now;** they depend on the conventions and
  the curated record set landing first.
- **Prerequisite:** v0.2.0 shipped; curated `docs/decisions/` exist.
- **Expected shape:** public-safe docs; the project-memory folder **convention** (README + short
  STATUS committed; PROGRESS generated/local); a tightened redaction/secrets-guard **spec**. No runtime
  code.
- **Risk:** convention without dogfooding becomes shelfware; mitigate by hand-authoring real records.
- **Later milestone:** v0.3.0 tooling builds on these conventions.

### v0.3.0 — decision-memory tooling + context-pack MVP
- **Tracks advancing:** T2 (tooling + context-pack MVP), T10 (CI secrets/redaction guard).
- **Tracks deferred:** T4/T5 (need a context pack to export), T6 (needs packs format), T7, T9 — **not
  now;** prerequisite is a working context pack.
- **Prerequisite:** v0.2.x conventions validated by dogfooding.
- **Expected shape:** `decisions show/new/promote/lint`, on-demand index, backlink resolver, a
  **context-pack builder MVP**; stdlib-only.
- **Risk:** building tooling before curation is proven; gate on the dogfood A/B signal.
- **Later milestone:** v0.3.x quality + export work.

### v0.3.x — context quality + extract/promote + inbox design
- **Tracks advancing:** T2 (agent-brief generation, rolling summaries, extract/promote workflow), T3
  (project STATUS/PROGRESS **export**), T5 (operator-inbox **design** only).
- **Tracks deferred:** T4 implementation, T5 implementation, T6 — **not now;** prerequisite is reliable
  context-pack quality.
- **Prerequisite:** context-pack MVP exists and is useful.
- **Expected shape:** agent-brief generation; rolling summaries every 5–10 decisions; STATUS/PROGRESS
  export (generated, default local); a **written design** for the operator inbox (event types, no
  secrets in notifications). No transport built.
- **Risk:** token-reduction that loses signal; measure context quality, not just size.
- **Later milestone:** v0.4.0 agent surfaces.

### v0.4.0 — Claude Code export + MCP read-only + Remote-Control-friendly
- **Tracks advancing:** T4 (context export → MCP read-only `list/show/export`; writes behind approval),
  T5 (early operator inbox: local events log + terminal/desktop notifications).
- **Tracks deferred:** our own remote transport — **not now;** prerequisite/decision is to **design
  around official Claude Remote Control**; revisit only with strong justification.
- **Prerequisite:** a stable context pack / agent brief to expose.
- **Expected shape:** read-only context export for Claude Code; an MCP read-only surface; approval-only
  writes; Remote-Control-friendly approval points; minimal local-first inbox.
- **Risk:** the inbox drifting into an insecure custom mobile transport; keep it minimal/local-first.
- **Later milestone:** v0.4.x packs; v0.6+ hosted (if validated).

### v0.4.x — skill / council packs
- **Tracks advancing:** T6 (Product/Code/Commercial-Direction/Defensive-Security/Docs review packs;
  source-fed `SKILL.md` **drafts**).
- **Tracks deferred:** T7 orchestration, T9 vector — **not now;** prerequisite is a stable pack format +
  context layer.
- **Prerequisite:** context pack + agent surfaces (v0.4.0).
- **Expected shape:** a handful of curated packs; generated packs are **drafts only**, human-reviewed,
  **source-cited, no invented facts**.
- **Risk:** low-quality skill proliferation / credibility damage; enforce the citation + curation gate.
- **Later milestone:** v0.5.0 paid validation.

### v0.5.0 — paid workflow validation
- **Tracks advancing:** T8 (support/consulting + Product/Code Review packs + templates).
- **Tracks deferred:** hosted/team/sync — **not now;** prerequisite is validated willingness-to-pay and
  resolved license/provenance.
- **Prerequisite:** packs exist; license/provenance (Track 10) cleared.
- **Expected shape:** education/service-first validation (support, training, curated review packs,
  workflow templates); core stays open; BYOK-first. *(No cost tables/pricing here.)*
- **Risk:** services consuming all maintainer time; cap effort and define a continue/stop signal.
- **Later milestone:** v0.6.0+ optional hosted layer if demand proves out.

### v0.6.0+ — optional hosted / team / sync layer (if demand exists)
- **Tracks advancing:** T8 (hosted convenience), supported by T10 (hosted security bar).
- **Tracks deferred:** dashboard (T7), vector (T9) — **not now;** prerequisite is paid demand +
  security architecture.
- **Prerequisite:** validated paid demand; license/provenance resolved; the full §G-style security bar.
- **Expected shape:** an **optional, opt-in** hosted layer (e.g. team sync of curated decisions / shared
  brief), **possibly in a separate private repo**; never gating the open core. BYOK-first; no unmetered
  token resale.
- **Risk:** building hosted infra before demand; tenant-data/security burden; keep it separate + opt-in.
- **Later milestone:** v0.7.0+ orchestration once memory/hosted layers are solid.

### v0.7.0+ — multi-agent orchestration / dashboard experiments
- **Tracks advancing:** T7 (orchestration experiments over shared context).
- **Tracks deferred:** anything that bypasses the context layer — **not now;** prerequisite is a working
  context/memory layer.
- **Prerequisite:** context pack + agent surfaces proven (v0.4.x+).
- **Expected shape:** experiments running multiple council/Claude sessions over a shared context;
  dashboard **visualizes/controls** workflow.
- **Risk:** the dashboard becoming the product / scope creep; keep it a thin layer over the memory core.
- **Later milestone:** v0.8.0+ retrieval scaling.

### v0.8.0+ — hybrid / vector retrieval (if plain retrieval is insufficient)
- **Tracks advancing:** T9 (hybrid/vector retrieval; time-boxed local-embeddings prototype).
- **Tracks deferred:** making vector a hard dependency — **not now;** prerequisite is demonstrated
  insufficiency of plain retrieval.
- **Prerequisite:** plain retrieval (tags/links/recency/keyword/pinned/summaries) shown insufficient at
  scale.
- **Expected shape:** an **optional** hybrid/vector accelerator; markdown + index remain fully usable
  without it.
- **Risk:** over-building retrieval infra; vector must **not become the product**.
- **Later milestone:** v1.0 stabilization.

### v1.0 — stable local-first decision-workflow product
- **Tracks advancing:** consolidation across T1–T3, T6, T8, T10 into a stable release.
- **Tracks deferred:** experimental T7/T9 remain **optional** layers, not core requirements.
- **Prerequisite:** the local-first council + decision-memory + context-pack workflow is dependable and
  dogfooded; license/provenance resolved.
- **Expected shape:** a stable **local-first decision-workflow product** — council + linked project
  memory + context packs + curated packs — with optional agent (MCP) and optional hosted layers.
- **Risk:** feature sprawl diluting the core; 1.0 means "the core is dependable," not "everything shipped."
- **Later milestone:** post-1.0 iteration on the optional layers as demand directs.

## Cross-cutting principles

- **Local-first and public-core by default;** optional layers (agent/MCP, hosted, dashboard, vector)
  never gate the core.
- **Prerequisite-driven, not date-driven** — a track advances when its prerequisite is met.
- **Curation gate + redaction** on everything promoted from local → committed/synced.
- **Markdown is canonical;** viewers/GUIs/agents consume it, they don't own it.
- **Nothing is "never."** Deferred work has a *prerequisite*, an *expected shape*, a *risk*, and a
  *later milestone*.
