# v0.5 guarded-executor track is functionally complete; readiness checklist precedes a tagged release

## Context

PR #72–#81 built the Workbench's guarded-executor track end-to-end: plan → dry-run → payload bridge →
bounded file execution → command resolver → allowlisted command execution → panel wiring for both.
Before considering a v0.5.0 tag, this PR records what the track now contains, what it deliberately
doesn't, a dogfood recipe, and blocker/security checklists
([plan](../plans/v0.5-release-readiness.md)).

## Decision

Accept the readiness checklist; **no executor/panel/CLI behavior changes in this PR.**

- **v0.5 now contains** the runtime store, orchestrator, deterministic trust boundary, advisory
  Approval Auditor, localhost panel with demo UX, payload artifacts + hash verification, bounded file
  execution, exact allowlisted command execution, and panel execute/result display for both.
- **v0.5 explicitly excludes** arbitrary shell, dynamic command args, package install/deploy/`git
  push`, provider/model/API calls from the executor, LAN/mobile/remote access, voice, background/
  autonomous loops, a plugin system, multi-user/hosted mode, and cloud sync — none partially built.
- **The demo does not seed an executable action.** "Create demo task" seeds a task + pending approval
  only (no `Action`, no payload artifact) — a deliberate PR #77 safety choice. The readiness doc states
  this plainly so onboarding/dogfood instructions don't overclaim what a fresh install can do without
  manual (temp-project-only) seeding.
- **A manual smoke checklist, security checklist, and release-blocker checklist** are defined (see the
  plan doc §4–§6) — covering approve/execute separation, trust re-check timing, browser-input scope
  (action id only), payload/command sandboxing, and the usual redaction/test/context-pack gates.
- **Context-pack budget pressure is flagged, not fixed.** PR #81's context check still passed 21/21,
  but only after reaching a later stage of the trim fallback chain than earlier PRs in this track.
  Future decision records in this area should stay in the ~4–5 KB range already established, rather
  than growing further — this is guidance for future PRs, not a mechanism change in this one.
- **This is not the release-cut process.** The generic version-bump/tag/CHANGELOG runbook remains
  [`docs/release-checklist.md`](../release-checklist.md); this document only establishes that the
  Workbench *track* is functionally ready for that process to eventually run against it.

## Rationale

- Writing down what's built vs. deliberately excluded, in one place, prevents scope drift and gives a
  concrete reference for reviewing any future "just this one small addition" request against.
- Being explicit that the demo doesn't execute anything avoids a predictable dogfood confusion
  ("I clicked approve, where's the Execute button?") without having to change the demo's safety
  posture to manufacture one.
- Flagging the context-pack trend now (rather than after a future PR silently drops required facts)
  keeps the project's own context-quality gate meaningful going forward.

## Alternatives considered

- **Cut a v0.5.0 tag directly in this PR** — rejected; the recommended sequence (release notes →
  real-repo dogfood → fix rough edges → maybe a hardening pass) should run first, matching how v0.3.1
  followed v0.3.0's dogfood cycle.
- **Make the demo seed an executable action to "prove" the loop** — rejected; no path in an arbitrary
  user's real project root is both trust-boundary-writable and guaranteed harmless to a reflex click;
  this was already decided in PR #77 and stands.
- **Tighten the context-pack budget mechanism now** — rejected; out of scope for a readiness checklist;
  flagged as guidance instead.

## Consequences

- Anyone picking up v0.5 next (dogfooding, reviewing, or planning v0.6) has a single accurate reference
  for scope, limitations, and what must hold before a tag.
- No dependency/provider/network/version change; no new runtime surface.

## Next actions

- Dogfood v0.5 on a real small repo (not this one) per the plan's §8 sequence.
- Prepare `docs/releases/v0.5.0.md` once the blocker checklist is clean and CI is green on the full
  OS matrix.
