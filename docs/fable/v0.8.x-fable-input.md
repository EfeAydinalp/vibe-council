## What the council concluded

- **Chosen theme: "Solidify the core, local-first."** Headline **agent launcher / session workflows**;
  supported by **vault polish** and **carefully-reviewed Workbench-proposal UX**. Zero new trust
  surface.
- **Personalization becomes a *model + read-only validator* in v0.8.x, not behavior.** Guide/context
  export stay **pointer-only**; the `.council/profile.*` local store and any preference
  parser/application engine stay **deferred to v0.9.x**, behind a formal, tested tighten-only
  preference-control model.
- **Mobile/LAN/voice is deferred to its own gated security line** (needs a pre-v0.9 threat-model doc);
  a **local same-machine notification** subset is the only part that might be safe now.
- **Website/commercialization stays behind license/provenance "Question 0."**
- The council rejected the initial lean (headlining preference behavior) as premature — "exactly how
  invariants erode."

## Risks that remain (Fable must keep these in view)

- **Preferences-as-commands** — the core future risk. Even a *validator* defines what preferences are
  "safe," which is trust-semantics-adjacent; v0.8.2 is **full-review** for this reason.
- **Untrusted `PREFERENCES.md` parsing** — must be memory-safe format, bounded, fail-closed, no code
  execution, no DoS.
- **Launcher overbuild / path safety** — a "launcher" must not become an interactive runtime or
  auto-write files; writes are confirmed, bounded (vault pointers, not dumps), and path-guarded (no
  writes outside project root, no symlink traversal).
- **Workbench-UX trust adjacency** — any UX PR can accidentally weaken an invariant; per-PR
  balanced/full + invariant re-verification.
- **Pack budget** — vault polish must keep the pack at 21/21 and never ingest the vault.
- **Localhost-only regression** — a mandatory CI assertion must guard it.

## Questions Fable must answer (the gating decisions)

1. **Preference-control schema (v0.8.2):** the exact enumerated set of tighten-only preference types,
   anchored in **2–3 concrete v0.9.x target behaviors**; for each type, the proof/test that every legal
   value can only *add* friction/checks. (Use the `default-review-preset` worked example in the brief
   as the template.)
2. **Validator shape:** format of `PREFERENCES.md` the validator reads (JSON/TOML preferred over YAML),
   its output (report + exit code), and its untrusted-input hardening. Is it a new `vibe config doctor`
   or folded into `vibe project doctor`?
3. **Application boundary:** confirm v0.8.x ships **no** preference application — or, if you judge a
   *single, formally-trivial* first application worth piloting (the second council pass floated a
   "refusal filter" that can only fully disable a command), justify it against the first pass's
   "no application in v0.8.x" and gate it behind full review + a feature flag default-off. Default
   expectation: **no application in v0.8.x.**
4. **Candidate follow-ons — in or deferred:** decide whether **Workbench-proposal UX** and the
   **Ultra-Safe named-profile** pilot are part of v0.8.x or move to v0.9.x; if named profiles are in,
   say whether they are an instance of the v0.8.2 model or a separate mechanism.
5. **Launcher command surface:** the exact `vibe init-agent`-style surface, what it writes (bounded
   vault-pointer files), and its confirm-before-overwrite + path-safety guarantees.
6. **Mobile/LAN/voice + notifications:** confirm mobile/LAN/voice is out of v0.8.x; decide whether a
   **local same-machine notification** enters v0.8.x or waits; note the required pre-v0.9 threat-model
   doc.
7. **Normative safety spec:** whether to produce a short, **testable** normative spec for
   "tighten-only / read-only / pointer-only / localhost-only" that the phase's tests assert against.
