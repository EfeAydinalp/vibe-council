# Plan: v0.5.1 — dogfood & hardening (no new features)

A **planning/checklist document**, not an implementation. `v0.5.0` (the AI Council Workbench MVP —
runtime store, orchestrator, deterministic trust boundary, advisory Approval Auditor, localhost
panel, payload artifacts + hash verification, bounded file execution, exact allowlisted command
execution, panel execute/result display) is **tagged and released**. Before any v0.6 feature work,
this plan scopes a **v0.5.1 dogfood/hardening pass**: find and fix the rough edges a fresh install and
a real (non-`vibe-council`) repo surface, without adding product surface. **No executor/panel behavior
changes, no command allowlist expansion, and no v0.6 work start in this task.**

Review date: **2026-07-02**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local.

> **Scope guard.** v0.5.1 is a **hardening release**, mirroring the v0.3→v0.3.1 precedent: no new
> command surface, no executor/panel capability change, no allowlist growth. If dogfooding surfaces a
> need for new capability (not just a rough edge), that's v0.6+ scope — stop and scope it separately
> rather than folding it in here.

---

## 1. Purpose

`v0.5.0` shipped a complete but freshly-built vertical slice. It has been exercised by this repo's own
test suite (570 tests) and by the PR-by-PR dogfood notes in `docs/plans/v0.5-release-readiness.md`,
but **not yet by a fresh install on a clean checkout, nor against a real, unrelated repo**. v0.5.1's
job is to close that gap: run the actual onboarding path a new user would follow, on both a clean
checkout of this repo and a real small project, on more than one OS, and fix what's rough — docs,
messages, tiny bugs — without touching the security model or adding scope.

## 2. Dogfood environments

Exercise the checklists below (§3–§7) across, in this order:

1. **Existing dev checkout** (this machine, this clone) — fastest signal, catches obvious breakage.
2. **A clean clone** (`git clone` into a fresh directory, fresh `.venv`) — catches anything the dev
   checkout's accumulated state was hiding (stale `.venv`, cached `.council/`, etc.).
3. **Windows PowerShell** (primary environment for this development machine).
4. **A Linux shell**, if available (WSL, a CI runner, or a spare Linux box) — the project explicitly
   targets a Windows/macOS/Linux CI matrix; at minimum, confirm CI is green there even if a manual pass
   isn't feasible on this machine.
5. **A temp/safe test repo** (a throwaway directory, never a real project) — the first place to
   exercise real Workbench execution (write/edit/command), matching the manual-seeding pattern from
   `docs/plans/v0.5-release-readiness.md` §3.
6. **A real small repo** — only **after** the temp repo passes cleanly. Pick something low-stakes
   (a personal scratch project, not anything shared or production), and review what any seeded action
   would do before approving/executing it.

## 3. Fresh install checklist

Run from a **clean clone**, not the existing dev checkout:

- [ ] `git clone <repo-url>` into a fresh directory.
- [ ] `uv sync` (or the project's documented equivalent) completes without error.
- [ ] `uv run python -m backend.cli --version` (or the installed `vibe --version`) prints
      `vibe-council 0.5.0`.
- [ ] `uv run python -m backend.cli doctor` (or `vibe doctor` if installed) reports the selected
      provider with no network/token spend.
- [ ] `uv run python -m backend.cli status` (or `vibe status`) shows a sane empty/initial workspace
      state.
- [ ] `.venv\Scripts\python.exe -m unittest discover -s tests -t .` (Windows) /
      `.venv/bin/python -m unittest discover -s tests -t .` (POSIX) is green (570+ tests). **Use the
      synced `.venv`'s own interpreter, not a bare `python`** — a system `python` without the synced
      deps fails with `ModuleNotFoundError: No module named 'dotenv'` and a wall of unrelated-looking
      test failures, which is an environment mismatch, not a real bug (confirmed during the PR #86
      clean-clone dogfood pass).
- [ ] `uv run python -m backend.cli lint --redaction` (or `vibe lint --redaction`) reports
      **0 critical**.
- [ ] `vibe decisions lint` passes.
- [ ] `vibe context build` runs without a Python traceback (warnings about budget trimming are
      expected and fine — see §8).
- [ ] `vibe context check` scores **21/21**.
- [ ] `vibe mcp inspect --context --health` scores **21/21**.

Any friction here (confusing error, missing step in a doc, a command that doesn't exist as documented)
is a v0.5.1 candidate fix (§9), not something to route around silently.

## 4. Workbench dogfood checklist

- [ ] `vibe workbench serve` starts and prints a URL bound to **`127.0.0.1`** (never `0.0.0.0`).
- [ ] A startup **token** is printed (unless `--no-token`).
- [ ] Opening the printed URL in a browser loads the panel.
- [ ] The **empty state** ("No tasks yet.") reads as understandable, not alarming or broken-looking.
- [ ] **"Create demo task"** seeds one task + one pending approval; the card is readable (title, risk
      label, human-readable rewritten prompt).
- [ ] **Approve / reject / hold** records the decision and the card updates accordingly.
- [ ] **Execution stays explicit** — approving alone never runs anything; no "Execute" button appears
      for the demo task (it seeds no `Action` by design — see release-readiness §3).
- [ ] A **manually seeded** (temp-repo only) approved `write_file`/`edit_file` action with a valid
      payload artifact shows an execute button and, on click + confirm, actually writes/edits and marks
      `completed`.
- [ ] A **manually seeded** approved `run_command` action for an **exact allowlisted** label shows an
      execute button and, on click + confirm, actually runs and reports a result.
- [ ] A **non-allowlisted** or mutated command shows **no** execute affordance and blocks if triggered
      directly.
- [ ] Posting a crafted request body (different command/argv/content) to the execute endpoint has
      **no effect** — only the server-resolved action/payload/argv is ever used.
- [ ] The result display (exit code, timeout/truncation flags, bounded stdout/stderr summary, or the
      file-write byte count) is **readable**, not a wall of raw text or a cryptic code.
- [x] `Ctrl+C` stops the server cleanly. **Status: verified by a real manual test (PR #88).** A
      maintainer ran `uv run python -m backend.cli workbench serve`, pressed Ctrl+C once in a real
      attached terminal, and the prompt returned immediately (clean shutdown). A `netstat` listener
      seen afterward on the same default port turned out to be an unrelated, pre-existing local
      service (`PhoneScriptRunner`, fingerprinted via its HTTP response), not vibe-council — there is
      no confirmed shutdown bug, and a source-code re-check confirms no code path can bind to
      `0.0.0.0` (`make_server()` hardcodes/validates loopback-only, now with an additional post-bind
      address re-check as defense-in-depth). See
      [`workbench interactive smoke report`](./v0.5.1-workbench-interactive-smoke-report.md) §7. The
      earlier `uv run`-wrapped-launch leftover-child-process pattern (killing only the outer `uv`
      process) remains a known harness/tooling quirk, not a Workbench bug.

## 5. Windows-specific hardening checklist

Windows is this development environment's primary platform, and the command executor's
`sys.executable`-based resolution was specifically designed to avoid Windows/POSIX drift — verify that
design holds in practice:

- [ ] Every command example in `README.md` / setup docs that targets PowerShell actually runs
      verbatim in PowerShell (no bash-isms leaking through).
- [ ] `scripts/install-vibe.ps1` (if a user would reasonably run it) completes and leaves a working
      `vibe` command.
- [ ] PATH / `vibe.cmd` behavior is unambiguous: does `vibe` resolve after install, and if not, is the
      fallback (`uv run python -m backend.cli ...` or an explicit PATH step) documented?
- [ ] The `uv run ...` fallback is documented somewhere reachable for a user whose `vibe` isn't on
      PATH yet.
- [ ] The panel's printed URL is clickable/copy-pasteable and opens correctly in a Windows default
      browser.
- [ ] The command resolver (`backend/workbench_commands.py`) is confirmed using `sys.executable` and
      **not** shelling out to `vibe.ps1`/`.cmd`/`.sh` for any allowlisted label (already true by design
      — this is a verification step, not a code change).
- [ ] No documented dogfood step requires shell-specific syntax (no bash `&&`/`||` chains presented as
      if they work unmodified in PowerShell, no PowerShell-only syntax presented as POSIX).

## 6. UX polish checklist

- [ ] Error messages (blocked/failed reasons) are understandable to someone who didn't write the code
      — not just technically accurate.
- [ ] The distinction between `completed` / `blocked` / `failed` is visually and textually clear on
      an action card.
- [ ] Risk labels (`low`/`medium`/`high`/`blocked`) read clearly, with enough context to act on them.
- [ ] Command output truncation is clearly flagged (not silently cut off with no indication).
- [ ] A missing or tampered payload artifact produces a message a human can act on ("payload artifact
      not found" / "payload verification failed"), not a bare exception trace.
- [ ] No raw payload content (file content, patch text, full command output beyond the bounded/
      redacted summary) is ever shown anywhere in the panel.
- [ ] The first-run empty panel doesn't read as broken or scary — it should invite the demo button,
      not look like an error state.
- [ ] The README quickstart section, read literally top-to-bottom by someone who has never used the
      tool, matches what actually happens at each step (no stale command names, flags, or output
      shapes).

## 7. Security regression checklist

Nothing here should have regressed since `v0.5.0` — this is a **re-verification**, not new design:

- [ ] Approval is **not** execution; approving never auto-executes.
- [ ] The browser sends **only an action id** (+ token) to the execute endpoint.
- [ ] Payload artifacts stay **local and gitignored** (`.council/runtime/payloads/`).
- [ ] The deterministic trust boundary **re-runs at execution time**, not just at approval time.
- [ ] The Approval Auditor remains **advisory only** — it cannot relax a blocked/high-risk verdict.
- [ ] `run_command` execution uses a **fixed, resolver-provided argv** and **`shell=False`**, always.
- [ ] No dynamic arguments, no arbitrary shell, no path to package install/deploy/`git push`.
- [ ] No provider/model/API call originates from the executor.
- [ ] No LAN/mobile/remote access — the panel is `127.0.0.1`-only.
- [ ] `git status --short` after any dogfood session shows **no** `.council/` or payload artifact
      staged (they're gitignored — this should be structurally impossible, but verify).

## 8. Known issues to watch

- **`uv.lock` self-version drift** — the `vibe-council` self-package entry in `uv.lock` has read a
  stale version since before `v0.5.0` (confirmed during `v0.5.0` prep: it wasn't synced, matching
  every prior release-prep commit's actual behavior). **Not a `v0.5.0` blocker** — it shipped that
  way. Decide during v0.5.1 whether to fix it in a small, separately-scoped hygiene PR (a `uv lock`
  run + verification that nothing else in the lockfile shifts unexpectedly), rather than folding it
  into a dogfood-fix PR.
- **Context-pack budget is close.** Recent PRs in the Workbench track needed progressively deeper
  fallback steps (down to "capped decision index entries") to still score 21/21. Future decision
  records should stay in the ~4–5 KB range already established (see
  `docs/decisions/2026-07-01-context-pack-budget-headroom.md`); don't let v0.5.1's own decision record
  (or dogfood-fix PRs) reopen this.
- **The demo doesn't cover every real execution path** — it seeds an approval only, never an `Action`
  or payload artifact (a deliberate `v0.5.0`/PR #77 safety choice). Real execution dogfood requires
  manual seeding (§4); don't let this checklist quietly get skipped because the demo alone "looks
  done."
- **Windows/Linux parity still needs real-world dogfood**, not just code review — §5 exists because
  the design is believed correct, not yet independently re-verified end-to-end on this pass.
- **The command allowlist is intentionally tiny** (7 entries). Finding it "too small" during dogfood
  is not itself a v0.5.1 bug — allowlist growth is a deliberate, separately-reviewed decision, not a
  hardening fix.

## 9. v0.5.1 candidate fixes (likely small; confirm during dogfood, don't invent now)

- Documentation corrections found while literally following the fresh-install/dogfood checklists.
- README quickstart corrections (stale command names, wrong flag names, output that no longer matches).
- Clearer panel messages (blocked/failed reasons, payload-missing/tampered wording, risk-label copy).
- A better first-run/empty-panel explanation if dogfooding shows it reads as broken.
- Better error/result summaries (execute-endpoint responses, action `result_summary` text) if dogfood
  shows real users can't parse them.
- Test coverage gaps discovered during manual dogfood that the existing suite doesn't already cover.
- Release-notes / public-docs cleanup (stale links, outdated version references elsewhere in `docs/`).
- **`uv.lock` hygiene** — only if explicitly scoped as its own small change with its own verification,
  per §8; not bundled silently into an unrelated fix.

None of these are scoped or committed to by this plan — they're the *category* of fix v0.5.1 targets;
the actual fix list comes from what dogfooding (§2–§7) actually finds.

## 10. Explicitly deferred to v0.6+ (do not implement here)

- **Agent-to-Workbench action bridge** — letting an external agent (Claude Code, Codex, etc.) propose
  Workbench tasks/actions directly, rather than manual seeding.
- **Action/payload proposal helpers** — tooling to generate a well-formed `Action` + payload artifact
  from a diff/plan, closing the "no path authors a real proposal end-to-end" gap noted in
  `docs/plans/v0.5-release-readiness.md` §3.
- **Claude/Codex integration docs** for the above, once the bridge itself exists.
- **Project/user personalization profile** and **council preset personalization**.
- **Mobile / LAN / voice approval** — the panel stays `127.0.0.1`-only through v0.5.1.
- **Hosted / team / commercial features** — unrelated to this track; license/provenance remains
  "Question 0" regardless.

## 11. Exit criteria for v0.5.1

- [ ] A clean-clone dogfood pass (§3) completes without a blocking issue.
- [ ] The Windows PowerShell path (§5) is confirmed and documented; a Linux pass or green CI leg is
      confirmed if feasible.
- [ ] A full panel smoke pass (§4) — including at least one manually-seeded file action and one
      manually-seeded command action, in a temp/safe project — completes successfully.
- [ ] `vibe lint --redaction` shows **0 critical** findings on the v0.5.1 candidate state.
- [ ] `vibe context check` and `vibe mcp inspect --context --health` remain **21/21**.
- [ ] No security regression against §7's checklist.
- [ ] Every issue discovered during dogfood is triaged: fixed in v0.5.1, explicitly deferred to v0.6+
      (§10), or logged as a known issue (§8) — nothing left silently unaddressed.

## 12. Acceptance criteria for this plan

- Defines concrete, checkbox-level dogfood/hardening/security-regression checklists (§3–§7), not vague
  intentions.
- Distinguishes "small fix v0.5.1 might make" (§9) from "explicitly deferred to v0.6+" (§10) so
  triage during dogfood has a clear default.
- Flags known pre-existing issues (§8) so dogfood doesn't rediscover and mis-scope them as new bugs.
- States exit criteria (§11) concrete enough to know when v0.5.1 is actually done.
- Adds no executor/panel/CLI behavior, no command allowlist growth, and no v0.6 work in this PR.
