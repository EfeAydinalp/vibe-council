# v0.5.1 is a dogfood/hardening pass on the released Workbench MVP, not new feature work

## Context

`v0.5.0` (the AI Council Workbench MVP) is tagged and released. Before any v0.6 feature work, the
project needs a focused pass that actually exercises a fresh install and a real repo — something the
release itself proved only via this repo's own test suite and PR-by-PR notes, not an independent
onboarding path ([plan](../plans/v0.5.1-dogfood-hardening.md)).

## Decision

Accept a docs-only v0.5.1 dogfood/hardening plan; **no executor/panel behavior change, no command
allowlist expansion, and no v0.6 work in this PR.**

- **v0.5.1 is a hardening release**, mirroring the v0.3 → v0.3.1 precedent: find and fix rough edges
  (docs, messages, small bugs) without adding product surface.
- **Concrete checklists, not vague intentions:** fresh-install (clean clone → `uv sync` → version →
  `doctor`/`status` → full test/lint/context/MCP gate), Workbench panel smoke (empty state → demo →
  approve/reject/hold → manually-seeded file/command execution → non-allowlisted block → result
  display → clean shutdown), Windows-specific verification (PowerShell examples, install script,
  PATH/`vibe.cmd`, `sys.executable`-based command resolution — no OS-specific launcher dependency),
  UX polish, and a security-regression re-check against everything `v0.5.0` already established.
- **Known issues are named up front** so dogfood doesn't rediscover and mis-scope them: `uv.lock`'s
  stale self-version entry (pre-existing since before `v0.5.0`, not a blocker, decide on a *separate*
  hygiene PR if fixed), the context-pack budget running close to its limit (future decision records
  should stay concise), the demo intentionally not seeding an executable action, and Windows/Linux
  parity still needing real-world (not just code-review) verification.
- **A v0.5.1 candidate-fix list stays a category, not a commitment** — actual fixes come from what
  dogfooding finds, not from inventing work now.
- **Everything longer-term stays explicitly deferred**: an agent-to-Workbench action bridge, action/
  payload proposal helpers, Claude/Codex integration docs, project/user personalization, mobile/LAN/
  voice approval, and hosted/team/commercial features are v0.6+ (or later) — not touched here.
- **Exit criteria are concrete**: clean-clone pass, Windows path confirmed (+ Linux/CI if feasible), a
  full panel smoke including at least one manually-seeded file action and one command action, 0
  critical redaction findings, context/MCP health still 21/21, no security regression, and every
  discovered issue triaged (fixed, deferred, or logged) rather than silently dropped.

## Rationale

- A released MVP that has only been dogfooded by its own author, in its own repo, understates real
  onboarding friction — a genuinely fresh install and a real project are the only way to find it.
- Naming known issues up front (§8 of the plan) prevents re-litigating already-decided tradeoffs
  (the demo's non-executing design, the tiny allowlist, the stale `uv.lock` entry) as if they were new
  discoveries during dogfood.
- Keeping v0.5.1 scoped to hardening — not features — matches the project's own v0.3→v0.3.1 precedent
  and avoids scope creep into v0.6 territory before v0.5 is proven solid.

## Alternatives considered

- **Skip a dedicated hardening release and go straight to v0.6 features** — rejected; the v0.3.1
  precedent showed dogfood-driven hardening catches real friction before it compounds into a feature
  release.
- **Fix `uv.lock` drift inline as part of this plan** — rejected; keeping it as an explicitly
  separately-scoped candidate avoids bundling an unrelated dependency-lock change into a docs/hardening
  PR.
- **Let dogfood decide scope with no written plan** — rejected; without concrete checklists and named
  known-issues, dogfood risks rediscovering settled tradeoffs or missing checks entirely.

## Consequences

- v0.5.1 has a clear, checkbox-level scope before any dogfood session starts.
- Future v0.6 planning can point at §10 of the plan for what's already agreed to be out of v0.5.1's
  scope, rather than re-deciding it.
- No dependency/provider/network/version change in this PR.

## Next actions

- Run the fresh-install and Workbench dogfood checklists (plan §3–§7) on a clean clone and a real
  small repo.
- Triage findings into: fixed in v0.5.1, deferred to v0.6+ (plan §10), or logged as a known issue.
