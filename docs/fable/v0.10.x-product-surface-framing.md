# v0.10.x — product-surface decision gate (decision document, not a plan)

> **Pre-Fable decision gate. No code, no behavior change, no version bump, no tag/release, no
> implementation.** This file is a **decision document**: it frames the candidate next surfaces after
> the released v0.9.1 and runs a balanced council-style review to decide **whether a v0.10.x line
> should open at all**, and if so on what surface — or whether the safe next step is a scoped v0.9.2
> dogfood/tooling patch, a pause for product-surface/commercialization planning, or no-new-surface
> maintenance. It changes no prompt construction, peer ranking, chairman synthesis, model/provider
> selection, the preference reader/validator or `PREFERENCES.md` machine-block semantics, the
> guide/context-export output, or the Workbench/importer/executor/guard/trust boundary. It adds no
> schema field and implements no `schema: 2`. **Fable is not used for this gate** — Fable reads the
> council-produced files only *after* the council picks a direction; this step decides whether that
> Fable step should even happen next. See the production band in §7.

This gate is the companion to the v0.9.1 outputs: [`council-review-lenses.md`](council-review-lenses.md)
(the documentation-only lens vocabulary) and
[`v0.10.x-dissent-preservation-sketch.md`](v0.10.x-dissent-preservation-sketch.md) (the design-only
safety framework any future persona/lens *behavior* must satisfy). Both were shipped as documentation
in v0.9.1; **neither is behavior, and no behavior has been decided.** This document exists because the
v0.9.x line is complete and released, the theme "apply the proven; describe the personas; **defer their
behavior**" has been fully delivered, and the next question — *"is describing the personas enough for
now, or is it time to build something?"* — is a genuine fork that should be resolved by the council
under the production band, not assumed.

## 2. Why this gate exists

The whole v0.7 → v0.9 arc was built on one discipline: **model-and-validate before you apply, and apply
the provable before the novel.** v0.9.x delivered the last safe application (the four keys) and then
*described* the personas without building them — because the council found **dissent suppression /
viewpoint bias** to be a **new risk class** the tighten-only proofs do not cover. That deferral was
correct. It also means the project now sits at a genuine fork with **no pre-decided next step**:

- The dissent-preservation sketch is a *sketch* by its own name (§13 "living design doc, not a
  contract"); it defines a bar and is honest that part of the bar (reframing-bias evaluation) is an
  **open research problem** — "not yet" is a legitimate verdict, and **cancellation is on the table,
  not just deferral.**
- The roadmap and risk register both mark persona **behavior** as deferred to v0.10.x+ **and gated**
  behind that framework — not scheduled.
- Opening a v0.10.x line under the production band commits the council to active debate + exact
  planning files, then Fable to an architecture/risk/PR breakdown, then Opus/Sonnet to scoped PRs.
  That is a real commitment of the most trust-adjacent kind the project has attempted. **It should not
  be entered by momentum.** This gate forces the choice to be made deliberately, by the council, with
  the safety framework already written as the reference bar.

The single most important thing this gate protects against: **treating "we documented the personas" as
"we are now building the personas."** Documentation is not a commitment to build (both v0.9.1 files say
so explicitly). This gate asks the council to decide the commitment on its own merits.

## 3. Candidate next surfaces

Five candidates, deliberately spanning "build the biggest thing" to "build nothing new." Each is scored
in §4 on the same axes the v0.9.x council debate used, so the comparison is apples-to-apples.

- **A — open v0.10.x for persona / review-lens *behavior*.** Turn one or more documented lenses into
  applied behavior (collect-stage prompt emphasis behind the full dissent-preservation framework:
  additive, own-seat-only, ranking/synthesis persona-blind, default-off, feature-flagged, with the
  adversarial dissent-canary suite as a merge gate). The dissent-preservation sketch's recommended
  slice is a **single hardcoded, default-off** persona (e.g. Security Guardian), not the full seven.
- **B — open v0.10.x for dissent-preservation *infrastructure only*.** Build the *test/observability
  scaffolding* the sketch requires — the dissent-canary fixture harness, ranking/synthesis
  byte-identity assertions, an off-state golden, a differential-coverage evaluation probe — **without
  applying any persona to any real prompt.** This is "prove the bar is buildable and measure the
  research residual" before any behavior ships.
- **C — do a scoped v0.9.2 dogfood/tooling patch (no new surface).** The optional patch slot the
  v0.9.x architecture plan explicitly reserved. Dogfood the v0.9.0 preference application + the v0.9.1
  docs on real work, fix rough edges (CLI UX, warning hygiene, doctor advisory wording, help text,
  pointer/link hygiene) — additive, reversible, no new command surface, no persona anything.
- **D — pause engineering and do product-surface / commercialization planning.** Step back from
  version-line engineering and run a product/direction pass (positioning, first-paid-MVP order,
  licensing "Question 0", who the buyer is) before choosing the next engineering line. Docs/strategy
  only; the two historical `docs/plans/` files are prior art for this.
- **E — no-new-surface maintenance only.** Keep the released surface exactly as is; take no new line;
  do only reactive maintenance (a real bug, a dependency/security fix) if and when one appears.
  "Stop and hold" as an explicit, choosable option.

## 4. Candidate analysis

Each candidate below is analysed on **product value**, **architecture risk**, **security/trust risk**,
**local-first fit**, **required prerequisites**, **what it commits the repo to**, and **what could go
wrong**. Scores use the v0.9.x debate's scale (low / med / high).

### A — open v0.10.x for persona / review-lens behavior

- **Product value: high (narrative).** The first *real* persona value — a Security-Guardian-flavored
  review that actually shapes a seat. This is the product's long-advertised direction.
- **Architecture risk: med–high.** Touches the council **prompt-construction** path and is adjacent to
  the fragile `full`-mode ranking/chairman code (a documented known issue). Requires a feature flag,
  a content validator, and the full canary harness before a single line of emphasis can ship.
- **Security / trust risk: high — this is the new class.** Dissent suppression / viewpoint bias does
  **not** inherit the mechanical add-friction proof the four keys enjoy; *"more scrutiny is not
  automatically more safety."* Reframing bias (a thought that never forms) is **not fully catchable by
  output-diff tests** — the sketch itself calls this a research residual.
- **Local-first fit: yes.** No network/store/UI surface is *required* by the behavior itself (and none
  may be added to "support" it — that is a stop condition).
- **Required prerequisites:** the entire §8 acceptance criteria of the dissent-preservation sketch —
  structural rules in code, content validator, the dissent-canary suite as a merge gate, ranking/
  synthesis byte-identity, off-state golden, observability, override + rollback, trust-boundary
  persona-blindness — **plus** at least a first cut of the reframing-bias evaluation. Reviewed at
  **full** council level and maintainer-approved.
- **What it commits the repo to:** applied prompt-shaping behavior — the most trust-adjacent surface
  the project has built — and an ongoing obligation to keep the dissent-canary suite green forever.
- **What could go wrong:** a persona quietly narrows scrutiny and the council's core value (independent
  dissent) erodes *invisibly*; scope-creep from one hardcoded persona toward a `--persona` free-form
  flag or a persona schema; interaction with the `full`-mode fragility; or the honest outcome that the
  reframing-bias problem is unsolved and the behavior must be **canceled**, having spent a full line to
  learn it.

### B — open v0.10.x for dissent-preservation infrastructure only

- **Product value: med (foundational).** No user-visible feature, but it converts the sketch's "partly
  a research problem" into a *measured* answer and de-risks (or kills) candidate A cheaply.
- **Architecture risk: low–med.** Tests, fixtures, and byte-identity assertions over the *existing*
  council path — no prompt emphasis wired, no behavior. Reads the council output; does not shape it.
- **Security / trust risk: low.** Nothing is applied; the harness can only *observe*. It touches no
  guard/executor/trust code and adds no persona text to any prompt.
- **Local-first fit: yes.** Pure local tests/tooling; no store, UI, or network.
- **Required prerequisites:** the sketch (exists). A discipline that the harness stays observation-only
  and never becomes a covert applier.
- **What it commits the repo to:** a test/observability layer and an honest evaluation of the reframing-
  bias residual — the evidence base a later "ship / cancel A" decision needs.
- **What could go wrong:** it *looks* like busywork if it produces no feature; the differential-coverage
  probe may be noisy/inconclusive across models (which is itself a finding); risk that "infrastructure"
  quietly grows a persona applier under the banner of "just for the test harness" (a stop condition).

### C — scoped v0.9.2 dogfood/tooling patch (no new surface)

- **Product value: med.** Delivers the "it's solid and pleasant to use" story; turns the v0.9.0
  application + v0.9.1 docs from "passes tests" into "trusted on real work." Precedented (v0.3.1,
  v0.5.1, v0.7.1 were exactly this shape).
- **Architecture risk: low.** Small, additive, reversible edges over shipped commands; no new command
  surface; no persona/schema/trust change.
- **Security / trust risk: low.** Preferences stay suggest/warn-only; guard/executor stay
  preference-blind; no new store/network/UI.
- **Local-first fit: yes.**
- **Required prerequisites:** a real dogfood pass to *find* the rough edges — and the honest caveat
  that the secondary-account audit found v0.9.1 **clean**, so there is **no known defect** driving this
  yet. A v0.9.2 needs findings before it has content.
- **What it commits the repo to:** little — the optional patch slot the architecture plan reserved.
  Low blast radius, easily deferred.
- **What could go wrong:** manufactured work — a hardening patch with nothing to harden is polish for
  its own sake; the additive-only discipline can erode into a "small new flag" that is really a new
  surface.

### D — pause engineering, do product-surface / commercialization planning

- **Product value: med–high (strategic).** Answers the question underneath all the others — *is
  persona/review-lens even the right next surface, or is the leverage in positioning / first-paid-MVP /
  licensing?* The commercialization prior art already council-marked personas as a **later, low-stakes
  upsell** and licensing/provenance as **"Question 0."**
- **Architecture risk: none (no engineering).**
- **Security / trust risk: none in-repo.** The only risk is committing private/commercial detail into
  the public vault — which the no-stage list already forbids.
- **Local-first fit: n/a (strategy).**
- **Required prerequisites:** none hard; the two historical `docs/plans/` files + the open-core
  decision records are the inputs. Keep commercial detail **private/local**, never committed.
- **What it commits the repo to:** nothing engineering; it *unblocks* a confident later choice among A,
  B, C by settling whether the next surface should be persona-shaped at all.
- **What could go wrong:** analysis-paralysis / treading water; strategy without a validated-demand
  signal (the commercialization council's own top risk); or leaking private commercial detail into a
  public doc.

### E — no-new-surface maintenance only

- **Product value: low now.** Preserves a clean, released, audited state; spends no risk budget.
- **Architecture risk: none.**
- **Security / trust risk: none.**
- **Local-first fit: yes.**
- **Required prerequisites:** none.
- **What it commits the repo to:** nothing — an explicit "hold" so the released line stays the stable
  baseline until a real driver (a bug, demand, or a decided direction) appears.
- **What could go wrong:** stalls momentum; the honest counter-risk is that "hold" can hide indecision
  that D would resolve more usefully.

## 5. Single council decision question

> **Is any candidate worth opening a v0.10.x line now, or is the safe next step a scoped v0.9.2 patch /
> a pause for product-surface planning / no-new-surface maintenance?**

Sub-questions for the council to weigh:

1. Given that persona/lens **behavior** (A) carries an *unsolved* research residual (reframing-bias
   evaluation) that the dissent-preservation sketch itself flags, is it premature to open A now — and
   if a line opens, should it be **B (infrastructure/measurement first)** rather than A?
2. Is there enough *evidence of a rough edge* to justify a v0.9.2 (C), given the audit found v0.9.1
   clean and there is no reported defect?
3. Does the real leverage lie in **product-surface / commercialization planning (D)** — settling
   whether persona/review-lens is even the right next surface — before any engineering line opens?
4. Or is **no-new-surface maintenance (E)** the honest answer: the released state is good, and nothing
   should open until a concrete driver appears?
5. What is the single biggest risk the framing above **underweights**?

## 6. Hard scope guard (binding on this gate and any work it authorizes)

This gate authorizes **only** the framing document and the balanced council-style review command below.
It permits, in this task:

- **No code.** No file under `backend/` changes; no test changes beyond none; no runtime of any kind.
- **No provider/model/API calls except the single council review command** in §8 (a real
  `vibe review --preset balanced --usage` over this file). No other model/provider/network call.
- **No guard / executor / Workbench / trust changes.** The deterministic trust boundary, guarded
  executor, command allowlist, and Workbench panel are untouched and preference/persona-blind.
- **No `schema: 2`** and no schema field of any kind; schema v1 stays frozen.
- **No prompt / ranking / synthesis changes.** The council collect/rank/synthesize path is untouched.
- **No persona behavior.** No prompt emphasis, no `--persona`/`--lens` flag, no persona preset object,
  no persona validator — nothing from the dissent-preservation sketch is *built* here.
- **No `.council/profile.*`** local store (none exists; none is created).
- **No UI / dashboard.** The only local UI (the Workbench panel) is untouched.
- **No hosted / network / LAN / mobile** surface of any kind.

Choosing a candidate in §5 does **not** authorize building it — each candidate has its own gate
(A/B require a full-review v0.10.x line; C requires a scoped v0.9.2 plan; D/E require no PR). This
document decides *direction*, not implementation.

## 7. Criteria for each verdict (what happens next per outcome)

The production band determines the follow-on step; this gate only selects which branch is taken:

- **If the council converges on a v0.10.x surface (A or B):** the next step is a **Fable architecture
  pass over this exact file** — Fable reads the council-produced files and produces the
  architecture/risk/PR breakdown; Opus/Sonnet then implement the scoped PRs. **No implementation starts
  from this gate.** For A specifically, the §8 acceptance criteria of the dissent-preservation sketch
  are prerequisites, and the line is reviewed at **full** council level.
- **If the council converges on v0.9.2 (C):** the next step is a **small dogfood/tooling plan** (a real
  dogfood pass to find rough edges first, then a scoped, additive, no-new-surface patch). **Fable is
  not used** for a routine patch — Opus/Sonnet plan and implement per the budget policy.
- **If the council says pause / product-surface planning (D):** **no implementation PR starts.** The
  next step is a product/strategy pass (kept private/local where commercial), not an engineering line.
- **If the council says no-new-surface maintenance (E):** hold — nothing opens until a concrete driver
  (a real bug, validated demand, or a decided direction) appears.
- **If the council is split or shallow:** **do not proceed.** Report the split and ask whether to run a
  fuller review. Per the project's review policy, `vibe full` is reserved for genuine Workbench / trust
  / executor / security-boundary ambiguity (and carries the documented None-content ranking-parser
  fragility), so it is **not** run automatically here.

## 8. Council guidance summary

_Human-curated reading of the balanced council-style review (raw `.council/` output stays
local/gitignored; the below paraphrases, it does not paste). The review was **deep, decisive, and
non-conflicted** — it converged on one direction rather than splitting, so `vibe full` was **not** run
(and would not be: `full` is reserved for genuine Workbench/trust/executor/security-boundary ambiguity
and carries the documented None-content ranking fragility)._

- **Review command:** `uv run python -m backend.cli review --preset balanced --usage --file docs/fable/v0.10.x-product-surface-framing.md`.
- **Verdict — do NOT open a v0.10.x line now.** The council read the framing's own evidence back at it:
  v0.9.1 is released and audit-clean, there is **no reported defect**, and — critically — there is **no
  validated demand** for persona behavior (no user requests, no dogfooding findings, no commercial
  signal; *"long-advertised direction" is not validation*). That is a **"hold and plan"** signal, not a
  five-way engineering trade-off. It explicitly cautioned that the elaborate gate risks **process
  ossification** — governance overwhelming purpose — and that the strategic questions in §5 are
  **product decisions, not code-review questions** an AI council is positioned to answer.
- **Chosen next step — pause for lightweight product-surface / commercialization planning (Candidate
  D), then hold (Candidate E) with a concrete trigger.** The council's concrete sequence: (1) spend a
  few days synthesizing the existing (private) commercialization context to decide whether persona
  behavior belongs in the near-term product/revenue plan and *what would count as a demand signal*;
  (2) if "personas later," record that and **close the v0.10.x question with a time-boxed revisit
  trigger** (e.g. revisit at 90 days or on X real user requests / Y commercial signal); (3) only if
  planning says "personas soon" open a **narrower** gate — on reframing-bias **solvability** — and even
  then prefer measurement/redesign over shipping behavior.
- **Top arguments (adopted):** no validated demand + an acknowledged **unsolved research problem**
  (reframing-bias evaluation) + a clean release ⇒ plan, don't build; the gate's own five-candidate
  structure creates **false balance** and defers the real work (demand/commercial validation) behind
  more process; run strategy as a **parallel track**, not a blocking gate, so engineering isn't frozen;
  keep a human maintainer **veto / escape hatch** — the council's verdict is an input, never an
  autopilot.
- **Top risks (adopted):** (1) **process ossification / opportunity cost** — meta-work crowding out
  shipping and demand validation; (2) **A and B both depend on an unsolved research residual** — the
  council would collapse them toward a single "not yet," and warned Candidate B (infra-only) can drift
  into a covert applier and its measurement probes may be **noisy/inconclusive**; (3) **Candidate C is
  not really a candidate** — "no known defect → needs findings before it has content" makes v0.9.2 a
  placeholder, not a safe answer; (4) **recursive-trust fallacy** — using an AI council to vet
  AI-behavior safety is a closed loop; keep human judgment authoritative; (5) the risk lens is **too
  narrow** (dissent-suppression only) — persona *misuse*, prompt-injection robustness, and market
  misalignment are under-weighted.
- **Should Fable run next? — No.** No v0.10.x surface was chosen, so there is nothing for a Fable
  architecture pass to consume. Fable is not used for a product/strategy pause.
- **Is v0.9.2 safer? — No / not as the answer.** The council rated C a **placeholder** with no driving
  defect; a dogfood/tooling patch needs real findings first. It is available later *if* a rough edge
  actually surfaces, but it is not the safe next step now.
- **Is a pause / product-planning pass safer? — Yes.** This is the council's lead recommendation
  (Candidate D → E), run **lightweight and human-led**, with commercial detail kept private/local and a
  concrete revisit trigger so "hold" does not become indefinite drift.
- **Review cost:** provider-reported **$0.25** (`prompt=30352 completion=14726 total=45078`; balanced
  preset, one pass). No `full` run.

**Net:** the safe next step is **not** v0.10.x and **not** a manufactured v0.9.2 — it is a short,
human-led product-surface / commercialization planning pass (kept private/local), after which the most
likely outcome is an explicit, time-boxed **hold** on persona behavior until demand and the
reframing-bias research question are resolved. Persona/lens **behavior** stays deferred (and remains
**cancelable, not merely deferrable**, per the dissent-preservation sketch). No implementation PR
starts from this gate.
