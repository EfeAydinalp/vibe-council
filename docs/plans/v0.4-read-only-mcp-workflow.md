# Plan: v0.4 — read-only MCP / Claude Code workflow

A **planning document**, not an implementation. v0.3.0 shipped the local-first decision-memory /
context loop; v0.3.1 hardened it (decision-CLI fixes, `context check` 21/21, CLI UX polish, dogfood
notes); release-link hygiene is done (tag-pinned absolute links in v0.3.0/v0.3.1 release bodies).
With that base stable, v0.4 can be scoped on its own.

Review date: **2026-07-01**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local. **No MCP server code is written in this task.**

> **Product sentence.** v0.4 makes vibe-council usable as a **read-only project-memory/context
> server** for Claude Code and local agents, **without granting write/action authority.**

> **Scope guard.** v0.4 is **read-only first**: no write-capable MCP tools, no decision promotion,
> no file/git/shell mutation, no remote approval transport, no hosted/team/sync, no
> dashboard/mobile, no vector DB, no LLM eval, no token-aware tokenizer (unless explicitly scoped
> later), no provider/model behavior changes. License/provenance remains the unresolved **"Question
> 0"** commercial gate — untouched here.

---

## 1. Purpose

Expose the **already-curated** project memory and the **already-generated** context pack through a
**read-only MCP interface** so Claude Code and local agents can query the project's memory directly
instead of pasting a file. The data already exists and is deterministic; v0.4 only adds a read-only
*access surface* over it. Nothing new is summarized, retrieved by vectors, or written back.

Queryable surface (all read-only):

- project status (`STATUS.md` snapshot)
- curated decisions (list + individual records)
- rejected alternatives (aggregated from curated decisions)
- current constraints / safety boundaries
- next actions (from status)
- release/docs metadata (versions, release-note pointers)
- context-pack summaries + health (the deterministic `context check` result)

## 2. Non-goals

Explicitly **out of scope** for v0.4 (deferred or permanently excluded):

- **No write-capable MCP tools** — no promotion, no file/edit/delete, no git, no shell, no deploy,
  no email/notification, nothing that mutates repo/file/system state.
- **No decision promotion through MCP** — promotion stays a human-reviewed CLI step.
- **No remote approval transport / operator action over MCP.**
- **No hosted/team/sync layer; no dashboard, mobile, or custom remote transport.**
- **No vector DB / embeddings / hybrid retrieval.**
- **No LLM-based context eval** — `context check` stays deterministic.
- **No token-aware tokenizer / budget rework** unless a later PR explicitly scopes it.
- **No provider/model behavior changes; no new runtime dependencies without explicit scope.**
- **No council run** triggered by MCP.

## 3. User story

> As a developer using Claude Code on this repo, I want the agent to read the project's curated
> decisions, current status, constraints, and the latest context pack **on demand** — so its answers
> reflect *why* past calls were made — **without** the agent being able to change files, promote
> decisions, run commands, or touch git. The agent reads project memory; I remain the only one who
> writes it.

Secondary: a local agent (not Claude Code specifically) can attach to the same read-only server for
the same memory, with the same boundaries.

## 4. Proposed read-only MCP resources / tools

The exact MCP shape (resources vs. tools) is an implementation choice for PR #56–#57; this lists the
intended *surface*. Everything here is read-only and deterministic.

### Resources (preferred for static reads)

| URI | Returns | Source |
|-----|---------|--------|
| `vibe://status` | project status snapshot | `docs/context/project/STATUS.md` |
| `vibe://decisions` | curated decision index (id, date, status, title, tags) | `docs/decisions/*.md` |
| `vibe://decisions/{id}` | a single curated decision record | `docs/decisions/<id>.md` |
| `vibe://context/latest` | the latest generated context pack | `.council/context/pack-latest.md` (local) |
| `vibe://rejected-alternatives` | aggregated rejected alternatives | curated decisions |
| `vibe://release-notes` | release metadata + tag-pinned note pointers | `docs/releases/*.md`, `CHANGELOG.md` |
| `vibe://constraints` | current constraints / safety boundaries | pack constraints + curated docs |

### Read-only tools (if the MCP client/impl needs callable tools)

- `list_decisions` — list curated decisions (optional `--tag`/`--status`-style filters).
- `show_decision` — return one curated decision by id (path-traversal guarded, `docs/decisions/` only).
- `get_project_status` — return the `STATUS.md` snapshot.
- `get_context_pack` — return the latest generated pack (or report it must be built first).
- `check_context_health` — return the deterministic `context check` result (`passed/total`, reasons).
- `list_rejected_alternatives` — return the aggregated rejected-alternatives view.

### Explicitly forbidden tools (must never exist in v0.4)

`promote_decision` · `write_file` · `edit_file` · `delete_file` · `run_command` · `git_commit` ·
`git_push` · `send_email` · `deploy` — and **anything else that mutates repo/file/system state**.
A v0.4 acceptance test should assert the server advertises **no** tool from this list.

## 5. Data sources and trust levels

| Source | Trust | MCP exposure |
|--------|-------|--------------|
| `docs/decisions/*.md` | **curated source-of-truth** (human-reviewed, redaction-linted) | exposed |
| `docs/context/project/STATUS.md` | curated status snapshot | exposed |
| `docs/releases/*.md`, `CHANGELOG.md` | curated public docs | exposed (metadata/pointers) |
| `.council/context/pack-latest.md` | generated, local/gitignored, redaction-scanned at build | exposed **locally only** |
| `.council/context/claude-code-context.md` | generated, local/gitignored | not a distinct resource (pack is canonical) |
| raw `.council/` reviews/drafts | local/gitignored, **may be unredacted** | **never exposed** |
| untracked private plans (e.g. commercialization plan) | private/local | **never exposed** unless explicit opt-in |
| `.env`, `.venv/`, `data/`, secrets, `.obsidian/` | secret/local | **never exposed** |

Principle: MCP exposes **curated, public, redaction-clean** content by default; generated packs are
exposed **locally** (the server runs on the user's machine); raw/private/secret content is never
exposed. Anything beyond curated+generated requires an explicit, documented opt-in.

## 6. Safety and privacy boundaries

- **Read-only by construction** — the server exposes no mutating tool; forbidden-tool list (§4) is
  enforced by a test.
- **Source-of-truth preserved** — `docs/decisions/*.md` and `STATUS.md` remain the only writers'
  surface (CLI + human review); MCP never writes them.
- **Local-first** — the server runs locally; the generated pack stays gitignored; nothing is sent
  anywhere the user didn't trigger.
- **Redaction reuse** — any text served goes through (or has already gone through) the existing
  redaction guard; the server must not serve content with a critical redaction finding.
- **No path traversal** — `show_decision`/`vibe://decisions/{id}` resolve to `docs/decisions/` only
  (reuse the CLI's existing guard).
- **No private/untracked exposure** — untracked plans, `.env`, `.venv/`, `data/`, raw `.council/`
  outputs, `.obsidian/`, and secrets are never served; opt-in (if ever added) is explicit + documented.
- **No network/provider calls** — the read-only surface is deterministic and offline; MCP transport
  is local.

## 7. Suggested PR sequence (small, ordered, flexible)

> **Numbering note.** PR #55 was used for a pre-MCP context-pack budget/headroom stabilization;
> the MCP track therefore starts at **PR #56** below.

- **PR #56 — MCP design skeleton (contract + command help, no server).** ✅ *done.* A testable
  read-only **contract** in code (`backend/mcp_contract.py`): read-only resources/tools, the explicit
  forbidden-tool list, validation helpers, and a `vibe mcp contract` command that *describes* the
  surface. **Contract-only — no server, no socket, no MCP dependency.**
- **PR #57 — read-only MCP server skeleton.** ✅ *done.* Dependency-free read layer
  (`backend/mcp_server.py`) for `get_project_status` / `list_decisions` / `show_decision` (↔
  `vibe://status`, `vibe://decisions`, `vibe://decisions/{id}`) over the curated docs, plus a
  read-only `vibe mcp inspect` smoke. Path-traversal guarded; no `.council/`/private-plan reads;
  forbidden + deferred tools unreachable (tests). **Transport deferred:** a spec-compliant MCP stdio
  server needs the `mcp` SDK, so this PR ships the read layer + smoke and leaves the stdio transport
  (and that dependency decision) to the next increment.
- **PR #58 — context pack resource + health.** ✅ *done.* Read layer adds `get_context_pack` +
  `check_context_health` (↔ `vibe://context/latest`), reusing `context_pack.build_pack`/`check_pack`
  **in memory** — MCP context reads generate **no** `.council/` files (verified by tests). Surfaced
  via `vibe mcp inspect --context --health`. Still dependency-free and transport-free.
- **PR #59 — minimal read-only MCP stdio transport.** ✅ *done.* `backend/mcp_stdio.py` speaks
  newline-delimited JSON-RPC 2.0 over stdin/stdout (`initialize` / `tools/list` / `tools/call` /
  `resources/list` / `resources/read` / `ping`) for the **existing** read-only surface only, via
  `vibe mcp serve --stdio`. **No new dependency** — implemented in stdlib (not the `mcp` SDK), so the
  normal CLI is unaffected; forbidden/deferred tools stay unreachable and context reads write no
  `.council/` files (bounded smoke test).
- **(interlude) — context-pack critical-section budget fix.** The recurring 14000-char cliff was
  fixed structurally so required/core sections compact instead of dropping (its own PR before more
  docs grew the curated set).
- **Claude Code / MCP setup docs + v0.4 dogfood.** ✅ *done.* `docs/mcp/claude-code-setup.md`
  (generic MCP stdio client pattern — exact Claude Code syntax not asserted) + a dogfood pass
  (`docs/dogfood/v0.4-mcp-local-dogfood.md`) covering the stdio smoke, no-write audit, and privacy
  audit. Docs/dogfood only — no new MCP surface.
- **Next — v0.4 release prep.** CHANGELOG `[0.4.0]`, `docs/releases/v0.4.0.md`, version bump,
  decision record; tag/release as a separate, explicitly-requested step.

Keep PRs small and reversible. **Dependency note (resolved):** the MCP stdio transport was
implemented in **stdlib** (minimal JSON-RPC), so **no MCP SDK runtime dependency was added**. If
fuller MCP spec compliance is needed later, an *optional* `mcp` extra can be adopted without
affecting normal `vibe` usage.

## 8. Acceptance criteria

v0.4 is "done" when:

- A **local read-only MCP server** exposes the §4 resources/tools over curated decisions + status +
  the generated context pack.
- The server advertises **no** forbidden (mutating) tool — asserted by a test.
- `show_decision`/decision resources are **path-traversal guarded** to `docs/decisions/`.
- Served content is **redaction-clean** (no critical findings); raw/private/secret content is never
  served.
- Claude Code can register the server and answer repo questions from project memory **without** any
  write/action capability.
- Everything stays **deterministic, local-first, offline**; no provider/model/network behavior change.
- Docs (setup + boundaries) and v0.4 dogfood notes exist; release prep mirrors the v0.3.x process.

## 11. Open questions

1. **MCP SDK vs. hand-rolled** — does the read-only surface justify an MCP SDK dependency, or can a
   minimal stdlib local transport suffice? (Decide in PR #56 with the dependency rationale.)
2. **Resources vs. tools** — Claude Code reads MCP *resources* well for static content; do we still
   need callable tools, or are resources enough for the read-only surface?
3. **Pack freshness** — should `vibe://context/latest` serve the last-built pack as-is, or refuse if
   stale / not built (and tell the agent to run `context build`)?
4. **`check_context_health` exposure** — is surfacing the deterministic `context check` result useful
   to an agent, or noise?
5. **Opt-in scope** — if a user wants to expose more than curated+generated content, what is the
   minimal, explicit, documented opt-in (and does v0.4 ship any opt-in at all, or defer it)?
6. **Multi-repo / which project** — does the server bind to the caller's project workspace the way
   the CLI does, and how is that surfaced to the agent?
