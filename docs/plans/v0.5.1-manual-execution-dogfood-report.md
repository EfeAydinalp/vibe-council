## 2. How actions were seeded

No CLI/HTTP path exists (by design — see plan §10, "Agent-to-Workbench action bridge" is
explicitly deferred to v0.6+) to create a real `Action` + payload artifact other than the
public `backend.workbench_*` APIs the test suite itself uses
(`tests/test_workbench_executor.py::_setup` is the reference pattern). A small, throwaway
Python script (not part of the repo, not committed) called `workbench_orchestrator`/
`workbench_runtime`/`workbench_payloads` directly — the same functions PR #74–#81's own
test suite exercises — to seed, per task: a task, an approval (`requested_action=
"<kind>:<target>"`), an **approved** decision, and a linked `Action`. For file actions, a
payload artifact was built and saved via `workbench_payloads.build_payload_artifact` /
`save_payload_artifact`. Four actions were seeded this way:

| Action | Kind | Target/label | Purpose |
|---|---|---|---|
| A | `write_file` | `notes/dogfood-note.md` | happy-path file execution |
| B | `run_command` | `git status --short` | happy-path allowlisted command execution |
| C | `run_command` | `echo not-allowlisted` | negative: not on the resolver/trust allowlist |
| D | `write_file` | `notes/dogfood-missing-payload.md` | negative: approved action, no payload artifact saved |

The Workbench panel was then started against the temp project
(`VIBE_CALLER_CWD=<temp dir> python -m backend.cli workbench serve --port N`) and driven
entirely over its real HTTP API (`GET /`, `GET /api/state`, `POST /api/actions/<id>/execute`)
— the panel itself never knows or cares whether a browser or `curl` sent the request; the
server-side code path exercised is identical either way.

## 3. File action smoke result — **PASS**

- `GET /` before execution showed action A with an **Execute** button and
  `executable_reason: "approved, payload verified, ready to execute"`.
- `POST /api/actions/<A>/execute`, sent with a **crafted JSON body**
  (`{"target":"../evil.txt","content":"MALICIOUS OVERRIDE CONTENT","overwrite":true}`), still
  wrote exactly the server-resolved payload: `notes/dogfood-note.md` containing the original
  seeded content only, byte-for-byte. **No `evil.txt` was created anywhere** (checked the
  temp project and its parent directory). This confirms `handle_execute` genuinely never
  reads the POST body — the code inspection (it takes only `action_id`) is empirically
  verified, not just read.
- Action A's status moved `pending` → `completed`; `result_summary` read
  `"wrote 54 bytes to dogfood-note.md"` — readable, byte-count-only, no raw content.
- `payload_summary` in `/api/state` and the rendered HTML showed only
  `{"content_bytes": 54, "kind": "write_file", "overwrite": false}` — never the actual note
  text, in either the JSON API or the HTML.

## 4. Command action smoke result — **PASS**

## 9. Result display readability — **PASS, with one bug found and fixed (§10)**

- `write_file`/`run_command` success and the two blocked/negative cases all render a plain-
  English `result:` line (byte count or `exit=<code>`, or the specific blocked reason) —
  none of the four action cards showed a raw exception, a wall of text, or a cryptic code.
- **One bug found**: the rendered panel's detail line for a `run_command` action that was

  actively **misleading** — the card simultaneously showed a false "does not resolve" claim
  above a true "ran ... exit=0" result line. Root cause and fix in §10.

## 10. Bugs found and fixed (both minimal, both covered by the existing test suite)

Both are display/metadata defects — **neither changes what the executor is willing to run,
what the trust boundary allows, or any allowlist.** No scope growth per the plan's guard.

### 10a. `ExecutionResult.dry_run` stayed `True` for every real execution

`backend/workbench_executor.py`: `validate_execution_invariant()` always builds its result
with `dry_run=True` (correct — it's a dry-run preview function). `_real_execute()` reuses
that same result object for the real-execution path but never corrected the flag, so **every**
real `/api/actions/<id>/execute` response — success, blocked, or timeout — reported
`"dry_run": true` alongside `"executed": true"`, a self-contradictory pair visible to any API
consumer (reproduced directly in §3/§4/§5/§6 above before the fix). Fix: `_real_execute` now
sets `result.dry_run = False` immediately after building `result`, before any return path —
one line, consistent with how every other field on that same object (`executed`, `blocked`,
`reason`, …) is already mutated in place throughout this function. Verified live: re-executed
a freshly-seeded command action against the running (fixed) server and confirmed
`"dry_run": false` in the response.

### 10b. Panel showed a false "does not resolve" message for a completed/blocked `run_command` action

`backend/workbench_panel.py`: `_action_view` only builds a `command_preview` for a `pending`
action (dry-run only, by design). `render_html`'s fallback branch for "no preview" didn't
distinguish *why* there was no preview — pending-and-genuinely-unresolvable (message is
accurate) vs. already-executed-so-there's-no-preview-to-show (message is false). Fix:
the fallback now only shows the "does not resolve" text while the action is still `pending`;
for any other status it shows the action's own recorded target/label instead, deferring to
the (already-correct) `result:` line for the actual outcome. Purely a display-string change —
no security/executor logic touched. Verified live: re-rendered `/` against the running
(fixed) server and confirmed the misleading string no longer appears anywhere in the HTML.

### Considered and declined (recorded for transparency, not applied)

A `balanced`-preset `vibe diff` review of this fix (see §13) recommended two further changes:
refactoring `validate_execution_invariant()` to take an explicit execution-mode parameter
instead of `_real_execute` correcting the flag post-hoc, and hiding the command label
(`act['target']`) from non-pending `run_command` action cards as a "sensitive data" concern.
Both were considered and **not applied** in this PR:

- The mode-parameter refactor would touch the core invariant-validation function itself,
  which is a materially larger, more invasive change than a hardening pass calls for (the
  plan's scope guard is explicit: "no executor/panel behavior changes... fix it minimally").
  The one-line correction in `_real_execute` is consistent with this function's existing,
  established pattern — mutating fields on the shared `ExecutionResult` object after
  `validate_execution_invariant` builds it (see `_fail`, `_do_write`, `_do_run_command`,
  all of which already do exactly this) — not a novel or riskier approach.
- The "hide the command label" suggestion doesn't hold up against this codebase's own
  documented model: `workbench_commands.py` explicitly documents argv/labels as
  "fixed/non-secret by construction... safe to log/display", and `act['target']` for
  **every** action kind (not just `run_command`) is already shown unconditionally in the
  card's `<b>` title regardless of status — selectively hiding it only in the detail line
  for one kind/status combination would be inconsistent, not more secure.

Left as a known follow-up if a future maintainer wants the more invasive refactor; not
blocking this PR.
