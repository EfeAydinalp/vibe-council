## 1. Purpose — why personalization exists

vibe-council now has a safe execution core (v0.5), an agent-to-Workbench proposal bridge (v0.6.0),
a role-aware onboarding guide layer (v0.6.1), a local-first Markdown project vault (v0.6.2), and a
read-only context export (v0.6.3). Personalization is the next phase because it **adjusts an existing
loop** rather than creating one. Concretely, it exists to:

- **Reduce repeated explanation across agents and projects.** Today, a user (or an agent) re-states
  the same preferences — preferred review preset, PR/commit style, sensitive paths, verbosity — every
  new session or repo. A profile captures those once so generated guides/exports/context can carry
  them forward.
- **Preserve user and project preferences durably.** Working style, spending style, review
  strictness, approval thresholds, and recurring workflows become stated, versionable facts instead
  of tribal knowledge re-derived each session.
- **Make generated prompts / guides / context exports more tailored.** `vibe guide`, `vibe context
  export`, and future onboarding output can read the profile and emit output shaped to how this user
  and this project actually work — without changing what those commands are allowed to do.
- **Keep memory local-first and human-readable.** Personalization stays **plain, committed-or-local
  Markdown** in the same spirit as the vault: readable in GitHub/VS Code/Obsidian/any editor, diffable,
  and owned by the user — not locked in a database or an opaque store.
- **Avoid a vector DB or database too early.** Personalization is a small, curated set of preferences.
  It does not need indexing, embeddings, or a server. A vector DB, if it ever appears, is an **optional
  search/index helper layered on top of the Markdown**, never the source of truth (see §3).

Personalization is a **convenience and consistency layer**, not a security or capability layer. It can
make the tool feel like it knows the user; it can never make the tool do more than the guarded core
already allows.

> **Scope discipline (balanced-review caution).** The review's main non-security critique was that the
> plan is more elaborate than v0.7's modest functional payload (Markdown scaffold + doctor check +
> export pointers + advisory guide text), and that the need is so far dogfood-anticipated rather than
> demand-proven. It even offered "defer v0.7 and dogfood v0.6.x first" as an alternative. The
> maintainer has directed that v0.7 planning proceeds — so the discipline this imposes is: **ship the
> smallest thing that helps.** PR A should build only the preference fields a real workflow already
> needs, collapse overlapping layers (§4), and prefer one committed file over several (§5) unless PR-A
> review finds a compelling reason to split. The nine layers in §4 are the **design space**, not a
> build list.

## 2. Non-goals (explicit — v0.7 does NOT do any of these)

These are as binding as the goals. A PR that does any of them is out of scope and should stop.

- **No vector DB in v0.7.** No embeddings, no similarity search, no index server for preferences.
- **No PostgreSQL / Redis / Docker / Next.js / LangGraph adoption in v0.7.** No new platform stack.
- **No hosted / SaaS implementation.** Personalization is local-first; nothing is served or synced.
- **No cloud sync** of profiles or preferences. The profile never leaves the machine except through an
  explicit, consented, user-driven step — and v0.7 does not build that step.
- **No secret storage.** Profiles/preferences never hold API keys, tokens, or credentials (see §5).
- **No automatic execution.** Personalization cannot trigger, schedule, or auto-approve any Workbench
  action. Approval remains separate from execution (invariant #1–#2).
- **No Workbench trust-boundary relaxation.** The deterministic trust boundary
  (`workbench_trust.py`) ignores preferences entirely. Preferences add friction on top; they never
  subtract it (invariant #17).
- **No `/council` real command.** `/council` stays a documented future host-command idea; `vibe` is
  the real CLI. v0.7 does not make `/council` real.
- **No model / provider / API calls** from any personalization command. Reading/writing a profile is
  deterministic and offline, like `vibe project doctor` / `vibe context export`.
- **No personalization that can loosen a safety rule.** A preference that would widen the command
  allowlist, relax a path deny, enable auto-execute, expand the network surface, or disable
  Host/token checks is **not a setting — it is a bug** and must be rejected.
- **No model ever sees the personal profile.** Preferences are never injected into a system prompt or
  sent to a provider as part of any model call. The personal profile is local/gitignored and stays on
  the machine; v0.7 builds no path that transmits it.

Deferred-but-adjacent (explicitly not v0.7): Skills/`SKILL.md` implementation (§10), any hosted/
open-core stack (§11), mobile/LAN/voice (v0.8), hosted/team (v0.9+).

## 3. Current memory model (canonical, as of v0.6.3)

v0.7 must build **on** this model, not replace it. The canonical approach today:

- **Markdown is the source of truth.** All durable project memory is plain, committed, human- and
  agent-readable Markdown. No database, no proprietary store.
- **`docs/context/project/*` is the project vault** — curated, committed, public-safe project memory:
  `README.md` (landing), `STATUS.md` (current state — read first), `ROADMAP.md`, `DECISIONS.md`
  (index/pointer into `docs/decisions/`, **not** a canonical store), `PROGRESS.md`, `RISKS.md`,
  `WORKFLOWS.md`, `NOTES.md`.
- **`docs/fable/*` holds phase / architecture briefs** — the phase-by-phase plan (this file is one of
  them), operating rules, security invariants, and roadmap.
- **`docs/releases/*` and `CHANGELOG.md` are release history** — what shipped, when, and with what
  safety posture.
- **`docs/context/agent-brief.md` and `docs/context/project/STATUS.md` are the compact current-state**
  surfaces agents read first.
- **The context pack / export is a budgeted projection / handoff**, not a store: `vibe context build`
  assembles a deterministic, char-budgeted pack (STATUS + decision index + constraints, `context
  check` 21/21) into gitignored `.council/context/`; `vibe context export --for <agent>` prints a
  read-only onboarding handoff. Both are **derived** from the Markdown, never authoritative over it.
- **`docs/decisions/*` is the canonical decision store**; the vault's `DECISIONS.md` only points at it.
- **A vector DB is later and optional** — an eventual **search/index helper** over the Markdown, if
  plain retrieval ever proves insufficient. It is **never** the source of truth and is out of scope
  for v0.7.

Personalization files (§5) join this model as **more curated Markdown**: project-level preference
files are committed and public-safe like the vault; a personal profile stays local/gitignored like
`.council/` runtime state. Neither becomes a database, and neither is injected into the budgeted
context pack in a way that reopens the 21/21 budget (the vault precedent: read directly / via
pointers, not dumped into the pack).

## 4. Personalization layers (proposed, all tighten-only)

Personalization is not one flat setting bag; it is a small set of clearly-scoped layers. Every layer
below is **advisory or tighten-only** — none can widen a guardrail.

1. **Global user preferences** — how *this user* likes to work regardless of project: verbosity,
   plan-first vs. dive-in, default explanation depth. **Local/gitignored** (personal, machine-scoped;
   see §5). Never committed.
2. **Per-project profile** — facts about *this project*: identity summary, stack, conventions,
   review policy, this project's sensitive paths. **Committed, public-safe** (team-shared).
3. **Per-agent role preferences** — default role per agent (e.g. Claude=coder, Fable=planner) and
   agent-specific tone. Shapes `vibe guide` / `vibe context export` output only.
4. **Council usage policy** — default review preset per project (cheap/balanced), when balanced/full
   is warranted, whether `--usage` is always on. Tighten-only: a project may *require* a stricter
   preset; it may not authorize `premium`/`--allow-premium` implicitly.
5. **Fable budget / usage policy** — restate the binding rule: Fable is architect/technical-lead
   only; Opus/Sonnet implement routine PRs; no broad Fable repo scans. A preference may make this
   *stricter* (e.g. "never Fable for this repo"), never looser.
6. **No-stage / private-artifact policy** — the project's never-stage list (extends the canonical one
   in [`WORKFLOWS.md`](../context/project/WORKFLOWS.md)). Additive only — it can add paths to never
   stage; it can never remove a protected path.
7. **Review preset policy** — default `vibe review`/`vibe diff` strictness and preset per change type
   (docs vs. code vs. security). Tighten-only.
8. **Release workflow policy** — commit-message convention, PR body shape, co-author line, required
   pre-release gates. Advisory formatting only; it can never skip a gate or authorize a tag/release.
9. **Workbench action-proposal policy** — which proposal kinds this project expects, which command
   labels it actually uses (a **narrowing** of the existing allowlist, never a widening), default
   risk threshold at which to demand extra confirmation. **Never widens** the trust allowlist or
   resolver allowlist (invariant #10–#11, #17). **Narrowing is advisory-display-only and must never
   remove a safety-critical command** (e.g. `vibe review`, `vibe lint --redaction`, `vibe project
   doctor`): the trust boundary/resolver remain the real gate — a preference cannot disable a gate,
   only annotate which labels a project routinely uses.

> **Layers 4 and 7 overlap** (council usage vs. review-preset strictness) and **layer 5 (Fable
> budget) is closer to an enforced project rule than a tunable preference.** That is deliberate for a
> planning enumeration — v0.7 does **not** implement all nine layers. PR A should collapse overlapping
> layers into the smallest concrete set it actually ships (the balanced review flagged the nine as
> over-granular for v0.7's scope); treat this list as the design space, not a build list.

Layering / precedence (proposed): **project (committed) tightens on top of global (personal)**, and
**neither can loosen the trust boundary**, which ignores all of them. For **guardrail-relevant,
ordered** preferences (e.g. review preset: `full` > `balanced` > `cheap`), the **stricter** wins
(tighten-only makes "stricter wins" always safe). For **non-ordered / non-comparable** preferences
(e.g. a PR-body template or commit convention where neither is "stricter"), **project-level always
overrides global/personal** — a deterministic, documented precedence, never a silent merge.

## 5. Data-model proposal (Markdown-first)

Prefer plain, curated Markdown files — consistent with the vault. Proposed minimal structure:

| File | Location | Committed? | Contains |
|---|---|---|---|
| `PROFILE.md` | `docs/context/project/PROFILE.md` | **Committed, public-safe** | Per-project identity/profile: what this project is, its stack, conventions, review policy pointer, this project's sensitive paths, default agent roles. Team-shared. |
| `PREFERENCES.md` | `docs/context/project/PREFERENCES.md` | **Committed, public-safe** | Project-level, team-agreed preferences: default review preset, no-stage additions, release/PR style, Workbench proposal policy (tighten-only). |
| `AGENTS.md` | `docs/context/project/AGENTS.md` *(or reuse the repo-root `AGENTS.md` guide target — decide in §9)* | **Committed, public-safe** | Per-agent role preferences and agent-specific guidance the guide/export layer reads. |
| Personal profile | `.council/profile.json` (or `.council/profile.md`) | **Local / gitignored** | The *global user* preferences of §4.1 — this user on this machine. Never committed, never sent to a model. Mirrors the `10-personalization-layer.md` "personal (local)" store. |

A leaner alternative is acceptable if it proves better in the PR-A design review: e.g. a single
committed `PREFERENCES.md` (or `PROJECT.md`) that also carries the profile fields, with
`PROFILE.md`/`AGENTS.md` folded in — decide during PR A with a `vibe review --preset cheap` pass. The
balanced review **recommended consolidating** the committed files (their content boundaries are fuzzy:
all three are "project-level, committed, public-safe") and, for the **local profile specifically**,
considering a structured-but-readable format (e.g. TOML via stdlib `tomllib`) over ad-hoc Markdown to
avoid implicit-schema brittleness. Treat both as **strong PR-A inputs**, not decisions made here.
**Whatever the shape, keep it minimal, Markdown-first for the committed vault files, and
vault-consistent.** Do not introduce a database or a new top-level directory (`.vibe/` is explicitly
discouraged; extend `docs/context/` — per [`08-obsidian-project-vault.md`](08-obsidian-project-vault.md)).

**Threat-model note (committed preference files are public metadata).** `PROFILE.md`/`PREFERENCES.md`
are committed and world-readable, so their contents (sensitive-path lists, review thresholds, team
habits) are reconnaissance-adjacent even though they hold no credentials. Keep them **generic and
public-safe**: state *that* a class of paths is sensitive without enumerating an exploitable map of the
crown jewels, and prefer policy over specifics where the specific would aid target selection.

**What must NEVER go in any personalization file (committed *or* the local profile):**

- Secrets, API keys, tokens, credentials.
- Precise private local data (absolute machine paths, usernames-in-paths, private hostnames) unless
  it is intentionally public-safe.
- Raw model / council outputs.
- Runtime payloads (`.council/runtime/payloads/*`).
- `.council/` runtime artifacts of any kind.
- Private local plans (`docs/plans/commercialization-and-hosted-platform-feasibility.md`,
  `docs/plans/v0.3.1-hardening-and-dogfood.md`).
- Health / medical / political / otherwise sensitive personal facts **unless** the user explicitly
  requests it *and* it is safe and appropriate for that file's committed/local scope. Default: omit.

The committed files (`PROFILE.md`/`PREFERENCES.md`/`AGENTS.md`) are subject to `vibe lint --redaction`
just like the rest of the vault — a critical finding blocks the commit. The local profile is
gitignored, but as **defense-in-depth** (the balanced review's recommendation) PR A should consider
adding `.council/profile.*` to the redaction checker's scope anyway, so an accidental commit or a
buggy read is still caught. Either way the local profile must never be sent to a provider.

## 6. Safety invariants (personalization-specific restatement)

These restate and specialize [`03-security-invariants.md`](03-security-invariants.md) for v0.7. None
may be weakened.

- **Personalization may tighten rules but never loosen security** (invariant #17). This is the one
  hard rule. A preference that would loosen a guard is a bug, not a feature.
  - **Concrete test for "tighten-only":** a preference is allowed only if it **cannot cause an
    action, command, or approval to succeed that would have been rejected without the preference.**
    Tightening (stricter), keeping (neutral), and adding (new constraints, e.g. more sensitive paths)
    are allowed; loosening (widening an allowlist, relaxing a deny, lowering a gate) is prohibited.
    Every v0.7 PR that touches a preference-reading path must carry this as an explicit review
    checklist item, and the new committed preference files must be covered by `vibe lint --redaction`
    (confirm/extend in PR A).
- **The deterministic trust boundary remains the authority.** `workbench_trust.py` ignores
  preferences entirely; preferences never feed into allow/deny/allowlist decisions.
- **The Approval Auditor remains advisory.** Preferences may make its summary stricter/louder; they
  can never lower risk or unblock.
- **Approval remains separate from execution.** No preference enables auto-execute, auto-approve, or
  a "trusted action" fast-path. Execution stays a distinct explicit step (invariant #1–#2).
- **The browser/client never sends payload or argv for execution** (invariant #3). Personalization
  does not add any client→server content channel; the execute request still carries only an action id.
- **Server-side ids/hashes stay server-side** (invariant #4). Preferences never let an agent or the
  browser supply ids, hashes, argv, cwd, env, or timeout.
- **Raw payload stays in payload artifacts only** (invariant #9). No preference surfaces or commits
  raw payload content.
- **No arbitrary shell** (invariant #10). Commands stay exact allowlisted labels → fixed argv,
  `shell=False`.
- **No command-allowlist expansion unless explicitly scoped** (invariant #10–#11). A project
  preference may **narrow** the labels it uses; it may never add one. v0.7 does not grow the allowlist.
- **Local-first caveat (state it plainly):** local artifacts stay local, **but cloud model calls may
  send prompts / files / diffs** to the provider unless a local provider (Ollama) is used. The
  personal profile itself is never auto-sent — but the user should understand that `vibe review`/
  `diff` (when using a cloud provider) do transmit the content they review. Personalization does not
  change this; it must not silently start sending the profile as part of any model call.

Verification after any v0.7 PR that touches Workbench-adjacent code: re-read
[`03-security-invariants.md`](03-security-invariants.md) against the diff; if any guarantee is weaker,
the PR is wrong. (Most v0.7 PRs are docs/read-only generator changes and touch no Workbench code at
all — keep it that way.)

## 7. Council-in-the-loop production system

The workflow this project runs by, and that v0.7 must operate within:

1. **User / product direction.** The maintainer sets direction and holds final authority.
2. **Council debates / reviews / criticizes.** `vibe review`/`vibe diff` (cheap/balanced/full) give
   an independent multi-model second opinion on plans and diffs. Advice, not authority.
3. **Council / Opus / Sonnet creates the phase brief if one is missing.** (This document is exactly
   that step for v0.7 — authored by Opus/Sonnet, reviewed by the council.)
4. **Fable reads only the curated brief / relevant context, as an expensive architect, if approved.**
   Fable does not scan the repo broadly; it reads the phase brief + minimal reading set, produces an
   architecture review or PR breakdown, and stops.
5. **Opus / Sonnet implements the PRs**, one small reviewable PR at a time.
6. **Council reviews plans and diffs** with cheap/balanced (full only for major architecture/security
   calls).
7. **The user is the final authority** — approves, merges, tags, releases. No model does those
   autonomously.

Binding constraints on model use (from
[`v0.6-followup-implementation-plan.md`](v0.6-followup-implementation-plan.md),
[`13-fable-implementation-playbook.md`](13-fable-implementation-playbook.md), and
[`RISKS.md`](../context/project/RISKS.md)):

- **Fable is not the routine implementer.** Routine PRs and docs work are Opus/Sonnet.
- **No broad Fable repo scans.** Fable reads the curated brief and the phase's minimal reading set —
  not the whole tree.
- **Fable creates the architecture / phase plan, then stops.** It does not carry on into
  implementation.
- **Routine implementation is done by Opus/Sonnet**, in small PRs, human-merged.
- **Every task prompt must carry a model header:** `MODEL: FABLE CODE` or `MODEL: OPUS/SONNET CODE`,
  so the intended model/budget is unambiguous before work starts.

## 8. Proposed PR breakdown for v0.7

Small, individually reviewable PRs, in dependency order. Each is **implemented by Opus/Sonnet** unless
the maintainer explicitly routes an architecture pass to Fable. Verify every PR with the standard gate
set (tests, `vibe lint --redaction`, `vibe decisions lint`, `vibe context build`/`check` 21/21, `vibe
mcp inspect --context --health` 21/21, `git status --short` clean).

### PR A — project PROFILE/PREFERENCES scaffold

- **Branch:** `docs/v0.7-profile-preferences-scaffold`
- **Commit:** `docs: scaffold v0.7 project profile and preferences`
- **Scope:** Add the committed, public-safe Markdown scaffold (`PROFILE.md`, `PREFERENCES.md`, and —
  per §9 — either a vault `AGENTS.md` or a decision to reuse the existing one). Each file states what
  belongs in it and the never-store list. **Docs only; no CLI/code.**
- **Files likely touched:** `docs/context/project/PROFILE.md`, `docs/context/project/PREFERENCES.md`,
  (maybe) `docs/context/project/AGENTS.md`, `docs/context/project/README.md` (list the new files),
  `CHANGELOG.md` (Unreleased/Added).
- **Tests required:** none (docs-only). Run the full suite to confirm no regression.
- **Review preset:** cheap (docs scaffold), balanced if the data-model shape is contested.
- **Non-goals:** no code, no CLI command, no context-pack change, no reading of these files by any
  command yet.
- **Stop conditions:** the scaffold implies a schema/database; a file would hold secrets/private data;
  the context pack changes.

### PR B — `vibe project doctor` checks profile/prefs presence

- **Branch:** `feat/v0.7-doctor-profile-check`
- **Commit:** `feat: project doctor reports profile/preferences presence`
- **Scope:** `vibe project doctor` gains **advisory** checks that `PROFILE.md`/`PREFERENCES.md` exist
  (informational — *not* a hard failure; their absence is fine). Read-only, no tokens, no writes.
- **Files likely touched:** the `project doctor` implementation module, its tests.
- **Tests required:** yes — assert doctor reports presence/absence correctly, stays read-only, exit
  code semantics unchanged (absence is advisory, not a failure).
- **Review preset:** balanced (it touches a real command).
- **Non-goals:** no fixing/scaffolding from doctor; no token spend; no change to the dangerous-staged
  check; doctor still writes nothing.
- **Stop conditions:** doctor would write a file, spend tokens, or turn a missing optional file into a
  hard failure.

### PR C — context export includes profile/prefs pointers or summaries

- **Branch:** `feat/v0.7-context-export-profile-pointers`
- **Commit:** `feat: context export references project profile and preferences`
- **Scope:** `vibe context export --for <agent>` adds **pointers** to `PROFILE.md`/`PREFERENCES.md`
  (the balanced review recommended committing to **pointers-only**, not summaries — smaller, matches
  the vault convention, and forces the agent to read the source), consistent with how the vault is
  referenced by pointer today. Redaction-checked; stdout-by-default unchanged.
- **Files likely touched:** the context-export module, its tests.
- **Tests required:** yes — export includes the pointers, stays read-only, never dumps raw private
  data, redaction guard still applies, existing `context export claude-code` path unchanged. **Add a
  negative test that the export never reads `.council/profile.*`** (the local profile must not leak
  into any handoff). **Enforce the pointer addition stays small** (target ≲ a few hundred chars) so
  the pack's 21/21 budget is not reopened.
- **Review preset:** balanced.
- **Non-goals:** no full profile dump into the export; no context-builder/pack change (do not reopen
  21/21); no model call; no reading of the local personal profile.
- **Stop conditions:** export would inline private data, read the local profile, bloat the pack, or
  change `context build`. A missing/malformed preference file must degrade gracefully (advisory,
  never a crash) — if it can't, stop.

### PR D — guide output uses preferences safely

- **Branch:** `feat/v0.7-guide-reads-preferences`
- **Commit:** `feat: agent guide reflects project preferences (tighten-only)`
- **Scope:** `vibe guide {claude|codex|fable}` optionally reflects project preferences (e.g. default
  review preset, PR style) in the generated instruction block — **advisory text only**, tighten-only.
  Read-only stdout generator; `--write` stays append-only/never-overwrite.
- **Files likely touched:** the guide-generation module, its tests.
- **Tests required:** yes — guide reflects a sample preference, **a negative "tighten-only" test** that
  a permissive-sounding preference (e.g. "PRs here are low-risk") never produces guide text that
  *encourages skipping review* or otherwise softens the safety workflow, guide never reads
  `.council/profile.*`, `--write` still append-only, no secrets in output, malformed preference file
  degrades gracefully.
- **Review preset:** balanced (safety text must be correct). Because generated guides are a high-trust
  output surface, treat this as a **strong** balanced review — escalate to full only if it surfaces a
  concrete safety-wording conflict.
- **Non-goals:** no preference that loosens a rule; advisory text must never *encourage* looser
  behavior (invariant #17 applies to the words, not just the code); no auto-`--write`; no model call;
  no reading of the local personal profile into output.
- **Stop conditions:** generated text could be read as widening a guard or nudging toward skipping a
  safety step, the guide reads the local profile, or it starts writing without `--write`.

### PR E — release prep

- **Branch:** `release/v0.7.0-prep`
- **Commit:** `chore: prepare v0.7.0 release`
- **Scope:** version bump (`backend/__init__.py`, `pyproject.toml`, `uv.lock` self-version line
  only), dated `CHANGELOG.md` `[0.7.0]` section, `docs/releases/v0.7.0.md`, README release-status
  update. **No tag, no GitHub Release** (those are separate manual maintainer steps).
- **Files likely touched:** `backend/__init__.py`, `pyproject.toml`, `uv.lock` (one line),
  `CHANGELOG.md`, `docs/releases/v0.7.0.md`, `README.md`.
- **Tests required:** full suite green; `vibe --version` prints the new version.
- **Review preset:** cheap (mechanical), balanced if any surface changed.
- **Non-goals:** no feature work in the release PR; no tag/Release; no dependency-graph change.
- **Stop conditions:** any dependency churn in `uv.lock` beyond the self-version line; a tag/Release
  implied without explicit instruction.

**Sequencing:** A → B → C → D → E. Each merges before the next starts. Any of A–D may be split further
if a diff sprawls. If the data-model shape (§5/§9) is unresolved, PR A does a short `vibe review
--preset cheap` on the proposed structure before writing it.

  safer, smaller, matches the vault convention, and avoids mis-summarization. PR C is written that way;
  confirm with the maintainer before implementing.
- **`AGENTS.md` collision (resolve before PR A).** The repo already uses `AGENTS.md` (repo-root) as a
  `vibe guide codex --write` **write target**. If the same file also became a **preference source that
  a command reads**, a guide `--write` append could corrupt/confuse preference parsing (a
  writable-and-readable file is a concrete corruption risk, not just a naming clash). **Lean:** use a
  distinct vault file (e.g. `docs/context/project/AGENT-ROLES.md`) for per-agent role preferences and
  keep root `AGENTS.md` as guide output only — decide in PR A.
- **Skills POC timing.** Should a docs-only `SKILL.md` proof-of-concept (§10) come **before or after**
  v0.7 — or not at all until explicitly scoped?
- **What is safe to commit publicly?** Confirm the boundary for `PROFILE.md`/`PREFERENCES.md`: which
  preference fields are team-shareable/public-safe vs. which must stay in the local profile.
  **Proposed working definition of "public-safe" (from the balanced review, confirm in PR A):**
  content is public-safe if it holds no secrets, no private user data, no absolute machine
  paths/usernames, and the author would be comfortable seeing it in a public GitHub repo — and all
  committed preference files are subject to `vibe lint --redaction` with a critical finding blocking
  the commit.
- **Schema stability / graceful degradation.** Even Markdown-first, the fields a command reads form a
  de-facto schema. **Hard requirement (not merely open) before PR C/D:** all preference parsing must
  degrade gracefully — a missing, renamed, or malformed field yields a clear advisory/info message and
  falls back to default behavior; **a parse error must never crash a command**. Still to agree: how a
  field rename/removal is handled over time — without turning the files into a versioned JSON schema.

## 11. Tech-stack note (avoid premature platform adoption)

- **Next.js / PostgreSQL / Redis / Docker / LangGraph are NOT current local-first requirements.** The
  product is a local-first CLI with plain-Markdown memory and a localhost-only panel; none of those
  are needed for v0.7 (or for the local core generally).
- **They may matter for a future hosted / open-core / SaaS layer** (v0.9+, gated behind a resolved
  license/provenance "Question 0" and validated demand) — not now, and not from this phase.
- **MCP and multi-provider models are already relevant** — read-only MCP (v0.4) and the
  OpenRouter/Ollama provider seam exist and are in scope for ongoing work.
- **Avoid premature platform stack adoption.** Adding a database/queue/web-framework before there is a
  hosted product to need them is exactly the "overbuilding hosted/vector layers before prerequisites"
  risk in [`RISKS.md`](../context/project/RISKS.md). Stdlib-first, Markdown-first, local-first.
