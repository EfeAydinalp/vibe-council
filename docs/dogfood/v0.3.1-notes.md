# v0.3.1 dogfood notes

_Date: **2026-06-30**. Public-safe observations only — paraphrased, no raw council output,
no secrets, no local absolute paths, no private strategy._

## Purpose

v0.3.1 hardening starts by **dogfooding the v0.3.0 decision-memory / context loop** on a
realistic local task, to find rough edges **before** building new features (v0.4 / MCP). This
is a docs-only log; it records what felt smooth, what felt rough, and which follow-up PRs
each finding maps to. It feeds the separate v0.3.1 hardening plan.

## Scope

**Commands exercised** (full v0.3 loop, run locally):

- `decisions new --from-run <local sample review>` → generated a LOCAL draft
- manual inspection of the generated draft
- `decisions promote <draft> --dry-run` (dry-run only — did not write into curated docs)
- `decisions lint`
- `context build`
- `context check`
- `context export claude-code`
- `operator status`
- supporting: full test suite, `lint --redaction`

**Intentionally not tested:**

- No live **council run** (`review`/`extract`) — no model/API/network calls; the `--from-run`
  input was a realistic local **sample** review, not a real spend.
- No real `promote` write into `docs/decisions/` (kept to `--dry-run` to avoid polluting the
  curated set during dogfood).
- No `operator set`/`clear` mutation beyond reading status.

## Observations

### Decision draft extraction (`new --from-run`)
- Produces a valid, well-structured **scaffold** (frontmatter + standard headings) and
  correctly marks it LOCAL/gitignored with a clear "next: review/redact → promote" hint.
- But extraction is essentially **title + source-path only**. The sample review's
  Recommendations / Rejected-alternatives / Constraints sections were **not** mapped into
  the draft — every content section is left as `_TODO_`. So the practical lift over plain
  `decisions new` (template) is small.
- The title slug is taken verbatim from the review H1, yielding a very long filename/id.

### Human review / redaction
- The draft lands in `.council/decisions/drafts/` (gitignored), which is the right place for
  human edit/redact before promotion. The workflow hint points at the exact next command.
- Because the body is all `_TODO_`, the "human review" step is effectively "write the whole
  decision," not "review an extracted draft."

### Promotion (`promote`)
- `--dry-run` clearly shows the intended target path and does not write — good.
- It **accepted a draft that was entirely `_TODO_` placeholders**: no check that the
  Decision / Rationale sections contain real content. A skeleton could be promoted into the
  curated set.
- The dry-run target filename uses the `DEC-<id>.md` form, but every existing curated record
  uses the `YYYY-MM-DD-<slug>.md` convention — an inconsistency a reader would notice.

### Decision lint (`decisions lint`)
- Clean, fast, clear summary (`0 error(s), 0 warning(s)` + `passed`). This is the UX baseline
  the other lints should match.

### Context build (`context build`)
- Deterministic, writes locally, prints redaction counts and the LOCAL/gitignored notice.
- Emitted `reduced recent full decisions to fit budget` — the default character budget
  truncates as the curated set grows (known char-budget limitation).

### Context check (`context check`)
- Per-check output is genuinely useful. Score **20/21 (95%)**, redaction critical=0.
- **The one miss is `memory:human-review`** (advisory). `signal:rejected-alternatives`
  **passes**. This corrects the earlier hypothesis in the v0.3.1 plan.

### Claude Code export (`context export claude-code`)
- Writes a local file, explicitly states `CLAUDE.md not modified` and that the output is
  LOCAL/gitignored/not-staged. Copy-paste usability looks fine for a single-file paste.

### Operator status (`operator status`)
- Clean empty-state ("No operator status yet."), exit 0. No surprises.

## Bugs / rough edges

- **`context check` advisory miss is `memory:human-review`, not rejected-alternatives.**
  - Symptom: steady state is 20/21; the miss is the human-review signal not surfacing in the
    built pack.
  - Expected: either the pack surfaces the human-review constraint (→ 21/21) or the advisory
    is documented as expected.
  - Suspected area: `context build` pack assembly + `check_pack` `memory:human-review` check
    in `backend/context_pack.py`.
  - Severity: medium.
  - Follow-up: **PR #50**.

- **`promote` accepts an all-`TODO` skeleton draft.**
  - Symptom: a draft with every content section as `_TODO_` passes `promote --dry-run`.
  - Expected: refuse (or at least warn) when required content sections are empty/placeholder.
  - Suspected area: `decisions promote` validation in `backend/decisions_docs.py`.
  - Severity: medium/high (curation quality risk).
  - Follow-up: **PR #49**.

- **`promote` target filename uses `DEC-…`, not the `YYYY-MM-DD-slug` convention.**
  - Symptom: dry-run target is `DEC-<id>.md`; curated records are `YYYY-MM-DD-<slug>.md`.
  - Expected: consistent with the established naming.
  - Suspected area: `decisions promote` filename derivation.
  - Severity: medium.
  - Follow-up: **PR #49**.

- **`new --from-run` does not map review content into the draft.**
  - Symptom: only title + source path are captured; all sections are `_TODO_`.
  - Expected: at least carry over recommendations / rejected-alternatives as a starting point.
  - Suspected area: extraction heuristic in `backend/decision_memory.py` / `decisions_docs.py`.
  - Severity: medium (limits the value proposition of `--from-run`).
  - Follow-up: **PR #49** (or a dedicated extraction-improvement PR).

- **`new --from-run` slug is verbatim from the review H1 (long/awkward).**
  - Symptom: very long id/filename from a descriptive heading.
  - Expected: a sensible length cap or a `--title` override hint in the flow.
  - Suspected area: slug derivation.
  - Severity: low.
  - Follow-up: **PR #49** or **PR #51**.

- **Lint verdict framing is inconsistent across lints.**
  - Symptom: `decisions lint` prints a `passed` verdict line; `lint --redaction` prints a
    counts summary (`N critical, M warning(s)`) but no explicit pass/clean verdict line. Also
    `lint --redaction` scans only *tracked* public docs, so newly-created untracked docs are
    not scanned until staged.
  - Expected: consistent pass/fail framing across lints; clarity (in help/output) that
    redaction scans tracked files only.
  - Suspected area: `lint --redaction` output formatting / file selection.
  - Severity: low (UX consistency).
  - Follow-up: **PR #51**.

## Open questions (answered by this session)

- **Is the 20/21 miss actually `signal:rejected-alternatives`?** **No.** The miss is
  `memory:human-review`; `signal:rejected-alternatives` passes. The plan's hypothesis was
  wrong and PR #50 should target the human-review signal.
- **Should `context build` always emit a rejected-alternatives section, even if sparse?**
  Less urgent than thought (the signal already passes). Still worth considering for richer
  agent context, but it is not what's failing the check.
- **Does `new --from-run` produce a draft easier than writing one manually?** Only
  marginally today — it pre-fills the title + source reference but leaves all substance as
  `_TODO_`. The real win (mapping review content into sections) is unimplemented (PR #49).
- **Does the Claude Code export feel copy-paste usable?** Yes for a single-file paste; it is
  explicit about being local and about not touching `CLAUDE.md`.
