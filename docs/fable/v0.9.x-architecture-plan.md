## 1. Executive summary

**The council's theme — "Apply the proven; describe the personas; defer their behavior" — is correct
and is accepted with two modifications: a numbering simplification (two releases, not three) and one
spec/CLI mismatch resolution the council could not have seen (schema v1's `full` value is a council
*mode*, not a `--preset` value).**

The council's caution is architecturally right for a reason worth stating precisely: the four v0.8.2
keys carry **mechanical** tighten-only proofs (an ordered floor and three additive constraints), while
persona emphasis carries only an **informal** one ("more scrutiny") — and the council's first-pass
review correctly identified that informal proof as a **false equivalence** (reframing can reduce
effective attention without removing anything). v0.9.x therefore applies only the objects whose safety
is provable today, and produces the *design* that could make persona behavior provable later.

**v0.9.x ships (two releases):**

- **v0.9.0 — apply the four proven keys**: a clamped internal reader over the existing validator, an
  explicit CLI-wins precedence (argparse sentinel), a review/diff preset floor + advisory
  full-council note, a `require_usage_flag` warning, doctor advisory staged-path warns, and a
  tests-only guard/executor-ignore tripwire. Every effect is **suggest/warn-only, additive,
  reversible (`--no-preferences`), stderr-only, and inert without a valid block**.
- **v0.9.1 — persona review lenses (pure documentation) + the v0.10.x dissent-preservation design
  sketch + dogfood/hardening.** No schema fields, no validator runtime, no prompt influence.

**v0.9.x does not ship:** persona behavior of any kind, persona schema objects/validator, a
`.council/profile.*` store, a session/workspace store, UI/dashboard controls, Workbench/trust/executor
changes, any network/hosted surface, any new dependency.

**An honest sizing note (maintainer should read this):** with the project baseline `DEFAULT_PRESET =
"balanced"`, the observable v0.9.0 behavior changes are deliberately small — a full-council advisory
note, a usage-flag warning, and two advisory doctor warns. **v0.9.0's real deliverable is the
application *pattern*** (single clamped consumption point, fixed precedence, byte-identical-without-
a-block golden tests, guard/executor-ignore tripwires) that every later application — including
personas — will reuse. Small observable delta, large structural value. That is by design, not
accident.

## 3. Council conclusion: accepted / modified / rejected

- **ACCEPTED — the theme and its safety reasoning.** Apply the four proven keys; personas as pure
  documentation; persona *behavior* (prompt emphasis, `--persona`, presets-as-schema) deferred to
  v0.10.x behind a designed dissent-preservation framework. The debate's central finding — dissent
  suppression is a **new risk class** not covered by the tighten-only proof shapes — is correct and
  binding.
- **ACCEPTED — all §"normative" brief sections**: override precedence (CLI wins → `--no-preferences` →
  valid block → baseline), warning hygiene (at-most-once, actionable, never blocks), backward
  compatibility (no block ⇒ byte-identical; suggestions to stderr; stdout machine-clean), and the
  per-key invariant test gate.
- **MODIFIED — release numbering** (explicitly delegated to Fable): **two releases, not three.**
  v0.9.0 = apply-the-keys (5 PRs incl. release prep); v0.9.1 = persona lenses + v0.10.x sketch +
  dogfood (3 PRs incl. release prep). A standalone "v0.9.2 = release prep" release prepares nothing;
  v0.9.2 stays **reserved as an optional hardening patch slot** if v0.9.1 dogfood finds issues.
- **MODIFIED — one schema/CLI mismatch resolved** (not visible at council altitude): schema v1's
  `default_review_preset` enum is `cheap|balanced|full`, but the CLI preset surface is
  `{cheap, balanced, premium}` — **`full` is a mode (`vibe full`), not a preset**. Resolution in §5.1;
  the schema file itself is **not** changed (v1 stays frozen); only application semantics are defined.
- **REJECTED — nothing.** No council conclusion is overturned. The one place Fable exercises delegated
  judgment against a council *option* is Q1 (`require_usage_flag` ships; §12 Q1) — inside the
  discretion the brief granted.

## 5. Architecture

### 5.1 The preset floor — semantics and the `full` mismatch (D3)

Ordering: `cheap < balanced < full` (schema side); CLI presets: `cheap`, `balanced` (`premium` exists
but is unreachable from preferences by construction — not in the schema enum). Baseline =
`DEFAULT_PRESET` (`balanced`, from `backend/config.py`).

Application table for `default_review_preset` on `review`/`diff` when the user passed **no** explicit
`--preset` and **no** `--no-preferences`:

| Block value | Effective preset | Extra output (stderr, one line) |
|---|---|---|
| *(no valid block / key absent)* | `DEFAULT_PRESET` | — (byte-identical to v0.8.2) |
| `cheap` (below baseline) | `DEFAULT_PRESET` | — (inert; the validator's conflict reporting covers visibility) |
| `balanced` (== baseline) | `DEFAULT_PRESET` | — (no-op; no notice, avoid noise) |
| `full` (above baseline) | `DEFAULT_PRESET` (**no preset change** — `balanced` *is* the strongest non-premium preset) | `[preferences] PREFERENCES.md sets a full-council review floor — consider 'vibe full' for this review (--preset/--no-preferences override; premium still requires --allow-premium)` |

Note the `full` row carefully: **nothing is silently reinterpreted** — the effective preset simply
stays at the baseline (which already is the strongest non-premium preset), and the *entire* effect is
the explicit advisory naming `vibe full`. The balanced review proposed instead making `full` a
**validation error**; **declined**: schema v1 is frozen (`full` is a valid enum value in the shipped
v0.8.2 validator; *"schema v2 arrives via its own reviewed spec release"*), and retroactively
invalidating a valid value in a patch is exactly the semantic drift that rule exists to prevent. The
notice-only application is honest, announced, and tighten-consistent.

Explicit `--preset X` → `X`, always, preference never consulted for the preset. `--no-preferences` →
baseline. **A preference can never lower, never block, never select `premium`, and never silently
change anything** (every non-no-op is announced on stderr).

### 5.2 The clamped reader (D1, D6) — `backend/preferences.py`

`validate_preferences()` (findings-only) is **unchanged**. One new public function:

```python
class Suggestions(NamedTuple):
    review_preset_floor: Optional[str]   # only "full" is representable in practice: the value is
                                         # emitted ONLY when strictly above baseline; else None
    require_usage: bool                  # True only when the block validly sets it
    extra_sensitive_paths: tuple         # validated relative prefixes, or ()
    never_stage_extra: tuple             # validated relative paths, or ()

NEUTRAL = Suggestions(None, False, (), ())

def effective_suggestions(project_root) -> Suggestions: ...
```

- Reuses the **same private extraction/validation** helpers (`_extract_json_blocks`,
  `_validate_object`) — no second parser, no relaxation. **Any anomaly** (missing file/block, invalid,
  oversized, multiple blocks, symlink escape, undecodable, any validation error) → **`NEUTRAL`**
  (fail-closed, mirroring the validator's ignored-state exactly: an invalid block validates as warn
  *and* applies as nothing).
- **Values are clamped at the source:** `review_preset_floor` is non-None only when the block value is
  strictly above `DEFAULT_PRESET` in the §5.1 ordering (so a consumer *cannot* implement "lower" even
  by mistake); the path tuples contain only entries that passed `_bad_relative_path`; `premium` is
  unrepresentable (not in the enum).
- **Contract amendment (visible, deliberate):** the module docstring changes from *"exposes findings,
  never settings"* to *"exposes findings, plus clamped tighten-only suggestions via
  `effective_suggestions()`; raw parsed JSON never escapes."* The v0.8.2 structural blocker evolves
  rather than silently breaking: the import test becomes **allowlist-first** (per the balanced
  review) — it scans **every** `backend/*.py` module and asserts the set of importers of
  `backend.preferences` is exactly `{"cli.py"}`; any new importer anywhere fails the suite by
  construction (strictly stronger than enumerating known-critical modules).
- **Fail-closed stays per-block, deliberately** (balanced-review finding **declined with reasons**):
  the review proposed per-key application ("a typo in one key shouldn't disable the others"). Rejected
  because (a) per-block ignore is the **more** conservative direction — on any anomaly, *nothing* is
  applied, and the "lost" effects are advisory warns, not protections (the trust boundary never
  depended on them); (b) applier and validator must agree — the doctor reports the block "invalid —
  ignored," and an applier that partially applies what the validator ignored is a
  consistency/confusion attack surface; (c) a partially-malformed **untrusted** block should not get
  partial effect. Per-key application may be revisited only alongside a schema revision with its own
  review.
- Explicitly **no** consumer-facing raw dict, no `get(key)` API, no environment override, no caching
  across processes, no write path, no `.council/` read.

### 5.3 CLI wiring (D2) — `backend/cli.py` only

- Shared model parser: `--preset` `default=None` (help text still names the default); new
  `--no-preferences` flag on the shared guard parser (the five mode commands).
- One helper, called **once, early**, by each mode command before any existing `args.preset` use:

```python
def _resolve_preferences(args, command, project_root) -> str:   # returns effective preset
    # 1. explicit CLI --preset wins (args.preset is not None)
    # 2. --no-preferences -> DEFAULT_PRESET, skip reading entirely
    # 3. suggestions = preferences_mod.effective_suggestions(project_root)  [NEUTRAL-safe]
    #    - review/diff only: apply §5.1 table (stderr notice on the full case)
    #    - all five mode commands: if suggestions.require_usage and not args.usage:
    #        one stderr warning (actionable, never affects exit code)
    # 4. else DEFAULT_PRESET
```

  `args.preset` is then **assigned the resolved value** so the existing downstream uses
  (`_council_size(args.preset)`, lock hashing, `run_mode_stream`) are untouched — the refactor is
  contained to the resolution point. The premium guard is unreachable from this path by construction.
- **Which commands consult what:** preset floor → `review` + `diff` only (the schema's anchored
  behavior names exactly these); `require_usage_flag` warning → all five model-spending commands
  (`review`, `diff`, `extract`, `mini`, `full` — one shared emission point in the helper);
  `extract`/`mini`/`full` preset stays `None → DEFAULT_PRESET` (no floor consultation).
- **Warning hygiene is structural:** the helper runs once per process; a one-shot CLI cannot nag
  within an invocation. Wording is fixed and tested:
  `[preferences] PREFERENCES.md sets require_usage_flag — pass --usage to see token usage (advisory).`

### 5.4 Output & observability

All preference-driven text goes to **stderr** (house rule; stdout stays machine-clean). Exactly two
possible lines exist (§5.1 full-floor notice, §5.3 usage warning) plus the doctor lines in §5.5; each
appears at most once per invocation; wording is pinned by tests. `--usage`/exit codes/JSON output are
never altered by a preference.

### 5.5 Doctor integration (D5)

In `project_doctor_report`'s staged-file section (after the existing dangerous/uv.lock loop), when
`effective_suggestions().never_stage_extra` / `.extra_sensitive_paths` are non-empty and git is
available: a staged path equal to / prefixed by an entry → one `[warn]` line naming the preference
source — **advisory only** (`ok` untouched, `[FAIL]` set unchanged, READY/NOT-READY semantics byte-
stable). **Volume-bounded** (per the balanced review): at most **10** preference-driven warn lines
per run; further matches collapse into one roll-up line (`… and N more (see PREFERENCES.md)`), so a
large staged set cannot flood the report. The existing `Preferences (machine-readable, advisory):`
section is unchanged. No `--no-preferences` on doctor: the doctor *reports on* preferences; it does
not act on them.

### 5.6 What is proven untouched

- **Guard/executor:** no `workbench_*` module imports `backend.preferences` (enumerated static test) +
  a behavioral test: trust evaluation / executor dry-run results are **byte-identical** with and
  without a maximal valid block present.
- **Guide/context export:** the v0.7.1 byte-stability/pointer-only invariant tests keep passing
  **unmodified** (if a v0.9.x diff needs to edit them, the diff is wrong — same tripwire as v0.8.x).
- **MCP / context pack:** no new inputs; pack stays 21/21; planning docs and lens docs are never
  ingested.
- **No `.council/profile.*`:** the reader touches exactly one file (`PREFERENCES.md`, realpath-inside-
  root), same as the validator.

## 8. Test plan (the mandatory no-loosening proof, §12 Q6)

House gates on every PR (unittest discover; doctor READY; redaction 0 critical / 30 warnings unless a
PR enumerates its delta; decisions lint; context build/check 21/21; MCP 21/21; no-stage check). Plus,
as **merge gates for the application PRs**:

1. **CLI-wins:** explicit `--preset cheap` with a `full` block → `cheap`; explicit `--usage` silences
   the usage warning.
2. **Floor-never-lowers:** `cheap`/`balanced` block values are inert; ordering test pins
   `cheap < balanced < full`; `premium` unrepresentable end-to-end (reader + CLI + argparse choices).
3. **Suggest≠enforce:** exit codes byte-stable in every preference state; nothing blocks; doctor
   `ok`/READY untouched by any preference; no preference-driven `[FAIL]`.
4. **Golden byte-identity:** no block / invalid block / `--no-preferences` → stdout **and** stderr
   byte-identical to v0.8.2 behavior for review/diff/doctor (fixture-based golden tests).
5. **Guard/executor isolation:** behavioral byte-identity for trust/executor with-vs-without a
   maximal block; **allowlist-first import scan** over every `backend/*.py` — the importer set of
   `backend.preferences` must equal `{"cli.py"}` exactly.
6. **Pointer-only tripwire:** the v0.7.1 guide/export invariant tests pass **unmodified**.
7. **Observability:** every non-no-op emits its pinned stderr line exactly once; stdout unchanged.
8. **Fail-closed adversarial:** the full v0.8.2 invalid-block matrix drives `NEUTRAL` (a
   table-driven test shared with the validator suite so the two can never drift).
9. **Determinism:** resolution identical from any cwd; no timestamps in any new output.

## 11. Risks & mitigations

- **Argparse-sentinel regression** (D2 touches every mode command's preset path) → golden
  byte-identity tests (§8.4) land in the same PR; `args.preset` reassignment keeps downstream code
  untouched.
- **Settings-API leak** (a future consumer imports the reader "because it's convenient") → enumerated
  import-allowlist test + clamped-values design (there is no raw value to leak).
- **Suggest→enforce creep** → stop conditions + exit-code/`[FAIL]` byte-stability tests; the doctor
  integration is warn-only by construction.
- **`full`-floor confusion** (users expect the preset to literally become "full") → the advisory
  notice names `vibe full` explicitly; release notes explain the mode/preset distinction.
- **Silent-change perception** → every non-no-op announces itself; golden tests pin the
  no-block state.
- **Lens docs read as behavior** → PR 6 stop conditions + vault-consistency tests (wording asserts
  "not applied / not a command").
- **v0.10.x sketch under-scopes the research problem** → PR 7 must end with an explicit feasibility
  verdict; if dissent-preservation looks like a multi-month research problem, the sketch says so and
  v0.10.x planning inherits that honestly.

## 12. Fable answers to the council's unresolved questions

**Q1 — `require_usage_flag` in v0.9.0 or wait?** **Ship in v0.9.0 (PR 3).** The council's
warning-spam concern was framed around session-style nagging; `vibe` is a one-shot CLI — the helper
gives exactly **one** emission point per process, so at-most-once is structural, not a throttling
design. The warning is actionable, stderr-only, never affects exit codes, and disappears under
`--usage`, `--no-preferences`, or no block. Demotion path: if dogfood shows agent-loop noise, v0.9.1
may downgrade wording/frequency — a one-line change behind a test.

**Q2 — How many persona lenses; named or "review lenses"?** **Both framings, three lenses (D7).**
The document is titled and framed as **"council review lenses"** (deflating the persona-as-agent
implication), with **three** elaborated named lenses — **Security Guardian, Cost Skeptic, Local-first
Guardian** — chosen because each anchors to an *existing, testable* project invariant (security
invariants; cost/budget policy; localhost-only). The other four are one-line stubs marked future.
Rationale: three elaborated lenses give v0.10.x real material without committing a seven-lens taxonomy
before any usage data; the stubs preserve the full vocabulary the maintainer named.

**Q3 — Extend schema v1 or keep unchanged?** **Unchanged, frozen (D8).** v0.9.0 applies existing
keys; no new key is needed. When persona presets eventually arrive they are **recommended as
`schema: 2`** via its own reviewed spec release: a preset bundles values *plus prose emphasis* — a
structurally new object, not an additive key — and v1's "unknown version → ignore whole block" gate
means old validators degrade safely. This is a recommendation recorded for v0.10.x planning, not a
design.

**Q4 — Safest architecture for applying the four keys?** §5 in full: one clamped reader
(`effective_suggestions`, fail-closed to `NEUTRAL`, values pre-clamped at the source), one resolution
helper in `cli.py` called once per command, argparse sentinel for CLI-wins, stderr-only pinned
notices, enumerated consumption points, golden byte-identity for every off-state, and an enumerated
import allowlist. The key property: **a consumer cannot express "lower" or "enforce" with the API it
is given.**

**Q5 — Where exactly is application allowed?** Exactly three surfaces (§9): review/diff **preset
selection** (never prompt text), the five mode commands' **usage warning**, and **doctor advisory
warns** for the two path lists. **Never:** executor, guard/trust, panel, importer, proposals, MCP,
guide, context export, prompt construction, ranking, synthesis. The executor/guard prohibition is
enforced by both a static import scan and behavioral byte-identity tests.

**Q6 — Mandatory tests to prove no loosening?** The nine-point matrix in §8 — CLI-wins,
floor-never-lowers, suggest≠enforce (exit codes/`[FAIL]` byte-stable), golden byte-identity for
no-block/invalid/`--no-preferences`, guard/executor isolation (behavioral + import scan),
pointer-only tripwire, announced-non-no-op observability, fail-closed adversarial table shared with
the validator suite, determinism.

**Q7 — What must the v0.10.x dissent-preservation sketch contain?** (PR 7 outline, binding for that
PR:) (1) **Threat model**: reframing bias, seat crowd-out, emphasis-as-suppression, cumulative-bias
across stages, and persona-text-as-injection; (2) **Structural rules**: additive collect-stage-only,
own-seat-only, ranking/chairman persona-blind, no persona text in synthesis inputs; (3) **Content
rules**: disallowed patterns (may not reference other personas/seats/ranking/synthesis; no
"ignore/deprioritize/overrule" language; length-bounded); (4) **Adversarial test-suite shape**:
dissent-canary fixtures — a seeded objection that must still surface with every persona enabled;
injection-shaped persona text must be rejected; (5) **Observability**: persona use labeled in stage
outputs and usage reporting; (6) **Override/rollback**: per-invocation flag, feature flag default-off,
kill-switch semantics; (7) **Acceptance criteria** a v0.10.x PR must meet before full review; (8)
**Open research questions + an explicit feasibility verdict** (including "this may be a research
problem" as an acceptable answer).

## Balanced review outcome (this plan)

The balanced council review of this plan returned **"conditional approval"** — *"exceptionally
rigorous… the core strategy is correct"* — with three findings it called mandatory plus
recommendations. Adjudication:

**Adopted:** the **allowlist-first import scan** (scan all of `backend/`, importer set must equal
`{"cli.py"}` — strictly stronger than enumeration; §5.2/§8.5); a **volume cap on doctor advisory
warns** (max 10 + roll-up line; §5.5); a **dogfood-before-release-prep step** for v0.9.0 (§7); the
**red-lines list** for all future preference work (§9); an explicit **`full`-row clarification** that
nothing is reinterpreted (the preset stays at baseline; the whole effect is the advisory notice;
§5.1); and the `argparse.SUPPRESS` alternative noted for implementers (§13).

**Declined with reasons:**

- **Per-key fail-closed** (apply valid keys from a partially-invalid block): per-block ignore is the
  *more* conservative direction — on anomaly nothing applies, the "lost" effects are advisory warns
  (never protections the trust boundary depends on), and applier/validator consistency (both treat the
  block as one unit) closes a confusion-attack surface. Recorded in §5.2.
- **Rejecting `full` as a schema validation error:** schema v1 is frozen; retroactively invalidating a
  shipped-valid enum value in a patch violates the "schema v2 arrives via its own reviewed spec"
  rule. The notice-only application (§5.1) is honest and announced.
- **Deferring the lens docs + dissent sketch entirely to v0.10.x:** contradicts the twice-reviewed
  council phase brief (v0.9.1 scope is a council decision). The sketch's mandatory feasibility
  verdict (§12 Q7 item 8) is precisely the mitigation for "don't promise unproven features."
- **Property-based testing (Hypothesis):** a **new runtime/test dependency**, against the stdlib-only
  non-goal the reviewer's own §10 restates. Table-driven stdlib tests give the shared-matrix property
  (§8.8) without it.
- **Consolidating to one release / four PRs:** the two-release split keeps "apply the proven" and
  "describe the personas" separately reviewable and separately shippable — the council's sequencing
  intent.
- **A separate `preferences_policy.py` module:** premature layering for three consumption points in
  one file; the clamped reader already isolates the policy surface. Revisit if consumers multiply.
