A **planning document**, not an implementation. `vibe-council v0.1.0` is published; the
[post-v0.1 roadmap](post-v0.1-roadmap.md) and the

first second provider.** This doc inventories the current OpenRouter coupling, proposes

## A. Current architecture inventory

The codebase is small and the coupling is concentrated, which makes the abstraction
tractable. Concrete coupling points (verified by reading the source):

### Where OpenRouter is coupled
- **`backend/openrouter.py`** — the only HTTP client. Owns: `httpx.AsyncClient`, the
  `Authorization: Bearer {OPENROUTER_API_KEY}` header, the request payload
  `{"model", "messages"}`, the response parse `data['choices'][0]['message']`
  (`content`, `reasoning_details`), usage extraction `data.get('usage')`, and the
  OpenRouter-specific HTTP error taxonomy (`_safe_reason`: 402 = insufficient credits,
  404 = model not found, 429 = rate limited, …). Public surface:
  `query_model_detailed`, `query_model`, `query_models_parallel`.
- **`backend/config.py`** — `OPENROUTER_API_KEY` (env), `OPENROUTER_API_URL` (hardcoded
  endpoint), and all model IDs in OpenRouter's `provider/model` namespace
  (`google/gemini-2.5-flash`, `anthropic/claude-sonnet-4.5`, …), env-overridable via
  `MODEL_*`. `PRESETS` reference those IDs.
- **`backend/council.py`** — imports `query_models_parallel`, `query_model` directly from
  `.openrouter`; every stage (`collect`, `critique`, `rank`, `synthesize`) calls them.
- **`backend/decision_memory.py`** — imports `query_model_detailed` directly from
  `.openrouter` for `extract`.
- **`backend/cli.py`** — `_require_api_key()` is OpenRouter-specific (checks
  `OPENROUTER_API_KEY`, the `sk-or-v1-...` placeholder). The `--usage` line and
  `guards.cost_note` literally say "as reported by OpenRouter".
- **`backend/main.py`** (FastAPI/web UI) — also rides on `council.py`; out of scope for
  v0.2 CLI work but inherits the abstraction for free.

### Where model IDs live
`backend/config.py` only: per-tier env-overridable slots (`GEMINI_FLASH`, `GPT`,
`CLAUDE_SONNET`, …) → `MODEL_ENV_VARS` → `PRESETS` (`cheap`/`balanced`/`premium`, each
`{council[], chairman, extract}`) → back-compat `COUNCIL_MODELS`/`CHAIRMAN_MODEL`. IDs are
opaque strings passed straight through to the provider.

### Where usage / cost accounting lives
- Each provider response carries an optional `usage` dict (`prompt_tokens`,
  `completion_tokens`, `total_tokens`, and OpenRouter's extra `cost`). It is threaded
  through every stage result.
- **`backend/guards.py`** — `aggregate_usage()` sums token fields and the `cost` field
  (never fabricated); `cost_note()` / `enforce_cost_cap()` consume the reported cost;
  `estimate_tokens()` / `token_guard()` produce the **pre-run** rough estimate (4 chars/
  token heuristic, no tokenizer dependency).
- **`backend/cli.py`** — `_usage_items()` collects the per-stage usage; `_print_usage()`
  renders `--usage`; `_finish_cost()` enforces `--max-cost` post-run; `_save_stages()`
  persists usage under `.council/usage|stages`.

### How CLI modes call council logic
`cmd_mode(args, mode)` → `_run_guarded()` (token guard + loop guard) → `asyncio.run(_run)`
→ `run_mode_stream(text, mode, preset)` in `council.py` → per-mode stage list (from
`config.MODES`) → `openrouter.query_*`. `run_mode_stream` is the single orchestration
source (the web UI shares it). Provider calls happen **only** at the leaf of this chain.

### What assumptions are OpenRouter-specific
1. A **single global API key** via one env var, Bearer auth, one endpoint URL.
2. OpenAI-shaped **request/response** (`choices[0].message.content`, `messages[]`).
3. A **`usage` dict that includes `cost`** (most providers report tokens, not dollars;
   local providers report neither meaningfully).
4. The **`provider/model` ID namespace** and OpenRouter's model catalog.
5. **HTTP status semantics** mapped to human reasons (402 credits, 404 model, 429 rate).
6. Implicit assumption that **a key is always required** (false for Ollama).

## B. Provider abstraction design

Goal: a **minimal** interface that two providers (OpenRouter, Ollama) can satisfy without
inventing a speculative N-provider framework. The council's prior guidance (post-v0.1
review) was explicit: scope to exactly two providers, keep the interface small.

### Proposed provider interface
A `Provider` protocol (stdlib `typing.Protocol` or a small ABC), one instance selected at
startup, injected where `openrouter.query_*` is imported today:

```text
class Provider(Protocol):
    name: str                       # "openrouter" | "ollama"
    async def chat(model, messages, *, timeout) -> ChatResult
    def requires_api_key() -> bool
    def describe_model(model) -> ModelInfo   # best-effort capabilities/metadata
    async def health() -> HealthStatus       # for `vibe doctor`
```

`query_models_parallel` stays as a thin helper that fans `chat` out with
`asyncio.gather` (unchanged behavior). Council/decision_memory call the **injected
provider**, not a module-level import.

### Request / response shape
- **Request (normalized):** `model: str`, `messages: list[{role, content}]`, `timeout`.
  Adapters translate to the provider's wire format. No provider-specific kwargs leak into
  council code in v0.2 (temperature/JSON-mode deferred).
- **Response (`ChatResult`):** `content: str | None`, `reasoning_details: Any | None`,
  `usage: Usage | None`, `raw: dict` (escape hatch, never required by callers). This is
  the exact shape `council.py` already consumes, so stages don't change.

### Usage / cost metadata shape (`Usage`)
Normalize to: `prompt_tokens`, `completion_tokens`, `total_tokens` (ints, optional), plus
`cost: float | None` and `cost_source: "provider" | None`. **Cost is never fabricated** —
if the provider doesn't report dollars, `cost` is `None`. This matches today's
`aggregate_usage` contract; only the *source label* is new, so the `--usage` line can stop
hardcoding "as reported by OpenRouter."

### Error handling
A small normalized error taxonomy mapped from each adapter:
`auth` | `not_found` | `rate_limit` | `insufficient_credits` | `timeout` |
`unavailable` | `bad_request` | `network`. Adapters map their wire errors into this set;
the safe one-line stderr message and graceful-degradation (`None` → continue with
successful responses) behavior is preserved. **No raw key ever logged.**

### Retry / rate-limit behavior
Out of scope as new behavior for v0.2 PR 1 (preserve today's no-retry path). Design leaves
room for a later, opt-in bounded backoff on `rate_limit`/`unavailable` (jittered, capped
attempts) — explicitly **not** in the first PR to avoid behavior regression.

### Streaming / non-streaming stance
v0.2 stays **non-streaming** at the provider layer (matches today; the SSE web path
buffers stage results, not token streams). `stream_chat` is a *named future extension* on
the protocol but unimplemented in v0.2. Don't let streaming creep into the abstraction PR.

### Model capabilities metadata (`ModelInfo`)
Best-effort, non-authoritative: `context_window: int | None`, `supports_system: bool`,
`is_local: bool`, `notes: str | None`. Used only for `vibe doctor` warnings and future
pre-flight estimates — never to gate a run in v0.2.

### Local provider differences (designed for, not built in B)
- **No API key**, no dollar cost, often no token-usage accounting.
- Localhost endpoint, model must be **pulled** locally (availability is a real failure
  mode), variable/unknown context window, quality far below hosted frontier models.
- These differences drive `requires_api_key()`, `cost=None`, and the doctor checks.

## C. OpenRouter adapter (PR 1 — must be a no-op behaviorally)

- **Preserve current behavior:** move today's `openrouter.py` body verbatim behind the
  `Provider` interface as `OpenRouterProvider`. Same httpx call, same payload, same parse,
  same error mapping, same `usage` (incl. `cost`). The existing functions can remain as
  thin wrappers delegating to the default provider so nothing downstream breaks.
- **Keep OpenRouter the default provider** — with no provider configured, resolution
  returns OpenRouter, so existing users see zero change.
- **Env-var compatibility:** `OPENROUTER_API_KEY`, `OPENROUTER_API_URL`, and all
  `MODEL_*` overrides keep working unchanged. `_require_api_key()` stays valid for the
  default provider.
- **Preserve `--usage`, `--max-cost`, `--max-tokens`:** usage flows through unchanged
  (OpenRouter still reports `cost`); the only edit is making the cost-source label
  provider-aware instead of hardcoded. Token guard is provider-agnostic already.
- **Acceptance:** the full existing test suite passes untouched; `vibe review/diff/extract`
  output is byte-for-byte equivalent for OpenRouter users.

---

## D. Ollama provider (PR 3 — after the adapter is stable)

- **Minimal local provider:** `OllamaProvider` talking to the local Ollama HTTP API
  (default `http://localhost:11434`), `requires_api_key() == False`.
- **Local model selection:** models referenced by Ollama tags (e.g. `llama3.1`,
  `qwen2.5`); a small local preset or `--model`/env override path so users pick what they
  have pulled. Do **not** reuse OpenRouter `provider/model` IDs for Ollama.
- **No API key required:** the key guard must become provider-aware so Ollama runs without
  `OPENROUTER_API_KEY`.
- **Cost = zero/unknown/local:** `cost=None`, `cost_source=None`; `--usage` reports tokens
  if Ollama returns them, otherwise "not reported," and never invents a dollar figure.
- **Quality warnings:** explicit, visible warning that local models produce weaker reviews;
  surfaced in `vibe doctor`, docs, and possibly a one-line stderr note on local runs. This
  pre-empts the "weak local model → blame the tool" perception risk the prior review flagged.
- **Docs needed:** install/run Ollama, pull a model, select the Ollama provider, expected
  quality/limitations, offline/zero-cost framing, and that local-first ≠ best-quality.

## E. CLI / config changes (PR 2 — selection, OpenRouter stays default)

- **How the user selects a provider:** precedence `--provider <name>` (per-run) →
  `VIBE_PROVIDER` env → workspace `.council/config.json` → **default `openrouter`**.
- **Env / config precedence:** explicit flag wins, then env, then project config, then the
  built-in default. Provider-specific settings (`OLLAMA_HOST`) read from env with sane
  defaults. No precedence change for existing OpenRouter knobs.
- **Proposed commands / options:** add `--provider` to the mode commands; extend
  `vibe status` and `vibe models`/`vibe presets` to show the active provider and its
  resolved model IDs; (later) `vibe doctor` (Section F).
- **Backwards compatibility:** with nothing set, behavior is identical to v0.1. All
  existing flags (`--usage`, `--max-cost`, `--max-tokens`, `--save`, `--yes`,
  `--allow-repeat`, …) keep their meaning.
- **Migration story:** none required for OpenRouter users. New capability is purely
  additive; document the new `--provider`/`VIBE_PROVIDER` path in README + CHANGELOG.

## F. `vibe doctor` / validation (PR 4 — after selection exists)

A read-only, no-model-call (or cheapest-possible) diagnostic:

- **Provider reachability:** can we reach the active provider's endpoint?
- **OpenRouter key check:** present, non-placeholder (reuses `_require_api_key` logic),
  never prints the key.
- **Ollama server check:** is `localhost:11434` (or `OLLAMA_HOST`) up and responding?
- **Model availability check:** for the active preset's models, are they present (Ollama:
  pulled locally; OpenRouter: best-effort catalog/ID sanity, no spend)?
- **Version / dependency / env checks:** `vibe --version`, Python/venv sanity, presence of
  required env, workspace status. Exit non-zero on hard failures, warn on soft ones.
- Explicitly **after** provider selection lands, so it can report on the *active* provider.

## G. Cost and usage

- **Normalizing provider-specific cost:** keep the `Usage` contract; `cost` is populated
  only when the provider reports dollars (OpenRouter) and `None` otherwise. Replace the
  hardcoded "as reported by OpenRouter" string with a provider-aware label
  (`cost_source`). `aggregate_usage`/`enforce_cost_cap` logic is unchanged.
- **How the local provider reports usage:** tokens if Ollama returns them, else "not
  reported"; cost always `None`. `--max-cost` becomes a no-op (with an honest note) for
  providers that don't report cost — never a fabricated block.
- **What's required before a pre-flight cost estimate (deferred):** (1) provider pricing
  tables (per-model `$`/token), (2) `ModelInfo`/context metadata, (3) the existing
  token estimator. With those, a rough "≈ $X on OpenRouter vs. free on Ollama" line
  becomes possible — but it's a *later* item, not v0.2.
- **What stays best-effort:** the pre-run token estimate (no tokenizer), post-run cost cap
  (provider-reported only), and any pricing math. Honesty over precision remains the rule.

## H. Risks

1. **Behavior regression** for OpenRouter users — the abstraction must be a true no-op in
   PR 1. Mitigation: move code verbatim, keep wrappers, require the existing suite to pass
   untouched + an equivalence check.
2. **Cost-reporting mismatch** — conflating "cost not reported" with "cost zero." Mitigation:
   `cost=None` vs `0.0` are distinct; `--usage`/`--max-cost` messaging stays explicit.
3. **Local model quality** — weak reviews erode trust. Mitigation: loud quality warnings +
   `vibe doctor` + docs framing local as offline/zero-cost, not best.
4. **Provider-specific response formats** — Ollama's payload differs from OpenAI-shaped.
   Mitigation: normalization lives entirely in adapters; council never sees raw shapes.
5. **Tests becoming brittle** — real network calls in tests. Mitigation: keep provider I/O
   behind the interface and mock at the `Provider` boundary; no live calls in CI.
6. **Scope creep** — streaming, retries, MCP, pricing tables, web-UI alignment, demo media
   all tempt their way into "provider abstraction." Mitigation: the PR slicing below is the
   contract; each is explicitly deferred. **No MCP / SaaS / demo work in v0.2.0.**
7. **Provider-selection precedence confusion** — too many knobs. Mitigation: one documented
   precedence chain, surfaced in `vibe status`.

## I. PR slicing (recommended)

1. **Provider interface + OpenRouter adapter — no behavior change.** Introduce `Provider`,
   `ChatResult`, `Usage`, `ModelInfo`; move `openrouter.py` behind it as the default;
   council/decision_memory call the injected provider. Existing tests pass untouched.
2. **Provider selection (config/env) — OpenRouter remains default.** `--provider` /
   `VIBE_PROVIDER` / workspace config precedence; `vibe status`/`models` show the active
   provider. Purely additive.
3. **Minimal Ollama provider + local-model docs.** Localhost adapter, no key, `cost=None`,
   quality warnings, docs.
4. **`vibe doctor` + provider/model validation.** Reachability, key/server checks, model
   availability, env/version checks.
5. **Later: pre-flight cost estimate.** Needs pricing tables + model metadata (Section G).
6. **Later: experimental MCP.** Built on the now-stable provider interface; behind a flag.
7. **Much later: demo / asciinema polish.** Explicitly deferred per current strategy.

PRs 1–4 are v0.2.0/v0.2.1 scope; 5–7 are later. Each PR is independently reviewable and
ships with its own tests.

## J. Release / docs hygiene audit

### Finding: the v0.1.0 GitHub Release "README not found" link
- **Symptom:** on
  <https://github.com/EfeAydinalp/vibe-council/releases/tag/v0.1.0>, the README/CHANGELOG/
  demo-transcript links don't resolve ("README not found" / broken navigation).
- **Confirmed cause:** the release body was pasted verbatim from
  [`docs/releases/v0.1.0.md`](../releases/v0.1.0.md), which uses **relative** links:
  `[CHANGELOG.md](../../CHANGELOG.md)`, `[demo transcript](../demo-transcript.md)`,
  `[README](../../README.md#install-the-global-vibe-command)`. Relative links on a GitHub
  **Release page** resolve against the release URL (`/releases/tag/v0.1.0`), not the file's
  repo path — so `../../README.md` lands on `/releases/README.md`, which 404s. The same
  markdown renders fine when viewing the *file* in the repo tree, which is why it slipped
  through. (Verified: `gh release view v0.1.0` shows the relative links in the live body.)
- **Exact proposed release-note fix:** in the **release body** (and the paste-ready source
  doc, since its whole job is to be pasted), replace the three relative links with absolute
  blob URLs pinned to the tag:
  - `https://github.com/EfeAydinalp/vibe-council/blob/v0.1.0/CHANGELOG.md`
  - `https://github.com/EfeAydinalp/vibe-council/blob/v0.1.0/docs/demo-transcript.md`
  - `https://github.com/EfeAydinalp/vibe-council/blob/v0.1.0/README.md#install-the-global-vibe-command`
- **Should source docs also change?** Yes — update `docs/releases/v0.1.0.md` to the
  absolute, tag-pinned URLs so it stays genuinely "paste-ready," and add a one-line note to
  the release checklist: *release-note links must be absolute (tag-pinned blob URLs), never
  relative.* Other in-tree docs (README, CHANGELOG) keep relative links — they render
  correctly in the repo tree and only the **release page** is affected.
- **Should this be its own PR?** **Yes — a tiny, separate hygiene PR, before any provider
  implementation.** It's unrelated to the refactor, trivially reviewable, and shouldn't be
  entangled with v0.2 code. Editing the live release **body** itself requires explicit
  approval and is **not** done automatically here.

## Council guidance summary

Ran `review --preset balanced` against this doc (2026-06-28). Output is advice to filter,
not authority — the section below is the **human-curated** reading, not a verbatim apply.
Raw output stayed in the gitignored `.council/`. The council graded the plan "75%
excellent" — strategy, PR slicing, and cost-handling design endorsed — but flagged three
genuine defects to fix before coding.

### Is provider abstraction still the correct v0.2 priority?
**Yes — unchanged.** The council endorsed the strategy, the four-PR slicing, and OpenRouter-
adapter-first sequencing. Nothing here reorders v0.2; the feedback is about *how* to build
PR 1 safely, not *whether* to.

### Strongest architectural warning
**"Verbatim move + no behavior change" is overstated and must be made precise.** Introducing
`ChatResult`/`Usage` *does* change the call sites in `council.py`/`decision_memory.py`, so
PR 1 is a behavioral no-op (identical outputs) but **not** a code no-op. Curated resolution:
keep the legacy `query_*` functions returning **today's dict shape** as the seam, build
`ChatResult` internally, and rewrite Section C to say "byte-equivalent output, existing
tests green" rather than "verbatim." A throwaway **spike** (interface + one updated call
site) should validate this before committing to PR 1's scope. Adopt.

### Preferred provider interface shape
Adopt two council refinements: (1) **`chat(request: ChatRequest) -> ChatResult`** with a
`ChatRequest` dataclass, so future optional params (temperature, json_mode) don't churn
every adapter signature; (2) **trim PR 1's protocol to `name` + `chat` +
`requires_api_key`**, and **defer `describe_model()`/`ModelInfo`/`health()`/`HealthStatus`
to PR 4** (`vibe doctor`) so we don't lock those shapes prematurely. Make `Usage` a
validated dataclass. Keep `raw` but mark it internal/never-persisted (see cost/security).

### What should be in PR 1
Interface (`Provider` + `ChatRequest`/`ChatResult`/`Usage`) + `OpenRouterProvider` as the
default, with council/decision_memory calling the injected provider. **Plus three
prerequisites the council rightly demands first:** (a) a **failure-propagation inventory**
(how partial stage failures cascade / graceful degradation today), (b) a **test-mocking
inventory** (what the suite mocks — `httpx` internals vs. `query_*`) so "tests pass
untouched" is verifiable, and (c) the spike above. Drop `guards.py` from PR 1's churn — it
only aggregates usage and needs no provider awareness yet (good catch).

### What should be delayed
`describe_model`/`health` → PR 4. Pre-flight cost estimate, MCP, retries, streaming,
pricing tables, demo media → later, as already planned. `reasoning_details` stays
display-only in v0.2 (never parsed).

### How to handle Ollama
- **Fix the PR 2/3 sequencing hazard:** PR 2 (provider selection) must **hard-error on a
  not-yet-implemented provider** ("`ollama` not available until v0.2.x") rather than leave
  `VIBE_PROVIDER=ollama` undefined. Adopt.
- **Provider-aware model selection:** OpenRouter-namespaced `MODEL_*`/preset IDs must not be
  silently passed to Ollama. Namespace them or hard-error on mismatch with an actionable
  message (`Model "x" not found. Run \`ollama pull x\``). Adopt.
- **Local-quality guard:** don't just warn — running a frontier-tuned default preset on a
  local provider should require explicit acknowledgment (a local preset or
  `--allow-degraded-quality`/`--yes`). Adopt as a v0.2 constraint.
- **SSRF:** restrict `OLLAMA_HOST` to localhost by default, gate remote behind an explicit
  flag. Adopt.

### How to handle cost reporting
The core design (`cost=None`, never fabricated, honest messaging) was praised. Two
adopted tweaks: (1) distinguish **"known free" vs "unmeasured"** via `cost_source`
(`"provider"｜"free"｜None`) so aggregation never treats unknown as zero; (2) for
`--max-cost` on a cost-unreporting provider, **prefer a clear note over a hard error** —
I *partially* diverge from the council here: a hard failure on an explicitly-passed
`--max-cost` is defensible, but silently ignoring is the real sin, so an explicit,
visible "`--max-cost` cannot be enforced on local providers" note (already the plan's
intent) is acceptable if it's loud. Revisit if it generates confusion.

### How to handle release/docs hygiene
Council agreed: the relative→absolute link fix is **correctly scoped as a tiny, separate
hygiene PR that lands first**, independent of provider work; do not auto-edit the live
release body. Bonus adopted: add a `scripts/prepare_release_notes.py <tag> <source>` (or a
checklist line) so relative-link regressions can't recur.

### Recommended next 3 actions
1. **Revise Section C + interface before any code:** restate PR 1 as a *behavioral* no-op,
   adopt `ChatRequest`, trim the PR 1 protocol, and produce the failure-propagation + test-
   mocking inventories. Validate with a throwaway spike.
2. **Land the release-link hygiene PR** (relative → tag-pinned absolute blob URLs in
   `docs/releases/v0.1.0.md` + checklist note; live-body edit only on explicit approval).
3. **Then implement PR 1** (interface + OpenRouter adapter, default unchanged) with the
   PR 2 hard-error-on-unimplemented-provider guard designed in, so the Ollama sequencing
   hazard is closed before PR 3.

### Where I diverge from the council (curated, not blindly applied)
- The "fundamental contradiction" framing is too strong — it's a **wording/scoping defect**,
  not a design dead-end; the dict-seam keeps PR 1 safe. Adopt the fix, reject the alarm.
- On `--max-cost`: lean toward a **loud note** over a hard error, pending real support-load
  signal.
- Defer (don't adopt now) the heavier suggestions — `Decimal`/currency on `Usage`, provider
  version-handshake matrix, `max_concurrency` knob — as **post-v0.2** hardening; they're
  real but would bloat a two-provider v0.2. Inverting to "Ollama-first" is explicitly **not**
  adopted (council itself rated it not-recommended).
