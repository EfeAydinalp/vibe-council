# 05 — v0.6.0 Agent-to-Workbench Bridge (design)

The keystone phase. This document is the design; [06-proposal-schema.md](06-proposal-schema.md) is
the concrete schema. Read [03-security-invariants.md](03-security-invariants.md) first.

## Design decision: file/CLI intake, not a network endpoint

**Do not add a network endpoint for agents.** The panel's HTTP surface is the most sensitive asset
in the product. The minimum viable bridge is a **local, file-or-CLI-based proposal intake**: an agent
writes a proposal JSON (or pipes it to a CLI command), and a **server-side importer** validates it,
mints ids, builds and hashes the payload artifact, and creates the approval — all locally. This adds
**zero new listening surface** and keeps the deterministic trust boundary as the sole gate.

Proposed CLI:

```sh
vibe workbench propose < proposal.json      # read a proposal from stdin
vibe workbench propose --file proposal.json # or from a file
```

(Exact flags are for the implementer to finalize; the shape is: read JSON, validate, import, print a
result with the minted ids + the panel URL to approve at.)

## The flow

```
agent writes proposal.json  (schema in 06-proposal-schema.md)
        │
        ▼
vibe workbench propose  →  importer validates the envelope + schema version
        │                   rejects disallowed kinds / fields outright (fail closed)
        ▼
dedup on proposal_id  →  if already imported, return the existing task/approval (no recreate)
        │
        ▼
server mints task_id (+ action_id for file/command kinds)   [ids are NEVER agent-supplied]
        │
        ▼
for file kinds: server builds the payload artifact FROM the submitted content,
                computes the hash SERVER-SIDE, saves write-once
        │
        ▼
server creates ApprovalRequest(status=pending) + Action(status=pending)
        │
        ▼
deterministic trust boundary evaluates  (unchanged)  →  advisory auditor rewrites  (unchanged)
        │
        ▼
panel shows a "proposed by agent: <name>" card
        │
        ▼
human approves  →  existing guarded executor runs the action  (UNCHANGED path)
```

The executor, trust boundary, payload verification, and panel-execute path **do not change**. The
bridge only adds an **intake** upstream of the approval. That is the whole point: none of the v0.5
guarantees move.

## Server-minted vs. agent-supplied (critical)

**Server mints / computes (never trust from the client):**
- `task_id`, `action_id`, `approval_id`
- `payload_hash` (over `{kind, target, payload}`, via existing `canonical_payload_hash`)
- all `status` fields, `created_at`, audit results/verdicts, risk levels

**Agent supplies (validated, then used only as *content*):**
- `proposal_id` (for dedup), `agent` {name, role, session}, `title`, `summary`
- `action.kind`, `action.target` (file kinds) or `action.command_label` (run_command)
- `action.payload` (file kinds only), `action.scope` hints

If a proposal contains a server-minted field (a hash, an id, a status, a verdict, a "pre-approved"
flag), **reject it** — do not silently ignore it. Asserting identity/verdict is a red flag.

## Payload artifact ordering (invariant #9 + #3)

The correct order is: **mint action id → build artifact from submitted content → hash server-side →
save write-once.** The client never supplies the hash. At execution time, the existing
`verify_payload_against_action` still re-checks hash self-consistency plus kind/target/action/
approval/task linkage — the bridge simply becomes another way to *populate* the artifact, not a new
trust path.

## Dedup / idempotency (roadmap risk #6)

An agent that retries (crash, timeout, double-send) must not create duplicate tasks/approvals.
Dedup on the agent-supplied `proposal_id`: on a repeat, return the existing `{task_id, approval_id,
action_id}` instead of creating new records. Store the mapping in the runtime tree. A missing
`proposal_id` should be rejected (it's required for safe retries).

## `requested_action` stringly-typed convention — replace at the boundary

Today `requested_action` uses a `kind:target` colon convention parsed in several places
(`workbench_executor._approved_target`, `workbench_auditor._payload_from_approval`). That's fragile
for internal callers and unacceptable as an **external** authorship contract (a `run_command` target
contains a colon). The importer must accept **typed fields** (`kind`, `target`/`command_label`,
`scope`) and construct the internal records itself — do not make agents author against the colon
convention.

## What the panel needs to display

- A "**Proposed by agent: `<name>` (`<role>`)**" badge on task/approval/action cards.
- The agent's `summary` as human context (escaped; never raw payload content).
- No new execute affordance — proposed actions flow into the exact same approve→execute path.
- (Nice-to-have, not required: a filter by source.)

## Non-goals for v0.6.0

- No new network endpoint for agents. No auto-execution. No broad/arbitrary shell.
- No command-allowlist expansion. No `cloud_call` proposals.
- No agent-supplied hash/ids/status/verdicts. No relaxing of any invariant in
  [03-security-invariants.md](03-security-invariants.md).
