## What this release covers

Governed by [`docs/plans/v0.5.1-dogfood-hardening.md`](../plans/v0.5.1-dogfood-hardening.md)
(PR #85's checklist), closed out by PR #86–#90:

### Clean-clone / Windows dogfood (PR #86)

A fresh `git clone` → `uv sync` → version/doctor/status/presets → the full test suite →
lint/decisions/context/MCP checks, plus an HTTP-level Workbench panel smoke (binding, page
content, token gating) — everything matched the existing dev checkout (0 critical redaction,
21/21 context/MCP, 570 tests at the time). Found and fixed three small docs bugs: a README
`uv sync` → bare-`python` quickstart example that fails without `uv run`; stale
`vibe workbench serve --help` text that predated real execution (PR #77/#80/#81); and a missing
venv-interpreter note in the dogfood checklist itself. **No executor/panel/CLI behavior changed**
— docs only.

### Workbench interactive smoke (PR #87)

A clean scratch-directory first run (`VIBE_CALLER_CWD` pointed at an empty temp dir) confirmed
the empty state, the "Create demo task" seed, approve/reject/hold, and token gating all matched
expectations. Investigated a leftover-process symptom from PR #86 and confirmed it was a
`uv run` process-tree artifact (killing only the outer `uv` process can leave its Python child
running), not a Workbench bug — a direct venv-python launch always released the port cleanly. A
simulated Windows `CTRL_C_EVENT` didn't trigger shutdown within 8s in automated testing either
way, but no human-attended real interactive terminal was available in that session to confirm or
rule out a real bug, so it was left **inconclusive** rather than reported as confirmed.

### Localhost bind/shutdown hardening (PR #88)

A maintainer's real, human-attended Ctrl+C test (`uv run python -m backend.cli workbench serve`,
physical Ctrl+C, prompt returned immediately) was clean. A `netstat` listener seen afterward on
the same default port was fingerprinted by its HTTP response and identified as an **unrelated,
pre-existing local service** (`PhoneScriptRunner`) coincidentally on the same port — not
vibe-council. This corrects PR #87's "inconclusive" classification: **no confirmed shutdown or
`0.0.0.0`-bind bug** exists or ever existed. Hardening was added anyway, as defense-in-depth, not
as a fix for a confirmed bug: `make_server()` now re-checks the actual bound address after
`bind()` (refusing/closing on anything non-loopback), a pure `effective_bind_host()` /
`_startup_lines()` pair so the printed URL can never drift from what's really bound, and a new
regression test that simulates Ctrl+C (`KeyboardInterrupt` from `serve_forever()`) and asserts
`server_close()` actually closes the socket. No change to the blocking serve loop, no new CLI
flag.

### Manual execution dogfood (PR #89)

The one exit-criterion PR #86/#87 hadn't covered: the real guarded-executor path through the
panel's actual HTTP API (the demo deliberately seeds no `Action`, so this had never been
exercised end-to-end via the panel before). In a throwaway temp/safe project (never this repo):

- A manually-seeded, approved `write_file` action executed successfully — wrote exactly the
  server-resolved payload, byte-for-byte; a crafted request body (different target/content) had
  **zero effect**.
- A manually-seeded, approved `run_command` action (`git status --short`) executed successfully —
  ran exactly the server-resolved, allowlisted argv; a crafted request body (different
  command/argv/env/cwd/timeout, including a destructive `rm -rf /`) had **zero effect**.
- A non-allowlisted command and a `write_file` action with no payload artifact both showed no
  execute affordance in the panel, and both failed closed even when the execute endpoint was hit
  directly.
- No raw payload content ever appeared in any HTTP response, before or after execution.

Found and fixed two small **display/metadata** bugs — neither changes what the executor or trust
boundary allows or blocks:

- `ExecutionResult.dry_run` incorrectly stayed `true` on every real execution response
  (`executed: true` alongside `dry_run: true` — self-contradictory). Now correctly `false` for
  real executions.
- A completed or blocked `run_command` action's panel card could falsely claim "does not resolve
  to an allowlisted argv" even after it had already run successfully — the fallback text was only
  ever correct for a still-pending, genuinely-unresolvable action. The card now shows the actual
  target/status for a non-pending action instead.

### `uv.lock` self-version hygiene (PR #90)

`uv.lock`'s `[[package]]` entry for the local `vibe-council` project had read a stale
`version = "0.2.0"` since before `v0.3.0` — `pyproject.toml`/`backend/__init__.py` stayed
correctly in sync with each other across every release since, but nobody re-ran `uv lock` and
committed the result, so every `uv sync`/`uv run` produced an accidental one-line `uv.lock` diff
that had to be deliberately left unstaged. Synced the entry (confirmed deterministic and
idempotent — a second `uv sync` afterward produces no further diff); no dependency package
versions or resolution graph changed. This release's own version bump follows
[`docs/release-checklist.md`](../release-checklist.md)'s documented (but previously unfollowed)
step of running `uv lock` after every version bump, so the pattern doesn't recur at the next
release.

## Security posture — unchanged from v0.5.0

Nothing in this patch relaxes or expands the security model:

- **Approval is separate from execution** — approving only records a decision; it never writes a
  file, edits a file, or runs a command.
- **The panel is localhost-only (`127.0.0.1`)** and **POSTs are token-gated** — no LAN/mobile/
  remote surface.
- **Payload artifacts stay local and gitignored**, write-once, hash/scope-verified before every
  real execution.
- **Command execution is fixed argv + `shell=False`, always** — no dynamic arguments, no
  `shlex`/string parsing, no shell.
- **The deterministic trust boundary re-runs at execution time**, not just at approval time; the
  advisory Approval Auditor still cannot relax a block.
- **No new command allowlist entries** — still the same 7-entry, verification/status-only
  allowlist as v0.5.0.

## Explicit non-goals (deferred, not started in this release)

## Known limitations (unchanged from v0.5.0)

- **Intentionally local-only** — no LAN/mobile/remote access; the panel binds `127.0.0.1` only.
- **Intentionally tiny command allowlist** (7 entries — verification/status commands only); not
  meant to grow casually.
- **No hosted/team/mobile/voice** yet — deferred to v0.6+ at the earliest.
- **The panel is intentionally simple** — no framework, no rich UI, no persistence beyond the
  runtime JSON store.
- **The demo still seeds no executable `Action`** (a deliberate v0.5.0/PR #77 safety choice) —
  real execution dogfood requires manual seeding in a temp/safe project, as PR #89 did.
