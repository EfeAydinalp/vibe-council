# v0.4 read-only MCP — local dogfood notes

_Date: **2026-07-01**. Public-safe observations only — no secrets, no local absolute paths, no raw
council output. All commands are read-only and local._

## Purpose

Dogfood the v0.4 read-only MCP workflow (contract → inspect → stdio transport) on this repo before
v0.4 release prep, and verify the read-only / no-write / privacy boundaries hold in practice. See the
setup guide: [`docs/mcp/claude-code-setup.md`](../mcp/claude-code-setup.md).

## Commands run

- `vibe mcp contract`
- `vibe mcp inspect --context --health`
- `vibe mcp serve --help`
- `vibe mcp serve --stdio` driven by a bounded set of JSON-RPC requests (initialize, tools/list,
  tools/call, resources/list, resources/read, a forbidden tools/call)
- supporting: full test suite, `lint --redaction`, `decisions lint`, `context build`/`check`/`export`,
  `operator status`

## Expected output summary

- `vibe mcp contract` — prints the 7 read-only resources, 5 read-only tools, and the 10 forbidden
  tools; exit 0.
- `vibe mcp inspect --context --health` — enabled resources/tools listed; `get_project_status`
  ~4.5k chars; `list_decisions` 31 curated records; `get_context_pack` ~10.7k chars (in-memory, not
  written); `check_context_health` **21/21 (100%)**; exit 0.
- `vibe mcp serve --help` — exit 0.

## No-write audit

- `.council/context/pack-latest.md` — **not created/modified** by MCP calls (mtime unchanged across
  the smoke run).
- `.council/context/claude-code-context.md` — **not created/modified** by MCP calls (mtime
  unchanged).
- `get_context_pack` / `check_context_health` build the pack **in memory** (via `context_pack`) and
  write nothing. `.council/` generated files remain CLI-only outputs.

## Privacy audit

- **The private commercial feasibility plan** (untracked, under `docs/plans/`) — **not exposed** by
  any resource/tool (the server only reads `docs/decisions/` + `STATUS.md`; plans are not a resource).
- **The untracked v0.3.1 hardening plan** (under `docs/plans/`) — **not exposed**.
- **Secrets / local state** — no `OPENROUTER_API_KEY`, `sk-or-v1-` key, real `.env`/`.venv`/`data/`
  path, or local absolute path appears in served content. (The pack's Constraints section *mentions*
  the strings `.env`/`.venv/`/`data/` only as part of the safety boundary text — "never exposed" —
  which is documentation, not a leak.)
- `show_decision` with a traversal id (e.g. `../plans/...`) returns "not found" and reads nothing
  outside `docs/decisions/`.

## Findings

- **Config friction:** minimal. The generic `mcpServers` stdio pattern works; the only nuance is
  ensuring the client launches the server in the target repo (the server reads the current project).
- **Docs gaps:** the exact Claude Code registration syntax isn't asserted — the setup doc uses a
  clearly-labeled *generic MCP stdio client pattern* and points to the client's own docs. That is the
  honest state until verified against Claude Code directly.
- **Is the transport usable for v0.4?** Yes. A client can initialize, list/read resources, and call
  the read-only tools over stdio; the read-only + no-write + privacy boundaries hold; health is
  21/21. It's a solid minimal read-only server for v0.4.
