# vibe-council v0.4.0 — read-only MCP / Claude Code workflow

The **read-only MCP** release. vibe-council can now serve its curated project memory — project
status, curated decisions, and the generated context pack + health — to Claude Code and local MCP
clients over a **read-only** stdio server, **without granting any write/action authority**. Still
deterministic and local-first: the transport is a minimal **stdlib JSON-RPC-over-stdio** server, so
there is **no `mcp` SDK dependency**; normal `vibe` usage is unaffected. Based on and crediting
[`karpathy/llm-council`](https://github.com/karpathy/llm-council).

The full changelog is in
[`CHANGELOG.md`](https://github.com/EfeAydinalp/vibe-council/blob/v0.4.0/CHANGELOG.md).

> **No commercial-clearance claim.** License/provenance remains an unresolved **"Question 0"**
> commercial gate; no `LICENSE` is added.

## Highlights

- **Read-only MCP contract** — `vibe mcp contract` prints the read-only resources/tools and the
  explicit forbidden mutation/action tools.
- **Dependency-free read layer** — `vibe mcp inspect` runs a read-only smoke over status + curated
  decisions, and (with `--context`/`--health`) the context pack + health, built **in memory** (no
  `.council/` files written).
- **Read-only stdio transport** — `vibe mcp serve --stdio` speaks newline-delimited JSON-RPC 2.0 for
  the surface above. **No `mcp` SDK, no HTTP port, no daemon.**
- **Context-pack critical-section stabilization** — core sections (decision index,
  rejected-alternatives index, human-review / source-of-truth constraints, status) are **compacted,
  not dropped**, under budget pressure; full decision bodies trim first.
- **Setup docs + dogfood** — a generic MCP stdio client pattern and a recorded dogfood pass with
  stdio smoke, no-write, and privacy audits.

```sh

vibe mcp serve --stdio               # read-only MCP stdio server (no write tools)
```

## MCP resources / tools

Resources: `vibe://status` · `vibe://decisions` · `vibe://decisions/{id}` · `vibe://context/latest`

Tools: `get_project_status` · `list_decisions` · `show_decision` (path-guarded to `docs/decisions/`)
· `get_context_pack` · `check_context_health`

## Safety boundaries

- **Read-only only** — no write/git/shell/provider/model tools; no decision promotion through MCP;
  `git_status` is forbidden for v0.4.
- **No raw `.council/` exposure** and **no private/untracked plans** exposed.
- **MCP context reads do not write `.council/`** (built in memory); the forbidden `write_file` tool
  returns a JSON-RPC error.
- `.council/` generated artifacts (packs, exports, drafts, operator status) stay **local/gitignored**
  — they are CLI outputs, never MCP side effects.
- **License/provenance remains "Question 0"** — no commercial offering, no `LICENSE`, no clearance
  claim.

## Known limitations

- The MCP transport is a **minimal stdlib JSON-RPC** implementation of the stdio methods, not the
  official `mcp` SDK; fuller spec compliance is a future *optional* extra.
- Exact Claude Code registration syntax is **not asserted** — the setup doc uses a clearly-labeled
  generic MCP stdio client pattern; verify against your client's own docs.
- The context budget is still a **naive char budget** (default 14000), now with core-section
  compaction; a token-aware budget remains deferred.
