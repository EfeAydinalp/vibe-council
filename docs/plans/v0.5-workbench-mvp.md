# Plan: v0.5 — AI Council Workbench MVP (vertical slice)

A **planning document**, not an implementation. v0.4.0 shipped the read-only MCP / Claude Code
workflow. A roadmap-correction council pass (`vibe review` + `vibe full`, `balanced`) then reviewed
everything built since v0.2.0 and corrected the direction toward a **user-visible product loop**.
This plan promotes that outcome into a concrete v0.5 target. **No runtime models, store, panel, or
server are implemented in this task.**

Review date: **2026-07-01**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local.

> **Scope guard.** v0.5 is a **vertical slice**, not more horizontal infrastructure. It adds the
> smallest end-to-end loop (task → visible stages → audited approval → safe execution → logged) for
> **one** workflow. No mobile, voice, personalization, hosted/sync, new MCP surfaces, new CLI
> namespace (unless required), or plugin system. License/provenance remains the unresolved
> **"Question 0"** commercial gate — untouched here.

---

## 1. Purpose

Since v0.2.0 vibe-council has built strong foundations (provider abstraction + Ollama + `doctor`,
decision memory + promotion, context packs, redaction guard, Claude Code export, operator status,
and the read-only MCP contract/read-layer/context-health/stdio transport). The council's diagnosis:
**strong engine, no steering wheel** — there is no user-visible product loop. v0.5 builds that loop
as a minimal, demo-ready **AI Council Workbench**, reusing (not expanding) the existing
infrastructure.

## 2. Council input summary

The correction was produced locally by `vibe review` (single consolidated) and `vibe full`
(multi-model peer-ranked) over a roadmap-correction brief; the raw input, raw outputs, and local
synthesis stayed under gitignored `.council/roadmap-inputs/`, `.council/reviews/`, and
`.council/runs/` (they are **source material only**, never committed). Both passes converged on:

- Infra since v0.2.0 is **foundation, not waste** — leverage it, stop expanding it.
- Build a **vertical slice** in v0.5, not horizontal layers; the panel + progress + **audited
  approval** ship together (a progress bar you can't act on has little value).
- The **Approval Auditor ships with the first approval UI**, but is **advisory** — **deterministic
  guards are the real security boundary**.
- Start with **Task/Stage/Approval data models + a structured `.council/runtime/` store**; Markdown
  stays a human projection, not the runtime DB.
- **Single-model + audited approval is the MVP**; wiring the existing multi-model council as a stage
  is **optional/last**.
- Mobile → voice → personalization come **later**; **"AI Council Workbench"** is the near-term
  product name; **"local-first AI project OS" is long-term/internal**, not near-term external
  messaging.

## 3. What changed from the previous roadmap

- **Was:** v0.5 = "Workbench panel + visible stage/progress indicator" as a standalone step, then
  v0.6 approval inbox, v0.7 mobile, v0.8 voice (horizontal layers).
- **Now:** v0.5 = a **complete minimal loop** (panel **and** audited approval together) for one
  workflow; mobile/voice/personalization are explicitly deferred to v0.6+.
- **Security framing corrected:** the Approval Auditor is not the boundary; **deterministic guards
  are**. AI adds a risk label + a short human-readable rewrite only.
- **Messaging corrected:** stop external "AI project OS" language; ship as "AI Council Workbench."

## 4. Product target: AI Council Workbench

**AI Council Workbench** — supervised, local-first AI-assisted work where AI proposes and **you
approve** before anything runs, with a visible stage/progress panel and an audited approval step.

- **Near-term external messaging:** *"AI agents discuss your work before acting — you're the final
  approver, and everything important is logged."*
- **Long-term internal vision (not external yet):** a broader local-first "AI project OS." Keep this
  as an *architectural* north star only; do not use it in external messaging until post-v1.0.
- **Long-term direction (v1.x, not now — do not implement):** the Workbench may later sit **between**
  normal chat/voice surfaces and coding/operator agents as a task-shaping + routing layer:
  natural chat or voice input → prompt/task refinement → council/worker review → deterministic guards
  + Approval Auditor prepare an approval card → user approves/rejects/holds (panel/mobile/voice) →
  approved action proceeds → decision/status/context logged. This is **out of scope** until after the
  Workbench MVP, approval flow, panel, mobile, and voice land.

## 5. v0.5 MVP workflow (pick ONE)

**"Safe repo changes with approval"** — the single, high-value workflow that constrains scope:

1. User provides a task (manual entry; no chat UI required yet).
2. A **single model** produces a plan + a proposed **file diff** (reuse `vibe review`/`diff` shape).
3. The plan/diff becomes an **ApprovalRequest** with proposed **Actions** (file writes only, at
   first).
4. **Deterministic guards** evaluate the actions; the **Approval Auditor** adds an advisory risk
   label + a short human-readable summary.
5. The panel shows stages + the pending approval; the user **approves / rejects / holds**.
6. On approve, only allowlisted actions run; the result + decision are **logged** (and can be
   promoted to `docs/decisions/` by the existing human-reviewed flow).

## 6. Data-model foundation (the first implementation PR)

Small, JSON-serializable, stdlib-only. Start with ~5 entities and a fixed state machine — resist a
generic workflow engine.

- **`Task`** — `id`, `user_input`, `status`, `stages[]`, `approval_request?`, `created_at`,
  `completed_at`.
- **`Stage`** — `name`, `status` (`pending|active|completed|failed`), `messages[]`, `artifacts[]`,
  `started_at`, `completed_at`, `worker` (model/role).
- **`ApprovalRequest`** — `id`, `task_id`, `summary`, `actions[]`, `audit` (`AuditResult`),
  `decision?` (`ApprovalDecision`), `external_access?` (cloud-egress consent block).
- **`ApprovalDecision`** — `id`, `request_id`, `verdict` (`approve|reject|hold`), `note?`,
  `decided_at`, `decided_by`.
- **`Action`** — `type` (`write_file` — first; later `execute_command`, `api_call`), `details`
  (`path?`, `diff?`, `command?`), guard evaluation result.
- **`AuditResult`** — `auditor_model`, `is_safe` (advisory), `risk_level` (`low|medium|high`),
  `human_summary`, `warnings[]`, `block_reason?`.
- **`Notification`** (optional, later) — `id`, `task_id`, `kind`, `message`, `created_at`.

## 7. Runtime store

- Location: **`.council/runtime/`** (gitignored, local), alongside the existing `.council/`
  workspace.
- Format: **JSON first** (one file per task, or an append log + per-task state) — stdlib `json`
  only; **no new dependency**, no SQLite yet (revisit only if JSON proves insufficient).
- Role: the **canonical live state** that the orchestrator writes and the panel reads. **Markdown
  under `docs/decisions/` stays a human projection** of *approved, completed* outcomes — never the
  runtime DB. Curated decisions remain the committed source-of-truth; runtime state stays local.

## 8. State machine

A fixed, small progression (not a generic engine):

```
PLANNING → AWAITING_APPROVAL → EXECUTING → COMPLETED
                     │
                     ├─ HELD      (user paused; can resume)
                     └─ (reject)  → COMPLETED (no actions run)
Any stage → FAILED   (model refusal, guard block, timeout, crash)
```

- Each stage carries `pending | active | completed | failed`; the active stage names the current
  worker/model and the next action.
- Explicit failure handling is required (model refuses, guard blocks, auditor times out) — degrade
  gracefully (e.g., fall back to single-model; surface a clear failure, never a silent hang).

## 9. Deterministic trust boundary (the REAL security gate)

Enforced in **code**, evaluated **before** any approval UI and before any action runs — the Auditor
does not gate:

- **Path allow/deny** — writes only under the project root; deny system paths and `~/.ssh/` etc.
- **Command allowlist / templates** — no arbitrary shell; only templated, reviewed commands (exec
  actions are a later increment).
- **Secret-file patterns** — refuse to read/write/expose `.env`, `*.key`, `*.pem`, token files.
- **Change-size limits** — max files / max lines changed per approval; larger → forced high-risk /
  block.
- **Cloud-egress consent** — an `ApprovalRequest.external_access` block (`needs_cloud`, `endpoint`,
  `data_shared`, `reason`); private/sensitive context requires explicit approval before leaving the
  machine; **redaction runs before any provider call**. Reuses the existing redaction guard.

## 10. Approval Auditor AI

- **Advisory only** — it never grants or denies; the deterministic guard (§9) is the boundary.
- **Hybrid & simple** — deterministic static checks first, then **one low-temperature AI call** for:
  a **risk label** (`low|medium|high`), a **short human-readable approval rewrite** (~20 words), and
  a flag to **block/escalate** an obviously suspicious request (too broad, risky command, secret/
  private-context leakage, scope violation).
- **Different model family** from the worker where practical (reuse the provider abstraction), so the
  auditor isn't the same model it audits.
- Output is a structured `AuditResult`; treat it as untrusted advice, not a guarantee.

## 11. Panel / progress indicator

A minimal local surface (HTML+JS or TUI — functional over fancy) reading the `.council/runtime/`
store and showing:

- the **current stage** and **completed stages** (the ChatGPT-style checklist feel);
- **pending approvals** with the auditor's risk label + human summary;
- the **active worker/model** and the **next action**;
- **approve / reject / hold** controls that write an `ApprovalDecision` and advance the state
  machine.

Latency/time-to-demo budgets matter: bound how long a stage can run before the user abandons; keep
the first update fast.

## 12. Which existing `vibe` commands the Workbench wraps

The Workbench orchestrates existing surfaces rather than reinventing them:

- **`vibe status`** — workspace/state entry point the panel builds on.
- **`vibe review`** / **`vibe diff`** — the plan/diff producers behind a task's proposal.
- **`vibe decisions`** — promote approved, completed outcomes into curated `docs/decisions/`.
- **`vibe context`** — the context pack feeding the workers.
- **`vibe operator`** — the minimal local status surface becomes the panel's backend state.
- **`vibe mcp`** — stays the **read-only knowledge source** (agents read curated memory; no writes).

## 13. What NOT to build yet

- **Mobile** (LAN web + token/QR) — v0.6+.
- **Voice** (local STT/TTS over the same API) — v0.7+.
- **Personalization / workflow memory** — v0.8+.
- **Hosted / team / sync** and **remote approval transport** — post-1.0, only after security design.
- **New MCP tools/resources or a new CLI namespace** — freeze surface growth until the panel exists
  and its APIs stabilize; the panel becomes the primary interface.
- **Plugin system**, native mobile apps, cloud voice APIs.
- **External "AI project OS" messaging** — internal/architectural only until post-v1.0.

## 14. Suggested v0.5 PR sequence (small, ordered)

1. **Roadmap decision + plan** (docs-only). ✅ *done.*
2. **Data models + `.council/runtime/` JSON store** (stdlib-only; no server). ✅ *done.*
   `backend/workbench_runtime.py` adds `Task` / `Stage` / `ApprovalRequest` / `ApprovalDecision` /
   `Action` / `AuditResult` dataclasses + a gitignored `.council/runtime/{tasks,approvals,audits,
   actions}/ + index.json` store (atomic-ish, sorted-keys JSON; id-sanitized, containment-guarded;
   no server/model/network). See [runtime store decision](../decisions/2026-07-01-workbench-runtime-store.md).
3. **Task orchestrator / state machine** (deterministic). ✅ *done.*
   `backend/workbench_orchestrator.py` manages the task lifecycle over the runtime store
   (`start_task` → `add_planning_stage` → `request_approval` → `decide_approval` (approve/reject/hold)
   → `mark_executing` → `complete_task` / `fail_task` / `hold_task`), plus `get_task_progress`
   (panel-friendly) and `list_pending_approvals`. **No real actions run** — an approved task records a
   `pending` Action for a later executor; no model/provider/git/shell calls. See
   [orchestrator decision](../decisions/2026-07-01-workbench-orchestrator-state-machine.md).
4. **Deterministic trust boundary** (paths / command allowlist / secret patterns / size limits /
   cloud-egress consent) — the real gate, before any approval UI. ✅ *done.*
   `backend/workbench_trust.py` deterministically evaluates a proposed action → allowed / blocked /
   requires-approval + risk + findings + cloud-egress consent. Unknown kinds and non-allowlisted /
   metachar commands are blocked; writes require approval; secrets/`.git`/`.council`/private plans
   and out-of-project paths are blocked; cloud calls need explicit consent and never auto-run. It
   **executes nothing** and is the boundary the future executor and the (advisory) Approval Auditor
   must consult. See [trust boundary decision](../decisions/2026-07-01-workbench-trust-boundary.md).
5. **Approval Auditor (advisory)** — ✅ *done (deterministic first version).*
   `backend/workbench_auditor.py` runs the deterministic trust boundary, then produces an
   `AuditResult` (risk level, findings, blocked flag, a short human-readable approval-prompt rewrite).
   **Advisory only:** risk/blocked/findings are copied verbatim from the trust evaluation, so a
   blocked/high-risk result stays blocked/high-risk — the auditor can never relax the guard.
   `model="deterministic"` (no LLM); an optional model-assisted rewrite is a later PR. No execution,
   no provider calls. See [approval auditor decision](../decisions/2026-07-01-workbench-approval-auditor.md).
6. **Local web server + minimal panel** — ✅ *done.* `backend/workbench_panel.py` +
   `vibe workbench serve`: a **localhost-only** stdlib `http.server` panel that renders task
   progress + pending approval cards (with advisory audit summaries) and records approve/reject/hold
   via the orchestrator. Binds `127.0.0.1` only (no LAN/mobile); POSTs require a startup token; inline
   HTML/CSS, no external assets. **Decisions only — no action execution, no provider/model calls.**
   See [local panel decision](../decisions/2026-07-01-workbench-local-panel.md).
   *Dogfood polish:* an empty runtime now shows a self-explanatory empty state + a **"Create demo
   task"** button (`POST /api/tasks/demo`, token-gated) that seeds one task + planning stage +
   pending approval + a saved advisory audit — **local runtime only, executes nothing** — so a
   first-time user immediately sees the see→decide loop.
7. **(optional/last)** wire the existing multi-model council as a discussion stage; e2e test + docs.

## 15. Acceptance criteria

- One end-to-end workflow ("safe repo changes with approval") runs: task → visible stages →
  deterministic-guard-checked, auditor-annotated **ApprovalRequest** → approve/reject/hold →
  logged outcome.
- **Deterministic guards** block unsafe actions regardless of the auditor; a test asserts the
  boundary is code, not the AI.
- Runtime state lives in gitignored `.council/runtime/` (JSON); curated `docs/decisions/` stays the
  committed source-of-truth.
- No mobile/voice/personalization/hosted; no new MCP surface or CLI namespace beyond what the slice
  requires; deterministic, local-first, stdlib-only where possible.

## 16. Deferred / open questions

- Structured store format (JSON vs. later SQLite) — start JSON; revisit only if needed.
- Panel technology (HTML+JS vs. TUI) — pick the smallest that demos well.
- Latency/time-to-demo budgets — set concrete numbers when the orchestrator lands.
- Whether the multi-model council belongs in the v0.5 slice at all, or strictly v0.6.
- License/provenance "Question 0" remains the gate before any commercial/hosted step.
