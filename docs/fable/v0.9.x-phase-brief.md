## Chosen theme: "Apply the proven; describe the personas; defer their behavior"

**v0.9.x finally *applies* the v0.8.2 preference substrate — but only the four mechanically-proven,
add-friction-only keys — and *describes* the council personas as documentation. It does not turn
personas into behavior.** The whole v0.8.x line was built on one ordering rule ("model-and-validate
first, apply later"); v0.9.x is the "later," and the council's clear ruling is to **apply the proven
object and hold persona *behavior* for its own deeply-reviewed line (v0.10.x).** Concretely:

- **Headline — apply the four tighten-only keys (v0.9.0).** The already-validated v0.8.2 keys finally
  influence behavior, **additively and reversibly**, with the deterministic trust boundary still
  authoritative: `default_review_preset` *suggests* a review floor; `require_usage_flag` *warns* when
  `--usage` is absent; `extra_sensitive_paths` / `never_stage_extra` extend the **advisory** doctor
  dangerous-staged surface. Every effect can only *add* friction/warnings.
- **Supporting — persona vocabulary as pure documentation (v0.9.1).** The seven council-review lenses
  (Cost Skeptic, Security Guardian, Product Strategist, Local-first Guardian, UX/User Advocate, Risk
  Officer, Commercialization Lens) are written up as **prose descriptions** — a shared vocabulary a human
  reviewer can invoke mentally — **not** schema objects, **not** a validator, **not** behavior. Plus a
  v0.9.0 dogfood/hardening pass.
- **v0.9.2 — release prep** (house set).

## Why this theme won

- **The council explicitly rejected shipping persona *behavior* (prompt influence) in v0.9.x** as
  premature: the **dissent-suppression / viewpoint-bias** failure mode is a genuinely *new class* that
  the existing "tighten-only" proof machinery **does not cover** — *"more scrutiny is not the same as
  more constraints."* Applying persona emphasis needs a formal safety definition, content-level
  safeguards, and adversarial tests that do not yet exist. That is a whole line, not a slice.
- **Applying the four proven keys is the highest risk-adjusted value:** the substrate, per-type
  tighten-only proofs, and validator already exist; the blast radius is small and auditable; it delivers
  the "personalization is real now" story with the *lowest-risk* object; and the guard still decides
  (preferences only *add* friction).
- **Describing personas as documentation now** gives the product its persona vocabulary without
  committing a schema/abstraction before real usage data from the four keys exists (the council warned a
  premature preset schema risks the wrong abstraction).
- **It preserves local-first credibility** — zero new trust/network/store/UI surface — the product's
  spine.
- **It sets up v0.10.x cleanly:** persona *behavior* becomes its own council+Fable line with the
  dissent-preservation framework as an explicit, pre-designed prerequisite.

## Alternatives rejected or deferred

- **C — Persona prompt-emphasis application: DEFERRED to v0.10.x** behind the hard prerequisite of a
  formal dissent-preservation + content-safeguard framework (additive collect-stage-only, ranking/
  chairman persona-blind, disallowed-pattern content rules, adversarial merge-gate tests, override
  precedence, observability, rollback). Not a v0.9.x slice.
- **B (personas *as schema* + validator): DEFERRED to v0.10.x.** In v0.9.x personas are **prose only**;
  formal preset schema objects wait for usage data from the four keys and for the persona-behavior design.
- **B′ — one hardcoded `--persona` code variant: NOTED for Fable, not committed.** A single code-level
  Security-Guardian collect-stage variant to *learn* before generalizing — recorded as a v0.10.x option,
  not a v0.9.x commitment; it is still persona *behavior* and carries the same dissent risk.
- **D — Session / decision-workspace substrate: DEFERRED** (separate track; risks pack-budget/store
  discipline; orthogonal to this theme).
- **E — `.council/profile.*` local store: DEFERRED** (new leak surface; the committed block covers the
  need; the redaction rule promotes WARNING→CRITICAL only when a store ships).
- **F — UI / dashboard persona controls: DEFERRED** (no applied behavior to control; the only local UI is
  trust-adjacent and frozen).
- **G — Workbench / approval UX: DEFERRED** (orthogonal; trust-adjacent; its own gated slice if pulled in).

## Committed scope (council-informed; Fable finalizes numbering)

- **v0.9.0 — Apply the four tighten-only keys (headline).** Wire the *validated* preferences into
  behavior, **additively and reversibly**, guard-authoritative:
  - `default_review_preset` → `review`/`diff` **suggest** this preset as the default when the user does
    not pass `--preset`; it can never *lower* an explicit choice and never selects `premium`.
  - `require_usage_flag: true` → commands **warn** (never fail) when `--usage` is absent.
  - `extra_sensitive_paths` / `never_stage_extra` → the doctor's **advisory** dangerous-staged / sensitive
    surface also flags these (advisory; never flips READY→NOT-READY on their own beyond the existing
    dangerous-staged rules).
  - **Mandatory guardrails:** per-invocation overrides (`--no-preferences`, and a way to override the
    suggested preset explicitly); **CLI flags always win**; preferences emit **suggestions/findings,
    never decisions**; **"suggest" can never become "enforce" without a major version + security review**;
    the deterministic trust boundary and executor **ignore preferences entirely** (unchanged).
- **v0.9.1 — Persona vocabulary (pure documentation) + v0.10.x design sketch + v0.9.0 dogfood/hardening.**
  A prose write-up of the review lenses in a **strictly bounded format** (a short paragraph + a few
  emphasis bullets per lens), labelled **"v0.9.1 documentation, subject to change"** — that they are a
  *human reviewer's mental lens*, not a command; an explicit statement that personas are **not applied,
  not schema, not validated** in this line. Plus a **written design sketch** (design only, no code) for
  the **v0.10.x dissent-preservation + content-safeguard framework** so the persona-behavior prerequisite
  is validated as feasible before v0.10.x starts. Plus a dogfood/hardening pass over v0.9.0 (override
  precedence, warning hygiene, edge cases, docs). **No behavior, no schema, no validator.** *(Open Fable
  questions: how many lenses to document — 2–3 experimental vs all seven — and named-personas vs a
  "how to think about reviewing" guide.)*
- **v0.9.2 — Release prep** (house set: version bump, dated CHANGELOG, `docs/releases/v0.9.x.md`, README,
  vault; cheap review; no tag/Release in the PR).

Exact numbering/bundling is Fable's call; this is the shape, not a contract.

## Safety invariants (bind every v0.9.x PR)

All of [`03-security-invariants.md`](03-security-invariants.md) hold unchanged. Specifically:

- **Preferences may only *add* friction/warnings; they can never loosen a safety/security/no-stage/trust
  rule.** The deterministic trust boundary and executor **ignore preferences entirely** — a preference
  is advice that adds friction, never authority that removes it.
- **CLI wins; suggest ≠ enforce.** An explicit CLI flag always overrides a preference; a preference is a
  *suggestion/finding*, never a decision; a suggested value can never be *enforced* (no auto-`--allow-premium`,
  no silently-raised gate that blocks) without a major version + security review.
- **Reversible & observable.** A `--no-preferences` escape hatch exists; when a preference changes a
  suggested default, that is stated in output (no silent behavior change).
- **Personas are documentation only in v0.9.x** — no persona is parsed, selected, or applied; nothing
  reads a persona into prompt construction, ranking, or synthesis.
- **Guide / context-export stay pointer-only** — path/reference only, never inlining preference/persona
  content, never reading `.council/profile.*`.
- **No `.council/profile.*` local store**, no preference/persona parser beyond the existing read-only
  validator, no UI, no session/workspace store — unless a later, separately-approved design ships them.
- **Approval stays separate from execution; no auto-execution; localhost-only; no arbitrary shell; no
  command-allowlist expansion; no CORS/LAN/mobile surface; no new runtime dependency** (stdlib-only
  default).

## Non-goals (v0.9.x)

No persona **behavior/application** of any kind (prompt emphasis, ranking, synthesis, generation — all
v0.10.x); no persona **schema objects** or persona validator (documentation only); no `.council/profile.*`
**local store**; no session/decision-**workspace** store; no UI/dashboard; no Workbench trust-boundary
loosening or allowlist growth; no arbitrary shell; no auto-execution; no hosted/SaaS; no vector DB; no
PostgreSQL/Redis/Docker/Next.js/LangGraph; no secrets storage; no dissent-suppression mechanism of any
kind; no `/council` real command; no model/provider/API calls introduced by planning; **no new runtime
dependency** (stdlib-only default — any exception needs a stated reason, security review, and pinning).

## Stop conditions (halt and re-scope)

- A preference starts to **enforce** rather than **suggest/warn** (e.g. blocks a command, auto-enables
  `premium`, or flips READY→NOT-READY on its own) — that is enforcement, not tighten-only application.
- Any persona **behavior** appears (prompt emphasis, ranking/synthesis influence, a `--persona` flag,
  persona schema objects, a persona validator) — that is the deferred v0.10.x line.
- The trust boundary/executor begins to **read** a preference (it must ignore them entirely).
- A `.council/profile.*` store, a session/workspace store, a UI, mobile/LAN/voice, or any network
  surface appears without a separately-approved design.
- Guide/context-export starts inlining preference/persona content (pointer-only is the rule).
- A new runtime dependency, a version bump implied without instruction, or any private artifact staged.

## Override precedence (v0.9.0 — normative)

The single most important UX/safety detail the council flagged. For every command that consults a
preference, precedence is **fixed and testable**, highest wins:

| Priority | Source | Effect |
|---|---|---|
| 1 (highest) | **Explicit CLI flag** (`--preset`, `--usage`, `--no-preferences`) | always wins; a preference can never override it |
| 2 | **`--no-preferences`** escape hatch | ignore the block entirely for this invocation (behavior identical to no block) |
| 3 | **Validated preference block** in `PREFERENCES.md` | *suggests* a default / *adds* a warning — never a decision, never enforcement |
| 4 (lowest) | **Project baseline / today's default** | unchanged from v0.8.2 |

A preference only ever moves behavior **from priority 4 toward more friction**, and only when no
higher priority speaks. It can never select `premium`, never lower an explicit choice, never block.

## Warning hygiene (v0.9.0 — normative)

`require_usage_flag` (and any advisory the keys add) must be **non-annoying**: a warning is emitted **at
most once per command invocation** (not per stage/model), is **actionable** (says exactly what to pass),
is **deduplicated**, and **never blocks** or changes exit codes. Whether `require_usage_flag` ships in
v0.9.0 at all, or is deferred until its warning UX is designed, is an **open question for Fable** (the
council offered "cut it or design the throttling" — either is acceptable; do not ship an every-call
nag).

## Backward-compatibility & output stability (v0.9.0 — normative)

- **No block / `--no-preferences` → byte-identical behavior to v0.8.2.** Preferences take effect only
  when a **valid** block is present; a missing/invalid block changes nothing (the validator already
  fails closed).
- **stdout stays machine-clean.** Any preference-driven suggestion/warning goes to **stderr** (house
  rule), so automation parsing stdout is unaffected; the suggested-default change is **stated** in
  output (no silent behavior change).
- A preference **changing a suggested default is a behavior change for opted-in users only** and is
  documented in the release notes.

## Test coverage required (v0.9.0 — merge gate)

Each of the four keys ships with unit + integration tests proving: **CLI-wins** (explicit flag overrides
the preference), **suggest≠enforce** (never blocks / never auto-`premium` / never flips READY→NOT-READY
on its own), **guard-ignores-preferences** (the deterministic trust boundary/executor behavior is
byte-identical with and without a block), and **reversible** (`--no-preferences` reproduces v0.8.2
behavior). Plus **adversarial-config** tests (loosening-shaped values stay rejected/inert) and a
**regression test per stop condition** below.
