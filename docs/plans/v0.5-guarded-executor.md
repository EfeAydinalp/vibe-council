# Plan: v0.5 — guarded executor (design + safety)

A **planning / safety document**, not an implementation. The Workbench see→decide loop is now
visible and **non-executing** (runtime store → orchestrator → deterministic trust boundary → advisory
Approval Auditor → localhost panel). The next major step is **guarded execution** — the first layer
that could actually modify files or run commands. Because it crosses a real risk boundary, it gets a
design/safety plan **before** any code. **No executor, CLI, panel, or action execution is added in
this task.**

Review date: **2026-07-01**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local.

> **Scope guard.** This PR is docs-only. The executor is deliberately **tiny and likely dry-run-first**;
> anything broader than bounded file write/edit (or exact allowlisted commands) is out of scope and a
> **stop condition** (see §13). License/provenance remains the unresolved **"Question 0"** gate.

---

## 1. Purpose

The guarded executor may eventually run an **approved** action, but **only** when every safety
condition holds: the user approved it, the **deterministic trust boundary** allows it, the action
kind is supported, its scope is narrow, it is logged, and blocked/high-risk cases can never bypass
the guard. Execution is a separate, explicit step from approval — approving records intent; executing
is a distinct, re-checked action.

## 2. Non-goals (out of scope for the first executor)

Arbitrary shell · arbitrary file writes · deploys · `git push`/any git mutation · package install ·
network calls · provider/model calls · cloud sync · LAN/mobile/remote execution · plugin system ·
autonomous loops · background workers/daemons · self-modifying policy. None of these are in the first
executor; several are permanent stop conditions (§13).

## 3. Initial supported actions (tiny first set)

1. **`write_file`** — only within allowed project paths; new file, or overwrite **only** with explicit
   approval of that overwrite; blocked for denied/private/secret paths; a **max size** limit.
2. **`edit_file`** — a simple bounded text replacement / small patch; **max files changed** and **max
   lines changed** (from the trust policy); blocked for denied/private/secret paths.
3. **`run_command`** — only **exact allowlisted** commands from the trust policy (read/test/lint
   first); **no shell metacharacters**; no arbitrary args unless explicitly modeled.

**Recommendation:** defer `run_command` execution to a later executor PR (PR #75). File write/edit is
a smaller, more bounded risk surface; prove that (dry-run first) before executing any command.

## 4. Execution invariant (all must hold, checked at execution time)

The executor must **refuse to run** unless **every** condition is true:

- the approval exists and its status is **`approved`**;
- the action exists, is **`pending`**, and is linked to that approval/task;
- a **fresh** deterministic trust evaluation is **not blocked** and still passes;
- the action **kind is supported** (§3);
- the action **target/scope matches the approved scope** (no substitution after approval);
- an **audit/log record** is created for the attempt.

If any check fails: the action becomes **`blocked`** (or stays `pending`), the **task is not silently
advanced**, the reason is **visible** to the panel/status, and there is **no partial side effect**
(write atomically / temp-then-replace; refuse rather than half-apply).

## 5. Re-check at execution time (advisory ≠ authorization)

The deterministic trust evaluation is **re-run immediately before execution** against the *current*
action/target/scope. An old `AuditResult` is **advisory only** and **cannot authorize execution** —
it is a human-facing summary, not a gate. If the fresh evaluation disagrees with the stored audit
(e.g. the file set changed), the stricter/blocked result wins.

## 6. Dry-run first

The **first executor implementation is dry-run only** (`dry_run=True` default): it previews what
*would* be written/run (paths, diff summary, command) and produces **no side effects**. Tests must
prove dry-run mutates nothing. Real writes land in a later PR (PR #74) behind the same invariant,
still opt-in and still re-checked. This lets the invariant and logging be proven before anything can
change the filesystem.

## 7. Logging

Every execution attempt creates/updates runtime records (under gitignored `.council/runtime/`):

- action `status` (`pending`→`running`→`completed`/`failed`/`blocked`);
- `started_at` / `completed_at`;
- `result_summary` (short, public-safe);
- a **trust evaluation summary** (risk + verdict);
- an **executor version**;
- an **error/blocked reason** when applicable.

**Never write raw secrets into logs** — reuse the redaction guard; log paths/decisions, not file
contents or credentials.

## 8. Panel behavior

The panel shows: the pending action, its trust summary, its audit summary, the execution result, and
any blocked/failed reason. **The panel must NOT auto-execute on approve.** Approval and execution are
**separate, explicit** steps:

1. **Approve** (records the decision — today's behavior, non-executing);
2. **Execute approved action** (a distinct button/step, dry-run first).

This separation is the primary guard against accidental execution.

## 9. CLI behavior

A future, explicit command is possible — e.g. `vibe workbench execute <action-id>` (dry-run by
default; `--apply` to actually run once real execution lands). **Recommendation:** for the first
executor, prefer a **CLI `execute --dry-run`** over a panel button — it is easier to gate, log, and
test, and keeps the panel non-executing a little longer. Not implemented now.

## 10. Security cases (must all be covered by tests)

blocked unknown action · blocked secret/private path · blocked out-of-project path · blocked shell
metacharacters · non-allowlisted command · approval/action mismatch (action not linked to an approved
approval) · stale approval (status not `approved`) · action modified after approval (target/scope
changed) · denied-path symlink (resolve before executing; refuse if it escapes) · path traversal ·
oversized write/edit (over size/line/file limits) · concurrent execution attempt · repeated execution
attempt (already `completed`/`running`) · log redaction (no secrets in logs).

## 12. Tests required before any executor release

- Dry-run performs **no mutation** (filesystem + runtime tree unchanged).
- Executor refuses when: no approval / approval not `approved` / action not `pending` / action not
  linked / kind unsupported / trust blocked / target-or-scope mismatch.
- Fresh trust re-check runs at execution time and a stored (stale) audit **cannot** authorize.
- Each §10 security case is blocked (unknown/secret/out-of-project/traversal/symlink/metachar/
  non-allowlisted/oversized).
- Real write/edit (PR #74) is atomic (no partial write on failure), respects size/line/file limits,
  and only touches allowed paths.
- Logs contain status/timestamps/summary/version and **no secrets** (redaction verified).
- Repeated/concurrent execution attempts don't double-apply or corrupt state.
- No provider/model/network/git/shell (until §3 `run_command` PR, which is `shell=False` + allowlist).
- All tests use temp dirs; no real-repo `.council/runtime/` artifacts.

## 13. Stop conditions (halt and ask for review / a council pass)

Stop implementation and request review if any of these arises:

- scope grows **broader than bounded file write/edit** (or exact allowlisted commands);
- **arbitrary command execution** is requested;
- **any cloud/network execution**;
- a **blocked** action appears executable, or approvals can be **bypassed**;
- **path/symlink uncertainty** (can't prove containment);
- **concurrency/race** risk that could double-apply;
- pressure to let the **advisory Auditor authorize** execution.

## 14. Acceptance criteria for the executor track

- Approval and execution are **separate**; the panel never auto-executes on approve.
- The deterministic trust boundary is **re-run at execution time**; the advisory Auditor never gates.
- First executor is **dry-run first**, tiny, fully invariant-tested, and mutates nothing until PR #74.
- Every attempt is logged (no secrets); blocked/failed reasons are visible; no partial side effects.
- Deterministic, local-first, stdlib-only where possible; no new provider/network surface.
