# Plan: v0.5 — Workbench release readiness & dogfood checklist

A **readiness/checklist document**, not a new feature. PR #72–#81 completed the guarded-executor
track: the AI Council Workbench can now propose, be approved, and be **explicitly, guardedly
executed** — for bounded file changes and for a small set of allowlisted commands. This document
records what v0.5 now contains, what it deliberately does not, how to dogfood it locally, and what
must hold before it ships as a tagged release. **No executor/panel/CLI behavior changes in this PR.**

Review date: **2026-07-02**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local.

This is a **track readiness check**, not the release-cut process — see
[`docs/release-checklist.md`](../release-checklist.md) for the generic version-bump/tag/CHANGELOG
runbook used for every past release (v0.1.0/v0.3.0/v0.3.1/v0.4.0). v0.5 is **not tagged by this PR**
(see §8): this document establishes that the track is *functionally* ready, not that today is
release day.

---

## 1. What v0.5 now contains

- **Runtime store** (`backend/workbench_runtime.py`) — `Task`/`Stage`/`ApprovalRequest`/
  `ApprovalDecision`/`Action`/`AuditResult` in a gitignored `.council/runtime/` JSON tree.
- **Orchestrator / state machine** (`backend/workbench_orchestrator.py`) — deterministic task
  lifecycle (start → plan → request approval → approve/reject/hold → executing → complete/fail/hold).
- **Deterministic trust boundary** (`backend/workbench_trust.py`) — the real security gate: path
  allow/deny, command allowlist, secret patterns, change-size limits, cloud-egress consent. Unknown
  kinds and non-allowlisted/metachar commands are blocked by default.
- **Advisory Approval Auditor** (`backend/workbench_auditor.py`) — wraps the trust evaluation into a
  human-readable approval prompt; copies risk/blocked/findings verbatim, so it can never relax the
  guard.
- **Localhost Workbench panel** (`backend/workbench_panel.py`, `vibe workbench serve`) — task
  progress, pending-approval cards, approve/reject/hold, and demo/dogfood UX (a **"Create demo task"**
  button that seeds one safe local approval — see §3 for exactly what it does and doesn't do).
- **Payload artifacts + hash/scope verification** (`backend/workbench_payloads.py`) — local, gitignored,
  write-once `.council/runtime/payloads/<action_id>.json`, hashed at creation and re-verified
  (hash + kind/target/approval/task agreement) before real execution.
- **Bounded file execution** (`backend/workbench_executor.py`) — real `write_file`/`edit_file`,
  atomic writes, size/line-delta limits, path/symlink guard, existing-file requires explicit
  overwrite, logs carry no content.
- **Exact allowlisted command execution** — a fixed label→argv resolver
  (`backend/workbench_commands.py`, no shell, no string parsing, `sys.executable`-based) plus real
  execution (`subprocess.run(shell=False)`, sanitized environment, project-root cwd, timeout, bounded/
  redaction-checked output capture).
- **Panel execute + result display** — a separate, explicit "Execute" step (with a confirm-dialog UX
  nudge, not a security boundary) for both bounded file actions and allowlisted commands; the browser
  only ever sends an action id.

## 2. What v0.5 explicitly does not contain

Arbitrary shell · dynamic command arguments · package install/deploy/`git push` (or any git mutation)
· provider/model/API calls from the executor · LAN/mobile/remote access · voice · background/
autonomous loops · a plugin system · multi-user/team/hosted mode · cloud sync. None of these are
partially built or "almost there" — they are out of scope by design (see the guarded-executor and
command-execution plans' stop conditions), and adding any of them is a **stop condition**, not a v0.5
task.

## 3. How to dogfood locally

A manual operator recipe — no code changes, run this against a scratch/local clone, not blindly
against unrelated production data:

1. **Sync and verify:**
   ```sh
   git pull
   .venv\Scripts\python.exe -m unittest discover -s tests -t .     # Windows
   .venv/bin/python -m unittest discover -s tests -t .             # macOS/Linux
   ```
2. **Start the panel:** `python -m backend.cli workbench serve` (or the global `vibe workbench serve`
   launcher). Note the printed URL and token.
3. **Open the URL** in a browser (`http://127.0.0.1:<port>/?token=<token>`).
4. **Click "Create demo task"** — this seeds one task + one pending approval (a proposed
   `write_file:docs/example.md`) with a saved advisory audit. **It creates no `Action` and no payload
   artifact** (a deliberate PR #77 decision: the panel's `project_root` is normally your real project,
   and there is no path that is both a real, trust-boundary-writable target *and* guaranteed harmless
   to a reflex demo click). **The demo does not let you execute anything** — after approving it, no
   "Execute" button appears, because there is no linked `Action`. This is intentional, not a bug.
5. **Approve / reject / hold** the demo approval and confirm the card updates and the decision is
   recorded (`approved`/`rejected`/`held`) — this proves the decision path, not execution.
6. **To actually exercise real execution** (bounded file write/edit, or an allowlisted command), you
   need to manually seed a `Task`/`ApprovalRequest` (approved) + linked `Action` (+ a payload artifact
   for file kinds) against a **throwaway temp project directory** — exactly what
   `tests/test_workbench_panel.py`'s `_seed_action` helper does. There is currently no panel/CLI
   affordance that authors a real proposal end-to-end (that's future work, e.g. wiring a council
   review/diff stage — see `docs/plans/v0.5-workbench-mvp.md` §14 item 8). Until then, dogfooding real
   execution means either running the test suite (which already proves this path extensively) or
   scripting a seed against a scratch directory, **never your real repo's project root**, unless you
   have specifically reviewed what the seeded action will do.
7. **Stop the server** with `Ctrl+C`.

## 4. Manual smoke checklist

- [ ] Panel binds to `127.0.0.1` (visible in the printed URL; refuses `0.0.0.0`/other hosts —
      `make_server` raises `ValueError` otherwise).
- [ ] A startup token is printed (unless `--no-token`), and `GET /`/`POST` without it are rejected
      (`403`) — POSTs require `X-Workbench-Token` or `?token=`.
- [ ] The empty state ("No tasks yet.") is self-explanatory and offers the demo button.
- [ ] "Create demo task" seeds one task + one pending approval; the card shows a risk label and a
      human-readable rewritten prompt.
- [ ] Approve/reject/hold updates the approval status and the task's stage; **no action ever runs**
      from a decision alone.
- [ ] A **manually seeded** (temp/scratch project only) approved `write_file`/`edit_file` action with a
      valid payload artifact shows "Execute approved file action" and, on click + confirm, actually
      writes the file and marks the action `completed`.
- [ ] A **manually seeded** approved `run_command` action for an exact allowlisted label (e.g.
      `git status --short`) shows "Execute approved command" and, on click + confirm, actually runs and
      reports exit code/output summary.
- [ ] A **non-allowlisted** command (or a mutated/extra-arg variant of an allowlisted one) shows **no**
      execute button and is blocked if triggered directly.
- [ ] Posting a crafted request body with different command/argv/content to `/api/actions/<id>/execute`
      has **no effect** — only the server-side resolved action/payload/argv is ever used.
- [ ] After exercising the above, `git status --short` in the repo shows **no** `.council/` entries
      staged (it's gitignored; nothing here should ever appear as a stageable change).

## 5. Security checklist

- [ ] Approval and execution are **distinct** actions — approving never triggers execution.
- [ ] Execution is always an explicit, separate call (`execute_action(..., dry_run=False)` /
      `POST /api/actions/<id>/execute`), never implicit.
- [ ] The deterministic trust boundary **re-runs at execution time** — a stored/stale `AuditResult`
      cannot authorize anything.
- [ ] The advisory Approval Auditor **cannot** relax or bypass a blocked/high-risk verdict (risk/
      blocked/findings are copied verbatim from the deterministic evaluation).
- [ ] The browser sends **only an action id** (+ token) to the execute endpoint — never file content,
      patch text, a command string, argv, cwd, env, or a timeout.
- [ ] Payload artifacts are **local and gitignored** (`.council/runtime/payloads/`), write-once, and
      hash/scope-verified before use.
- [ ] Command labels resolve to a **fixed, pre-built argv** — never a parsed/derived string.
- [ ] Real command execution always uses `shell=False`.
- [ ] The subprocess environment is **allowlist-built** (no inherited secrets/API keys/`.env`).
- [ ] A timeout and an output-byte cap are enforced on every real command execution; a critical
      redaction finding in output blocks the result instead of storing it.
- [ ] The panel binds to `127.0.0.1` only; POSTs are token-gated; there is no CORS wildcard.

## 6. Release blocker checklist

Any of the following blocks a v0.5.0 tag until resolved:

- [ ] Any test failing (`python -m unittest discover -s tests -t .` not green).
- [ ] Any **critical** redaction finding (`vibe lint --redaction`).
- [ ] `vibe context check` scoring below **21/21**.
- [ ] `vibe mcp inspect --context --health` scoring below **21/21**.
- [ ] Any `.council/` artifact (including `.council/runtime/payloads/`) staged/committed.
- [ ] Either private/local plan file (`commercialization-and-hosted-platform-feasibility.md`,
      `v0.3.1-hardening-and-dogfood.md`) staged/committed.
- [ ] An accidental `uv.lock` change unrelated to an intentional dependency bump.
- [ ] Any path to arbitrary command execution (shell, dynamic args, unreviewed allowlist additions).
- [ ] Browser-supplied payload/command content being accepted anywhere in the execute path.
- [ ] The panel auto-executing on approval (approve and execute must stay separate).
- [ ] Any way for the advisory Auditor, a cached preview, or a stale audit to bypass the deterministic
      trust boundary at execution time.

## 7. Known limitations

- The panel is **intentionally simple** — no framework, no rich UI, no persistence beyond the runtime
  JSON store. This is deliberate, not a placeholder for something bigger arriving soon.
- The demo **does not** seed an executable file or command action (§3) — don't imply otherwise in
  external messaging or onboarding docs.
- The command allowlist is **intentionally tiny** (7 entries: a handful of verification/status
  commands) — it is not meant to grow casually; each addition is its own reviewed decision.
- There is **no remote/mobile/LAN access** — the panel is `127.0.0.1`-only by design, not by omission.
- **The context pack is close to its budget.** PR #81 still scored 21/21, but only after the fallback
  chain reached "capped decision index entries" — one step further into degradation than earlier PRs
  in this same track. **Future decision records should stay concise** (roughly the 4–5 KB range recent
  Workbench records have used, not longer) to avoid tipping required advisory facts out of the pack;
  see the existing budget guidance in
  [`context-pack-budget-headroom`](../decisions/2026-07-01-context-pack-budget-headroom.md) and
  [`context-pack-critical-section-budget`](../decisions/2026-07-01-context-pack-critical-section-budget.md).
  Several PR #76/#79/#80/#81 decision records were deliberately trimmed for exactly this reason.
- **Windows/Linux differences should be exercised before a broad release** — the command executor's
  `sys.executable`-based resolution and environment sanitization (`SystemRoot`/`SystemDrive` on
  Windows only) were designed for cross-platform parity but have primarily been exercised on Windows
  in this development environment; a CI-green run on the Ubuntu/macOS/Windows matrix (per
  `docs/release-checklist.md` §2) should be confirmed before tagging.

## 8. Recommended next steps after this PR

Do **not** jump into hosted/team/cloud work. In order:

1. **v0.5.0 release notes** (`docs/releases/v0.5.0.md`, following the existing v0.4.0 pattern) once the
   §6 blocker checklist is clean and the CI matrix is green.
2. **Dogfood on a real small repo** (not this one) to surface rough edges the manual smoke checklist
   might miss.
3. **Fix rough edges** found during dogfooding — likely a small, focused v0.5.x patch, not new
   features.
4. **Maybe a v0.5.1 hardening pass**, mirroring the v0.3.1 dogfood-hardening precedent.
5. **Only then** consider LAN/mobile access or a richer operator workflow (already deferred to v0.6+
   in `docs/plans/v0.5-workbench-mvp.md` §13).
