# v0.10.x dissent-preservation framework — design sketch (design only)

> **v0.10.x design sketch, subject to change — design only, no implementation, nothing here is
> built, wired, parsed, selected, or applied.** This document outlines the safety framework that
> would have to exist *before* any council **persona / review-lens behavior** (prompt emphasis, a
> `--persona`/`--lens` flag, persona preset objects) could ship. **Persona/lens behavior is
> not implemented yet** and stays deferred to v0.10.x or later. No production code changes because this
> file exists. It changes no prompt construction, peer ranking, chairman synthesis, model/provider
> selection, the preference reader/validator or `PREFERENCES.md` machine-block semantics, the
> guide/context-export output, or the Workbench/importer/executor/guard/trust boundary. It adds no
> schema field and implements no `schema: 2`. It is the *design input* to a future council + Fable
> line, not a commitment to build.

This sketch is the companion to [`council-review-lenses.md`](council-review-lenses.md): that file
names the lenses as documentation-only vocabulary; this file specifies what a future *behavioral*
version of them must satisfy. It exists because the v0.9.x council debate
([`v0.9.x-council-debate.md`](v0.9.x-council-debate.md)) found that **dissent suppression / viewpoint
bias is a new risk class** the existing tighten-only proof machinery does not cover — and the
v0.9.x architecture plan ([`v0.9.x-architecture-plan.md`](v0.9.x-architecture-plan.md) §12 Q7) made
this sketch, with an honest feasibility verdict, PR 7 of v0.9.1.

## 0. Status and scope

- **Design only. No implementation.** No file under `backend/` gains a persona/lens code path from

- **Out of scope of this sketch:** any actual persona behavior, any `schema: 2`, any preference

## 1. Purpose and the principle it protects

The council's value is **independent, multi-model second opinion** — several seats reason
separately, minority findings surface, and a chairman synthesizes without erasing disagreement.
A persona/lens is a *reviewer's viewpoint* ("read this the way a Security Guardian would"). The
hazard is that turning a viewpoint into applied behavior can, even with the best intent, **quiet the
very disagreement the council exists to preserve.** The single non-negotiable principle this
framework protects:

> **Dissent preservation:** a lens may only *add* scrutiny, questions, or emphasis. It may **never
> hide, downrank, outrank, filter, reorder, reweight, or override** another lens, another seat,
> another reviewer, or the council's independent second opinion. Every viewpoint stays visible; a
> **minority opinion** is never removed; a dissenting finding is never silenced because a lens was in
> use.

Everything below is machinery to make that principle *inspectable and testable* rather than a
promise.

## 2. Threat model (for future persona / review-lens behavior)

The failure modes a v0.10.x persona-behavior PR must defend against — none of which the four proven
tighten-only preference keys can produce, because those keys only add friction/warnings and never
touch prompt text:

- **Reframing bias.** A persona prompt re-frames the question ("focus on attack surface") so that a
  seat spends its attention on one axis and *never raises* an objection it otherwise would have. No
  ranking or synthesis was touched, yet effective scrutiny narrowed. This is the subtle core risk.
- **Emphasis-as-suppression.** "Emphasize X" read by a model as "deprioritize everything else."
  Additive-sounding text produces subtractive attention.
- **Seat crowd-out.** Applying the same persona to multiple seats collapses viewpoint diversity —
  several seats now think alike, so a lone dissent has fewer places to originate.
- **Cumulative bias across stages.** Persona text that leaks past the collect stage into ranking or
  synthesis inputs compounds: a biased frame plus a biased ranking silently double-counts.
- **Persona-text-as-injection.** Persona prose is *untrusted input*. Text like "ignore the safety
  rubric" or "rank the other seats last" must be structurally impossible to honor.
- **Silent behavior drift.** A persona that changes output without saying it was active makes the
  bias unobservable and un-auditable.

### 2.1 How a lens can accidentally *suppress dissent*

Suppression here is rarely explicit ("delete objection Y"). It is **emergent**:

1. A persona narrows the frame → a seat never *forms* the dissenting thought (pre-emptive
   suppression at the collect stage — the hardest to detect because nothing is deleted; a thought
   simply never appears).
2. Persona text bleeds into the **ranking** stage → the dissenting seat is scored lower.
3. Persona text bleeds into **synthesis** → the chairman's summary drops or softens the minority
   view.
4. The persona is applied to **every** seat → homogenized reasoning, so the objection has no
   independent origin.

The defense is structural (§3) — keep persona text *additive*, *own-seat-only*, and *out of ranking
and synthesis entirely* — plus adversarial tests (§5) that prove a seeded objection still surfaces.

### 2.2 Why "more scrutiny" is **not** automatically "more safety"

The tempting fallacy is that a lens can only help because "it just asks for more care." The council's
central correction: **"more scrutiny" is not the same as "more constraints," and it is not the same
as "more safety."** Attention is finite. Directing it *toward* one axis directs it *away* from
others; a security-flavored review can miss a cost or UX regression precisely because it was told to
look at security. Adding a viewpoint can therefore **reduce** total coverage even though every word
of the persona reads as "be more careful." This is why persona emphasis does **not** inherit the
mechanical add-friction proof the four preference keys enjoy, and why it needs content-level
safeguards and adversarial tests, treated like a threat-model change rather than a copy edit.

## 3. Structural rules (for future persona behavior)

Binding on any v0.10.x persona-behavior design. These are architectural constraints, not style
guidance:

- **Additive, collect-stage-only.** A persona may only **append** advisory emphasis to the *collect*
  prompt of its **own seat**. It may never remove, reorder, reweight, or replace the base rubric.
- **Own-seat-only.** A persona influences exactly one seat's collect prompt. It cannot reach another
  seat's prompt, output, or score.
- **Ranking is persona-blind.** The peer-ranking stage receives **no** persona text and has no
  knowledge of which persona (if any) shaped a seat. Ranking inputs are byte-identical whether a
  persona was active or not.
- **Synthesis is persona-blind.** The chairman-synthesis stage receives **no** persona text in its
  inputs. The synthesis must still represent every seat's findings, including minority ones.
- **No hidden prompt mutation.** Persona text is appended visibly and labeled; there is no **silent**
  edit of a base prompt, and no persona-driven change to ranking or synthesis of any kind, hidden or
  otherwise.
- **Default-off, feature-flagged.** Persona behavior is disabled unless explicitly turned on for that
  invocation; the default council run is byte-identical to today.
- **Trust boundary untouched.** The deterministic trust boundary, the guarded executor, the command
  allowlist, the Workbench guard, and approval-separate-from-execution are **completely
  persona-blind** and unchanged. A persona can never loosen a gate, lower a review level, authorize
  `premium`/`--allow-premium`, skip a guard, or reach the executor/trust code.

## 4. Content rules (for future persona behavior)

Persona prose is untrusted input and is validated against disallowed patterns before it can be used:

- **May not reference other personas, seats, ranking, or synthesis.** No "rank the others lower," no
  "you are the most important seat," no "the chairman should prefer you."
- **No suppression / override language.** Disallowed verbs and their paraphrases: *ignore*,
  *deprioritize*, *overrule*, *override*, *outrank*, *downrank*, *hide*, *drop*, *silence*,
  *suppress*, *disregard the rubric*, *skip the guard*.
- **No safety/gate manipulation.** Cannot instruct loosening a safety/security/no-stage/trust rule,
  cannot request `premium`, cannot ask to bypass approval or execution gates.
- **Additive framing only.** Content may say "*also* consider X"; it may not say "consider X
  *instead of* Y."
- **Length-bounded.** Each persona's emphasis text is short and capped, so it cannot dominate or
  drown the base rubric.
- **Curated, not free-form.** Persona text is a curated, reviewed preset (a bundle of the existing
  tighten-only preference values plus bounded advisory emphasis), never arbitrary user/agent-supplied
  prose injected at runtime.

## 5. Dissent-canary test ideas

The adversarial test suite is the merge gate for **any** persona-behavior PR. Shape:

- **Dissent-canary fixtures.** Seed a scenario with a known objection that a specific seat *should*
  raise (e.g. a cost regression hidden in a security-flavored diff).
  Assert that **a seeded canary objection must still surface verbatim** with every persona enabled —
  the objection is never dropped, softened below a threshold, or ranked out of the summary.
- **Minority-survival.** With one seat dissenting from a majority, assert the minority finding
  appears in the collect outputs, is not downranked *because* of a persona, and is represented in the
  synthesis.
- **Injection rejection.** Persona text shaped like an instruction ("ignore the rubric," "rank the
  others last," "skip the guard") is **rejected** by the content validator, never applied.
- **Ranking/synthesis byte-identity.** The ranking-stage and synthesis-stage inputs are
  byte-identical with and without any persona active (proves §3's persona-blind stages).
- **Off-state golden.** With the feature flag off (default), the entire council run — prompts,
  ranking, synthesis, output — is byte-identical to the pre-persona baseline.
- **Combinatorial coverage.** All personas × representative seat/scenario combinations, not just the
  single-persona happy path (seat crowd-out only shows under multi-persona load).

These tests are **necessary but, by construction, not sufficient** — reframing bias (§2.1 case 1)
suppresses a thought that is never written down, so no output-diff test can fully catch it. See the
feasibility verdict (§9).

## 6. Observability / audit requirements

Bias that cannot be seen cannot be reviewed. Any persona behavior must be **inspectable**:

- **Labeled in output.** Every stage output records which persona (if any) shaped which seat. The
  effect is never silent.
- **Labeled in usage reporting.** `--usage` / usage output names the active persona(s) so a
  maintainer can correlate spend and framing with which lens was on.
- **Reproducible.** Given the same inputs + persona selection, the applied persona text is
  deterministic and can be shown verbatim on request (no hidden, dynamically-generated persona
  prose).
- **No new store required to observe.** Observability is via existing stage outputs and stderr
  usage reporting — it does **not** justify a `.council/profile.*` store, a database, or a UI (all
  deferred; see §12).

## 7. Override and rollback rules

- **Explicit user / CLI choice always wins.** A persona is a *suggestion of emphasis*, never a
  decision. An explicit CLI flag (preset, usage, `--no-preferences`, or a future explicit
  persona-off flag) overrides any persona; the same CLI-wins precedence the v0.9.0 preference
  application already follows.
- **Per-invocation override.** A single flag disables persona emphasis for that run
  (e.g. `--no-persona` / reusing `--no-preferences`), producing behavior byte-identical to the
  default off-state.
- **Feature flag default-off.** Persona behavior is globally off unless explicitly enabled; shipping
  it does not change the default council run.
- **Kill-switch / rollback.** A single switch disables all persona behavior everywhere with no data
  migration and no residue — the code path is **reversible / disable-able** by construction, and its
  removal returns the tool to the byte-identical baseline.
- **Suggest never silently becomes enforce.** A persona can never be promoted from advisory emphasis
  to an enforced gate without a major version + full security review.

## 8. Acceptance criteria (before any persona behavior can ship)

A v0.10.x persona-behavior PR may not merge unless **all** of the following hold. These are
prerequisites, not polish:

1. The structural rules (§3) are implemented and enforced in code (collect-stage-only, own-seat-only,
   ranking/synthesis persona-blind, default-off).
2. The content validator (§4) rejects every disallowed pattern, with tests.
3. The dissent-canary suite (§5) passes, including combinatorial and injection-rejection cases, and
   is a **merge gate**.
4. Ranking and synthesis inputs are proven byte-identical with/without personas.
5. The off-state golden is byte-identical to the pre-persona baseline.
6. Observability (§6) is implemented: persona use is labeled in stage output and usage.
7. Override + rollback (§7) exist and are tested: CLI wins, per-invocation off, feature-flag
   default-off, kill-switch reversible.
8. The trust boundary / executor / guard / Workbench remain persona-blind, proven by the existing
   isolation-scan style tests extended to persona code.
9. No `.council/profile.*` store, no UI, no network surface, no new dependency is introduced.
10. Reviewed at **full** council level (it defines applied trust semantics), and approved by the
    maintainer.

## 9. Feasibility verdict for v0.10.x

**Honest verdict: partially feasible, partially a research problem — feasible enough to attempt
behind the framework, but not safe to ship as a broad, on-by-default feature.**

- **Feasible now (engineering):** the *structural* guarantees (collect-stage-only, own-seat-only,
  ranking/synthesis persona-blind, default-off, kill-switch), the *content* validator, and most of
  the *canary* suite are ordinary, testable engineering. These catch explicit suppression, injection,
  and stage bleed.
- **Research-grade residual risk:** **reframing bias** (§2.1 case 1) — a persona quietly narrowing
  attention so a thought never forms — is *not* fully catchable by output-diff tests, because there
  is no deleted artifact to detect. Measuring "did this frame reduce coverage?" is an open evaluation
  problem, not a unit test.
- **Recommended first step (if v0.10.x proceeds):** the most conservative slice — a **single,
  hardcoded, default-off** persona (e.g. Security Guardian) as a collect-stage-only variant, shipped
  *with* the full canary suite, used to gather real evidence before generalizing to a persona
  vocabulary or any preset schema. Do **not** open with all seven personas, a `--persona` free-form
  flag, or a persona schema.
- **Acceptable outcome:** "not yet — the reframing-bias evaluation is unsolved" is a legitimate
  verdict for a future v0.10.x review. This framework does not promise the feature; it defines the
  bar and is honest that one part of the bar is a research question.
- **Cancellation is on the table, not just deferral.** If prototyping and the adversarial suite show
  dissent-preservation is infeasible, persona/lens *behavior* may be **canceled — not merely
  deferred**. Passing these doc tests must never be read as proof the problem is solved; the hardest
  challenge (reframing bias) remains open until measured on real model behavior.

## 10. Stop conditions (halt → full council review)

If any of these appear in a purported "docs" or "small" persona PR, stop and escalate to a **full**
council review (and maintainer sign-off) — they are not incremental changes:

- Persona text reaches the **ranking** or **synthesis** stage, or influences another seat.
- Any **silent** prompt mutation, or any hidden ranking/synthesis change.
- A persona can **suppress, downrank, outrank, hide, drop, reorder, or reweight** another seat's or
  reviewer's output, or remove a **minority opinion**.
- A persona loosens a safety/security/no-stage/trust rule, lowers a review level, authorizes
  `premium`, or the **trust boundary / executor / guard / Workbench** begins to read persona/lens
  state.
- Persona behavior ships **on by default**, or lacks a per-invocation override / kill-switch.
- A `.council/profile.*` store, a UI/dashboard, or any network/LAN/mobile/hosted surface is added to
  "support" personas.
- A `schema: 2` / persona preset schema is introduced without its own reviewed spec release.
- A new runtime dependency is added for personas (stdlib-only default).

## 11. Open research questions

- How do you *measure* reframing bias (reduced coverage from a narrowed frame) without a deleted
  artifact to diff against? Is a differential-coverage eval (same scenario, persona vs. no-persona,
  count distinct findings) enough, and how noisy is it across models?
- Are persona presets, when they arrive, `schema: 1`-additive or a reviewed `schema: 2`? (Deferred to
  the v0.10.x line; **not** decided here.)
- Does applying the *same* persona to multiple seats ever have a legitimate use, or is own-seat-only
  strict enough that multi-seat personas should be forbidden outright?
- What is the right length cap and disallowed-pattern list for persona prose, empirically?
- **Is persona-in-seat the right shape at all?** A **persona-free fallback** — running multiple
  *concurrent councils* with different preference sets and comparing their separate outputs — removes
  the suppression risk entirely (separate outputs cannot suppress each other) at the cost of multiplied
  spend. If the in-seat design cannot clear this bar, this fallback (or transparent per-finding bias
  reporting) is the Plan B; both stay design options here, neither is implemented.

## 12. What this sketch does NOT do (non-goals, restated)

No persona/council behavior application; no prompt emphasis wired anywhere; no ranking/synthesis
influence; no schema extension and no `schema: 2`; no preference validator or reader change; no
review/diff/doctor behavior change; no guide/context-export behavior change; no
Workbench/proposal/importer/panel/executor/trust change; no `.council/profile.*` local store; no
UI/dashboard; no network/LAN/mobile/hosted mode; no arbitrary shell; no auto-execution; no
model/provider/API call; no vector DB/database; no dependency change; no version bump; no tag/release.

## 13. Document status & lifecycle

- **Living design doc, not a contract.** This is a v0.9.1 *sketch*, subject to change. It may be
  revised — or its whole approach **abandoned** — as v0.10.x prototyping produces evidence. Nothing
  here is a commitment to build persona behavior.
- **Amendment requires review.** Because it frames future trust semantics, a material change to this
  framework (weakening a structural/content rule, relaxing an acceptance criterion, or softening a
  stop condition) is a **full council review + maintainer** decision, recorded like a threat-model
  change — not a routine doc edit.
- **v0.10.x is blocked until the bar is met.** No persona/lens *behavior* PR may open until the §5
  adversarial dissent-canary suite exists with quantitative thresholds and the §8 acceptance criteria
  can be satisfied and reviewed at **full** level. Documentation and passing doc tests do **not**
  satisfy this bar.
