## 3. Shutdown investigation

### 3a. What was tested

Two subprocess-based experiments were run, each launching the server, confirming it was serving via
HTTP, then delivering an **actual Windows `CTRL_C_EVENT`** (the real console signal a literal Ctrl+C
keypress generates — via `os.kill(pid, signal.CTRL_C_EVENT)` on a process started with
`CREATE_NEW_PROCESS_GROUP`), not a forced kill:

| Experiment | Launch method | Result |
|---|---|---|
| A | `uv run python -m backend.cli workbench serve` | Server did **not** exit within 8s of the `CTRL_C_EVENT`; still served HTTP afterward. Force-killing the direct `uv` child process **left the actual `python.exe workbench serve` process running** (a real leftover — `uv run`'s spawned child was not part of the killed process's tree in a way `.kill()` reached). |
| B (control, no `uv run`) | `.venv\Scripts\python.exe -m backend.cli workbench serve` directly | Server **also** did not exit within 8s of the `CTRL_C_EVENT`. However, force-killing this process **did** cleanly release the port with no leftover — unlike experiment A. |

### 3b. What this does and doesn't tell us

- **The "leftover process after stop" symptom from PR #86 is explained**: it is specific to `uv run`'s
  process-tree wrapping (killing the `uv` process it spawns does not reliably kill the actual Python
  child `uv` in turn spawns). This is a **harness/`uv run`-process-tree artifact**, not a
  `workbench_panel.py` bug — confirmed by experiment B, where killing the directly-launched process
  (no `uv run` in between) cleanly released the port every time this session.
- **The "`CTRL_C_EVENT` didn't trigger shutdown within 8s" symptom is NOT explained by `uv run`** —
  it reproduced identically with and without the wrapper (experiments A and B). Read in isolation,
  this could suggest a real gap in `workbench_panel.serve()`'s Ctrl+C handling.
- **However, `workbench_panel.serve()`'s shutdown code is the textbook-correct pattern**:
  ```python
  try:
      httpd.serve_forever()
  except KeyboardInterrupt:
      pass
  finally:
      httpd.server_close()
  ```
  This is exactly how `http.server`/`socketserver` document handling Ctrl+C, and it's the same
  pattern used by virtually every stdlib-HTTP-server tutorial. There is no code reason visible in this
  function for a real interactive Ctrl+C to fail.
- **`os.kill(pid, signal.CTRL_C_EVENT)` from a non-interactively-attached sending process is a
  documented Windows fragility.** `GenerateConsoleCtrlEvent` (what this call wraps) depends on
  console attachment semantics that don't always hold when the *sending* process itself was started
  by a non-interactive wrapper (this session's Bash/PowerShell tools are explicitly invoked with
  `-NonInteractive`/non-TTY semantics) — a known class of false negative for this exact simulation
  technique, independent of the target program's own correctness.
- **A real, human-attended interactive terminal test (physically pressing Ctrl+C) was not available
  in this session.** There is no human physically at a keyboard in this fully automated agentic
  session, and this session's terminal tools are explicitly non-interactive wrappers — they cannot
  provide the same console attachment a real opened Terminal/PowerShell window would. Escalating to
  GUI/computer-use automation to open a visible terminal window and simulate a keypress was
  considered and **not** attempted: it adds meaningful complexity and a new permission grant for a
  result that would likely still be an automated *simulation* of a keypress rather than a genuine
  human-in-the-loop test, and would not materially resolve the ambiguity in §3b.

### 3c. Classification

Per this pass's explicit instruction: **do not classify the `CTRL_C_EVENT` simulation result as a
confirmed Workbench bug unless a real manual interactive Ctrl+C also fails.** That manual test was
not available.

**Classification: inconclusive / Windows console-signal simulation issue possible.**

- The leftover-process symptom from PR #86 is **resolved**: classified as a **harness-only artifact**
  of `uv run`'s process tree, not a Workbench bug (§3b, experiment B is the control that shows clean
  shutdown without `uv run` in the chain).
- The "`CTRL_C_EVENT` doesn't trigger shutdown within 8s" symptom remains **inconclusive** — it may be
  a genuine gap, or (more likely, given the code review in §3b) an artifact of simulating Ctrl+C via
  `os.kill` from a non-interactive sending process. **No code change is made in this PR** — the
  evidence does not meet the "confirmed, reproducible" bar this pass was instructed to require before
  touching `workbench_panel.py`.

## 5. Unresolved follow-ups

- **A genuine human-attended interactive terminal test remains the only way to fully close the
  Ctrl+C question.** Recommended: at some point, a maintainer runs `vibe workbench serve` (or
  `uv run python -m backend.cli workbench serve`) in a real, visible terminal window and physically
  presses Ctrl+C once, confirming the prompt returns and the port closes. Until then, treat this as
  **likely fine** (the code follows the standard, correct pattern) but not independently proven from
  this session.
- If a maintainer *does* observe a real hang on physical Ctrl+C, the next step would be adding a
  short `poll_interval` to `serve_forever()` or an explicit signal handler as a small, targeted fix —
  not attempted here since the bug isn't confirmed.
- The `uv run` leftover-child-process pattern (§3b) is worth keeping in mind for any future tooling
  that launches `vibe workbench serve` programmatically (e.g., CI smoke tests, future automation) —
  prefer launching the venv's own interpreter directly over wrapping every launch in `uv run` when a
  clean, reliable stop is required.

## 6. Whether a v0.5.1 code fix is needed

**No.** No confirmed, reproducible bug was found. The one ambiguous finding (§3) is explicitly
inconclusive per this pass's own classification rule, and the existing shutdown code already matches
the standard, correct pattern for the mechanism it uses.

## 7. PR #88 follow-up — real manual Ctrl+C test, and a correction

A maintainer performed the real human-attended interactive terminal test §5 asked for:

- Command: `uv run python -m backend.cli workbench serve`.
- Server printed `http://127.0.0.1:8765/?token=...`.
- Pressed **Ctrl+C once** in the real attached terminal — **the prompt returned**, exactly as
  expected for a clean shutdown.
- `netstat -ano | findstr :8765` afterward showed:
  ```
  TCP    0.0.0.0:8765    0.0.0.0:0    LISTENING    2820
  ```

**Investigation.** PID 2820 was still live when this was investigated. It was fingerprinted directly:

- `curl -sv http://127.0.0.1:8765/` returned `HTTP/1.0 404 Not Found` with header
  `Server: PhoneScriptRunner/1.0 Python/3.12.7` and a plain-text body `not_found`.
- vibe-council's Workbench panel **never** sets a custom `Server` header (Python's default
  `BaseHTTPRequestHandler` banner would read `BaseHTTP/... Python/3.10.x` — this repo's venv is
  Python 3.10.6, not 3.12.7), and its own 404 handler returns **JSON** `{"error": "not found"}`, never
  the plain-text `not_found` body observed.
- `GET /api/state` on the same PID returned the identical generic `not_found` response — again not
  the panel's JSON state shape.
- A source-code re-check confirmed `workbench_panel.serve()` **hardcodes** `host="127.0.0.1"` with no
  CLI flag to override it, and `make_server()` raises `ValueError` for any host outside
  `("127.0.0.1", "localhost", "::1")` — there was already no code path that could produce a `0.0.0.0`
  bind through `vibe workbench serve` as shipped.
- A fresh, independent re-run (`uv run python -m backend.cli workbench serve --port 18765`, a port
  guaranteed not to collide with anything else) bound to **`127.0.0.1:18765`** correctly, and stopping
  it released the port cleanly with no leftover process.

**Conclusion:** PID 2820 (`PhoneScriptRunner`) is an **unrelated, pre-existing local service on the
tester's machine** that happened to occupy port 8765 — not vibe-council's Workbench panel. Combined
with the maintainer's own observation that "the prompt returned" immediately after Ctrl+C (itself
direct evidence the foreground process exited cleanly), **the original §3 "inconclusive" classification
is corrected**: there is no confirmed 0.0.0.0-bind bug, and no confirmed shutdown bug from this
manual test either. The apparent leftover listener was a coincidental, unrelated process on the same
default port.

**Hardening added anyway (PR #88), as defense-in-depth, not as a fix for a confirmed bug:**
- `make_server()` now re-checks the **actual bound address** after `bind()` (not just the requested
  host string) and refuses/closes if it isn't a genuine loopback address.
- `effective_bind_host()` — a single, explicit place to read the real bound host from, used by both
  the startup banner and tests.
- `_startup_lines()` — the printed URL/token banner is now a pure, testable function built from the
  server's *actual* bound address, so the printed URL can never drift from what's really bound.
- A new regression test simulates Ctrl+C (`KeyboardInterrupt` raised from `serve_forever()`) without
  blocking or spawning a thread, and asserts `server_close()` actually closes the socket
  (`httpd.socket.fileno() == -1`) — locking in that *our* cleanup code runs correctly, independent of
  whatever OS/console mechanism delivers the interrupt.
- No change to the blocking `serve_forever()` loop itself, and no new CLI flag — neither was
  warranted given no confirmed bug.
