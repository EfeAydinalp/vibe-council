# v0.4 starts as read-only MCP / Claude Code workflow

## Context

v0.3.0 shipped the local-first decision-memory / context loop; v0.3.1 hardened it and
release-link hygiene is complete. The next milestone is making that curated memory queryable by
Claude Code and local agents. The risk to manage is **authority**: an MCP surface could easily grow
write/action power. This record scopes v0.4 deliberately. The full plan is in
[v0.4 read-only MCP workflow](../plans/v0.4-read-only-mcp-workflow.md).

## Decision

**v0.4 starts with a read-only MCP surface only.**

- Expose **curated decisions**, **project status**, **rejected alternatives**, **constraints**,
  **next actions**, **release/docs metadata**, and the **generated context pack** over a **local,
  read-only** MCP interface (resources like `vibe://status`, `vibe://decisions/{id}`,
  `vibe://context/latest`; read-only tools like `list_decisions` / `show_decision` /
  `get_context_pack` / `check_context_health`).
- **Write-capable MCP and remote approval transport are explicitly deferred** — no
  `promote_decision`, `write_file`, `edit_file`, `delete_file`, `run_command`, `git_*`,
  `send_email`, `deploy`, or anything that mutates repo/file/system state. A test asserts the server
  advertises none of these.
- **Curated `docs/decisions/*.md` and `STATUS.md` remain the source of truth**; promotion stays a
  human-reviewed CLI step. MCP never writes them.
- **Generated/local/private artifacts are excluded by default** — generated packs are served only
  locally; raw `.council/` outputs, untracked private plans, `.env`, `.venv/`, `data/`, `.obsidian/`,
  and secrets are never exposed. Any broader exposure needs an explicit, documented opt-in.
- **Claude Code is the first target integration**; a generic local agent is the secondary target.
- Sequenced as small PRs (#55 design/help → #56 server skeleton → #57 pack + health → #58 Claude
  Code docs → #59 dogfood → #60 release prep). No code/MCP server in the planning PR.

## Rationale

- Read-only first captures most of the value (agents answering from real project memory) with the
  least authority risk, and keeps the local-first / no-surprise-action posture intact.
- Keeping promotion human-reviewed and the curated docs as the only write surface preserves the
  v0.3 trust model.
- A forbidden-tool contract enforced by a test makes the read-only boundary auditable, not aspirational.

## Alternatives considered

- **Ship read + write MCP together** — rejected; write/action authority over MCP is the highest-risk
  surface and deserves its own design after read-only proves useful.
- **Skip MCP, keep paste-only `context export`** — rejected; the export works but a read-only server
  removes manual copy-paste and is the natural next step.
- **Add vector retrieval / LLM eval alongside MCP** — rejected; deferred and prerequisite-gated;
  v0.4 stays deterministic.
- **Hosted/remote MCP transport** — rejected; local-first only; hosted layer stays behind the
  license/provenance gate and validated demand.

## Consequences

- v0.4 is a bounded, read-only access layer over existing curated/generated memory; no new authority.
- Write-capable MCP, remote approval, rolling summaries, token-aware budget, vector retrieval, LLM
  eval, and hosted/sync remain deferred.
- License/provenance remains the commercial "Question 0".

## Next actions

- PR #55: MCP design skeleton (docs + command help, no server).
- Then PRs #56–#60 per the plan; tag/release as a separate, explicitly-requested step.
