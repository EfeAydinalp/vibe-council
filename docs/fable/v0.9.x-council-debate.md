# v0.9.x — Council direction debate

> **Planning document. No code, no behavior change, no version bump, no tag/release.** This file
> captures the **council-led** evaluation of the whole v0.9.x line: candidate directions, arguments
> for/against, risks, sequencing, and what remains for the Fable architecture step to decide. It is
> the **input** to the first council review pass (§"Council review mechanism"); the council's criticism
> is then folded back in below (§"Council review findings") and drives
> [`v0.9.x-phase-brief.md`](v0.9.x-phase-brief.md). **Fable is not used yet** — this is the council
> planning step that precedes the Fable architecture pass.

## Council review mechanism (how "the council" is used here)

vibe-council's multi-model mechanisms are `vibe review` (multiple model critiques → consolidated
review), `vibe full` (collect → peer ranking → chairman synthesis), and `vibe mini`. Per the task and
the project's review policy, this planning round uses **`vibe review --preset balanced`** as the
council debate/review mechanism — a genuine **multi-model** critique consolidated into one review.
`vibe full` is the fuller debate form but is **deliberately avoided** for the debate/brief passes
because of the documented None-content ranking-parser fragility (see `CLAUDE.md` "Known issues");
`review --preset balanced` is the safe, faithful multi-model choice for architecture/roadmap questions.
Two passes are run: one over **this options file**, and one over the resulting
[`v0.9.x-phase-brief.md`](v0.9.x-phase-brief.md). Findings are folded back in below and into the brief.

  (`vibe init-agent` launcher: read-only report + guarded append; localhost-only guard), **v0.8.1**
  (vault polish: capped [`RELEASES.md`](../context/project/RELEASES.md) + STATUS-trimming workflow),
  **v0.8.2** (tighten-only **preference schema v1** + a **read-only doctor validator**). Repo reports
  `0.8.2`.
- **The v0.8.2 substrate (what v0.9.x builds on):** a normative
  [`preference-schema-v1.md`](preference-schema-v1.md) and a single bounded (`≤ 4096-byte`) fenced
  `json` block in [`PREFERENCES.md`](../context/project/PREFERENCES.md) with `schema: 1` and exactly
  four **tighten-only** keys — `default_review_preset` (`cheap|balanced|full` floor, never `premium`),
  `extra_sensitive_paths`, `never_stage_extra` (additive relative-path constraints), `require_usage_flag`
  (warn-when-absent bool). A pure [`backend/preferences.py`](../../backend/preferences.py) validates the

  doctor` as `Preferences (machine-readable, advisory):`. **Nothing applies the block.**
- **Deliberately NOT built (the deferred set):** any preference **application** (a parsed value
  influencing a command); named **council personas** as behavior; a `.council/profile.*` local store; a
  preference-aware `vibe guide`/`context export` (those stay **pointer-only**); UI/dashboard controls;
  mobile/LAN/voice.
- **Personas are already documented as the future direction** ([`preference-schema-v1.md`](preference-schema-v1.md)
  §6): council-member review lenses — **Cost Skeptic, Security Guardian, Product Strategist, Local-first
  Guardian, UX/User Advocate, Risk Officer, Commercialization Lens** — to arrive as **curated presets of
  the tighten-only schema values plus advisory review-emphasis prose**, *"an instance of this model, not
  a new, broader mechanism,"* explicitly **never** able to *"suppress another persona's dissent, override
  the review policy, loosen a gate, or change trust/executor behavior."* v0.9.x must honor that framing.
- **Binding invariants** ([`03-security-invariants.md`](03-security-invariants.md)): approval separate
  from execution; the deterministic trust boundary is the authority; no auto-execution; localhost-only;
  no arbitrary shell; no allowlist growth; **personalization tighten-only (#17)**. These bound every

- **Model policy:** the council plans (this step); **Fable is the architect for the version-line

## The central question of v0.9.x

The whole v0.8.x line was built on one ordering decision: **model-and-validate first, apply later**.
v0.9.x is the "later." So the real debate is **not** *"should we do personalization?"* but **"what is
the *first* thing that is safe to actually *apply*, and does persona behavior belong in that first
step or one step behind it?"** The task is explicit: **do not blindly assume persona behavior should be
applied immediately** — compare alternatives and risks.

Two axes structure the candidates:

- **Axis 1 — what gets applied:** the four proven schema keys (add-friction-only, mechanically
  provable) vs. **personas** (a new prompt-emphasis surface with a unique failure mode: biasing or
  suppressing dissent).
- **Axis 2 — how far it goes:** docs + validate only (the v0.8.2 posture, repeated for a new object) vs.
  bounded application vs. broad application/UX.

## Candidate directions

Each candidate is scored on **product value**, **architecture risk**, **security/trust risk**,
**local-first fit**, and **prerequisite readiness**.

### A. Apply the four proven schema keys (the "cash in the substrate" direction)
Let the *already-validated, mechanically-tighten-only* v0.8.2 keys finally influence behavior, still
additively and reversibly: `default_review_preset` *suggests* a review floor (never lowers an explicit
choice, never `premium`); `require_usage_flag` adds a warning when `--usage` is absent;
`extra_sensitive_paths` / `never_stage_extra` extend the **advisory** doctor dangerous-staged / trust
surface. No personas involved.
- **For:** this is exactly what the substrate was built for; each type already has a per-type
  tighten-only proof and a validator; the blast radius is small and auditable; it delivers the "personal­
  ization is real now" story with the *lowest-risk* object; it keeps the deterministic trust boundary
  authoritative (preferences only *add* friction/warnings, the guard still decides).
- **Against:** modest visible value ("my review floor is balanced now"); it is not the persona story the
  product ultimately wants; risk of scope-creep from "suggest" → "enforce."
- **Value: med · Arch risk: low-med · Security risk: low (add-friction-only, guard still authoritative)
  · Local-first: yes · Prereqs: schema ✓, validator ✓.**

### B. Define + validate named persona presets (docs + validate, NO behavior)
Extend the schema/spec so a **named persona preset** is expressible as a curated bundle of tighten-only
values + advisory review-emphasis prose (e.g. `Security Guardian` = `default_review_preset: full` +
`extra_sensitive_paths: [...]` + an emphasis note). Doctor validates persona-preset docs read-only.
**No prompt influence, no council behavior.** This is the v0.8.2 move repeated for a new object.
- **For:** faithfully continues "model-and-validate first"; gives the persona vocabulary a *tested*
  home before any application; zero trust surface; lets the maintainer curate/audit the seven packs as
  data; a clean base for a later bounded application.
- **Against:** "another docs+validate release" may feel like treading water; presets that never do
  anything are hard to motivate/test for real value; risk of designing the schema wrong without the
  feedback of one real use.
- **Value: med (foundational) · Arch risk: low · Security risk: low · Local-first: yes · Prereqs:
  schema ✓.**

### C. First bounded persona *application* in council prompt construction
Let a **selected** persona preset **append advisory emphasis** to the *collect-stage* prompt of council
review/diff — strictly **additive** to the base rubric, **feature-flagged default-off**, and bound by a
hard rule: a persona may only **add** a lens to its own seat's prompt; it may **never** remove/reweight
the base rubric, **never** touch the peer-ranking or chairman-synthesis stages, and **never**
hide/downrank/suppress another seat's dissent.
- **For:** the first *real* persona value (a Security-Guardian-flavored review); if bounded to additive
  collect-stage emphasis it is arguably still "tighten-only" (more scrutiny, never less); high product
  narrative.
- **Against:** this is **application of prompt-shaping behavior** — a genuinely new trust-adjacent
  surface. The unique risk is **dissent suppression / viewpoint bias**: a persona that reframes the
  question can crowd out other lenses even without touching ranking. Proving "adds emphasis, never
  suppresses" is harder than the mechanical add-friction proof for the four keys. Interacts with the
  `full`-mode ranking/chairman path (a known-fragile area).
- **Value: high · Arch risk: med · Security risk: med (viewpoint-bias/dissent-suppression is the new
  class) · Local-first: yes · Prereqs: B (defined presets) + a dissent-preservation test harness.**

### D. Decision-workspace / session files (substrate, not personas)
Start a persistent, local, gitignored **session / decision-workspace** substrate (durable
`plan.md`/decision drafts, session continuity) before any persona application.
- **For:** addresses real dogfa­­ood friction (session continuity); orthogonal to the persona risk; could
  make persona application easier to observe/audit later.
- **Against:** not the stated v0.9.x thrust (personas); risks reopening the pack-budget / artifact-store
  discipline; "workspace" can balloon; may overlap the deferred `.council/profile.*` question.
- **Value: med · Arch risk: med · Security risk: low-med (new local store surface) · Local-first: yes ·
  Prereqs: none hard, but scope discipline needed.**

### E. `.council/profile.*` local store
A machine-local, gitignored profile store (the long-deferred item) so preferences/personas can live
per-user outside the committed `PREFERENCES.md`.
- **For:** the "real" personalization home; separates private per-user tuning from the committed vault.
- **Against:** the redaction rule already promotes `local-profile-path` WARNING→CRITICAL *when a store
  ships*; it is a new leak surface and a new parse surface; the committed-`PREFERENCES.md` block already
  covers the project-level need; every prior line deliberately deferred it. Strongly premature before
  any application design exists.
- **Value: low-med · Arch risk: med · Security risk: med-high (leak surface) · Local-first: yes but
  risk-adjacent · Prereqs: a separately-approved narrow design.**

### F. UI / dashboard persona controls
A visual/workspace UX for selecting/tuning personas.
- **For:** the eventual product surface; discoverability.
- **Against:** wildly premature — there is no applied persona behavior to control yet; a UI is a new
  surface (and the only local UI, the Workbench panel, is trust-adjacent and frozen-except-bugfixes);
  contradicts local-first minimalism.
- **Value: low now · Arch risk: high · Security risk: med-high · Local-first: risk-adjacent · Prereqs:
  C shipped and proven.**

### G. Workbench / approval UX
Resume the deferred Workbench-proposal UX polish.
- **For:** real dogfood value; the bridge exists.
- **Against:** orthogonal to the persona thrust; trust-boundary adjacency means full/balanced per PR;
  no concrete dogfood pain currently named; better as its own gated slice than folded into a
  persona-themed line.
- **Value: med · Arch risk: med · Security risk: med (trust adjacency) · Local-first: yes · Prereqs:
  none, but competes for focus.**

## Cross-cutting safety analysis (must be explicit)

- **Product value:** the highest *narrative* value is C (a persona actually shaping a review); the
  highest *risk-adjusted* value is **A then B** (apply the proven keys; define/validate personas as
  data) with C as a **gated stretch**.
- **Architecture risk:** A is small and local; B is docs+tests; C touches the council prompt-construction
  path (and is adjacent to the fragile `full` ranking/chairman code); D/E/F add new stores/surfaces.
- **Security / trust risk:** the four keys are *mechanically* add-friction-only (guard still decides);
  personas introduce a **new class**: **viewpoint bias / dissent suppression**, which is not caught by
  the existing tighten-only proof shapes. This is the debate's central safety finding.
- **Local-first constraints:** every candidate except F/E stays local-first with no new surface; E adds a
  local store leak surface; F adds a UI surface. A/B/C add **no** network/hosted/LAN surface.
- **Should named persona presets become behavior yet?** The council's lean: **not as the first step.**
  Define + validate presets (B) first; apply the *proven keys* (A) as the first real application; allow a
  **bounded, additive, dissent-preserving** persona emphasis (C) only as a **full-review, default-off,
  test-gated** stretch — never the unconditional headline.
- **May prompt construction use persona presets safely?** Potentially yes, but **only** under a hard,
  tested envelope: additive to the *collect* stage of the persona's *own* seat; the peer-ranking and
  chairman-synthesis stages are **off-limits** to persona influence; a mandatory test asserts a persona
  can only **add** text and can **never** remove, reorder, reweight, or drop another seat's output.
- **Is schema v1 enough, or does it need extension?** v1's four keys are enough to *apply* (A). Personas
  need a **v1-compatible extension** (a `personas`/preset vocabulary) — but per the spec's own rule,
  *"schema v2 arrives via its own reviewed spec release"*; whether persona presets are `schema: 1`
  additive or a `schema: 2` is an open question for Fable.
- **Can persona presets hide/suppress dissent, and how to prevent it?** They must not. Prevention:
  (1) presets are **additive-only** by construction (bundle tighten-only values + append-only emphasis);
  (2) persona text touches only the collect stage of its own seat; (3) a **dissent-preservation test**
  is a merge gate for any persona-application PR; (4) the chairman/ranking stages remain persona-blind.
- **Guide/context export pointer-only?** **Yes — keep it.** Every prior line held this; v0.9.x has no
  reason to inline persona content into guide/export. Pointer-only forever.
- **`.council/profile.*` deferred?** **Yes — keep deferred** unless Fable designs a narrow, safe,
  read-only approach; the committed block covers the need and a store is a new leak surface.
- **UI/dashboard wait?** **Yes — defer.** No applied behavior exists to control; the only local UI is
  trust-adjacent and frozen.

## Provisional lean (to be tested by the council review, then finalized in the brief)

- **v0.9.0 — apply the four proven keys (A):** the add-friction-only application the substrate was built
  for. Additive, reversible, guard-authoritative. **balanced** review (it is the first application, but
  of mechanically-proven types).
- **v0.9.1 — define + validate named persona presets (B):** persona presets as curated bundles of
  tighten-only values + advisory emphasis, validated read-only in doctor. **Docs + tests only.**
  **balanced** (defines new trust-adjacent vocabulary).
- **v0.9.2 — bounded, additive, dissent-preserving persona emphasis in council prompts (C):**
  feature-flagged **default-off**, additive collect-stage only, ranking/chairman persona-blind, with a
  mandatory dissent-preservation test. **FULL** review (defines trust semantics of applied behavior).
- **Deferred (unchanged):** `.council/profile.*` store (E), UI/dashboard (F), Workbench-UX (G),
  mobile/LAN/voice. Guide/context export stay **pointer-only**.

The council must weigh whether C belongs in v0.9.x at all or waits for v0.10.x, and whether v0.9.0
should be A (proven keys) or B (define personas) first.

## What the council agrees on (pre-review author position)

- **No blind persona application.** Personas are not turned into behavior as an unconditional headline.
- **Apply the proven before the novel:** the four mechanically-tighten-only keys are the safest first
  *application*; personas get *defined + validated* before any prompt influence.
- **Dissent preservation is a first-class, tested invariant** the moment any persona touches a prompt.
- **Pointer-only guide/export, deferred profile store, deferred UI** — all preserved.
- **Fable decides sequencing/numbering and the schema-version question**, not this file.

## What remains unresolved (for the council review, then Fable)

1. Does v0.9.x **apply** anything, or repeat "docs+validate" for personas and defer all application to
   v0.10.x?
2. If it applies: **A (proven keys) or C (persona emphasis)** as the headline — and is C in-line or
   deferred?
3. Are persona presets **`schema: 1` additive** or a **`schema: 2`** (own reviewed spec)?
4. The exact **dissent-preservation** guarantee + test shape for any persona-application PR.
5. Whether a **session/decision-workspace** (D) is worth starting in this line or is a separate track.
6. Confirm **`.council/profile.*` (E)**, **UI (F)**, **Workbench-UX (G)** stay deferred.
7. Whether the persona vocabulary risks **implying** a store/UI and how to keep it docs-first.

## Council review findings (folded back after `vibe review --preset balanced`)

The first balanced pass (multi-model critique → consolidated review; provider cost ≈ **$0.21**) was
**more conservative than the author's provisional lean, and its criticism is adopted**. Headline
findings:

1. **Do not ship C (persona prompt influence) in v0.9.x.** The dissent-suppression / viewpoint-bias
   risk is *real, novel, and underspecified*: the existing "tighten-only" proof machinery **does not
   apply** to prompt emphasis, and **"more scrutiny" is not the same as "more constraints"** — a persona
   can reduce effective attention by *reframing* even with every seat present. Shipping C without a
   mature safety framework (formal definition + adversarial tests) is premature.
2. **"Tighten-only for persona emphasis" is a false equivalence.** Flagged as the single most important
   correction: additive-emphasis does *not* inherit the mechanical add-friction proof the four keys
   enjoy. Persona text needs **content-level** safeguards (disallowed patterns — cannot mention other
   personas/ranking/chairman, cannot instruct ignoring safety) and adversarial merge-gate tests, treated
   like threat-model changes, not content edits.
3. **A is the clear win — cash in the substrate.** Applying the four already-proven keys delivers the
   "personalization is real" narrative at minimal risk (validators + per-type proofs exist).
4. **B (define personas *as schema*) is premature without usage data from A.** Designing seven curated
   presets before seeing how users configure the four keys risks the wrong abstraction. If B ships in
   v0.9.x, keep it as **lightweight documentation** (prose review-lens descriptions), **not** schema
   objects or a validator.
5. **A→B→C compresses too much novelty into one line.** Spread it: apply preferences now; describe
   personas; make persona *behavior* a later, deeply-reviewed line.
6. **C is unshippable as sketched** — no override precedence, no observability, no rollback, no success
   criteria, no content-level safeguards. These are prerequisites, not polish.

**Additional alternatives the council raised (recorded for Fable):** B′ — ship *one* hardcoded persona
(e.g. Security Guardian) as a code-level `--persona` collect-stage variant to learn before generalizing;
stricter-C scoping — a version-line invariant that personas may influence **review/analysis prompts
only, never code-generation/execution prompts**; and mandatory per-invocation override flags
(`--no-preferences`, `--ignore-default-preset`) with a binding **"CLI flags always win; preferences emit
findings/suggestions, never decisions; 'suggest' cannot become 'enforce' without a major version +
security review."**

**Resolution adopted (drives the phase brief):**

- **v0.9.0 = A** (apply the four proven keys) — additive, reversible, override-flagged, guard-authoritative.
- **v0.9.1 = persona vocabulary as PURE DOCUMENTATION** (prose lenses, no schema/validator/behavior) +
  a v0.9.0 dogfood/hardening pass.
- **v0.9.2 = release prep.**
- **Deferred to v0.10.x (its own council+Fable line):** persona presets **as schema** (B formalized),
  persona **prompt application** (C, gated behind the full dissent-preservation + content-safeguard
  framework as an explicit prerequisite), session/workspace (D), `.council/profile.*` (E), UI (F),
  Workbench-UX (G). Guide/context export stay **pointer-only**.

The council's net message: **v0.9.x should apply the proven and *describe* the personas — not apply
persona behavior.** That is exactly the caution the planning task demanded, and it is carried into
[`v0.9.x-phase-brief.md`](v0.9.x-phase-brief.md).
