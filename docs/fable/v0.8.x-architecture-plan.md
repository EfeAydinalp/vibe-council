# v0.8.x — Fable architecture plan (version-line)

> **Fable architecture pass — planning only.** This plan consumes the council-produced files
> ([`v0.8.x-council-debate.md`](v0.8.x-council-debate.md),
> [`v0.8.x-phase-brief.md`](v0.8.x-phase-brief.md),
> [`v0.8.x-fable-input.md`](v0.8.x-fable-input.md)), resolves the open questions, and produces the

> behavior change, no version bump, no tag/release.

## 1. Executive decision

**The v0.8.x theme is "Solidify the core, local-first," as the council chose. The pivot is correct
and is preserved — no re-pivot.**

Why the council was right: the product's differentiator is a *guarded, auditable, local* loop.
Preference behavior as a headline would have spent the phase on the one direction with an unsolved
safety design (text → behavior) while the actual adoption bottleneck — session/onboarding friction —
sat unaddressed. The council's ordering (make the core easier to adopt; make personalization
*auditable before executable*) is the correct engineering sequence, and it happens to also be the
correct product sequence: a launcher makes every future feature easier to reach.

**v0.8.x ships:**

- **v0.8.0 — `vibe init-agent`** (the launcher/session headline): a dry-run-first onboarding
  orchestrator over the *existing* guide/doctor machinery, plus a **mandatory localhost-only CI
  guard**.
- **v0.8.1 — vault polish**: `vibe summarize-history` (local digest) + a capped, committed
  `RELEASES.md` pointer file, under the existing pack budget.
- **v0.8.2 — the preference-control model**: a normative tighten-only schema (docs) + a **read-only
  validator** folded into `vibe project doctor`. **No application.**

**v0.8.x does not ship:** preference application of any kind (including the second council pass's
"refusal filter" pilot — rejected below), named profiles, a `.council/profile.*` store, Workbench UX
changes (deferred out; bugfixes only), local notifications, anything non-localhost, any new
dependency.

## 3. Resolving the council's open questions

**Q1 — What exactly is the preference-control model?**
A **closed, enumerated, versioned JSON schema** carried in a single optional fenced code block
inside the existing `docs/context/project/PREFERENCES.md` (Markdown stays the source of truth; the
block is the machine-checkable region). v1 has a `schema: 1` field and exactly **four** preference
types, each anchored in a concrete v0.9.x behavior and each tighten-only **by construction**:

| Key | Type | Tighten-only proof shape | Anchored v0.9.x behavior |
|---|---|---|---|
| `default_review_preset` | enum `cheap\|balanced\|full` | ordered floor-raise: value must be ≥ the project baseline; can never name `premium` (not in the enum) | review/diff *suggest* this preset as the default |
| `extra_sensitive_paths` | array of relative path prefixes | **additive deny** — can only add prefixes to the sensitive set, never remove | trust policy treats them as extra-guarded |
| `never_stage_extra` | array of relative paths | additive no-stage extension — can only grow the never-stage list | doctor's dangerous-staged check also flags these |
| `require_usage_flag` | bool where only `true` is meaningful | adds a warning when absent; `false` is identical to unset (cannot disable anything) | commands warn when `--usage` is missing |

Every v1 type is either an **ordered floor-raise** or an **additive constraint** — the two shapes for
which "every legal value can only add friction/checks" is provable per type, mechanically, in tests.
Unknown keys, wrong types, absolute paths, or `..` segments are **validation errors**. No type in v1
can name a command to run, a path to allow, or a check to skip — the schema has no vocabulary for
loosening.

**Q2 — Any preference-aware behavior in v0.8.x?**
**No. Validators/docs/tests only.** The second council pass floated a formally-trivial "refusal
filter" pilot; **rejected for v0.8.x**: it contradicts the first pass's ruling, the wiring it would
"prove" is exactly the wiring v0.9.x must design deliberately (with the schema in hand), and a
disable-a-command filter interacts with the guide/doctor surface in ways that deserve their own
reviewed design, not a pilot smuggled into a validation phase. The clean boundary — *v0.8.x validates,
v0.9.x applies* — is worth more than an early wiring proof.

**Q3 — Validator format/output?**
**Folded into `vibe project doctor`** as a new advisory section, `Preferences (machine-readable,
advisory):` — **not** a new `vibe config doctor` command. Rationale: zero new command surface, the
doctor already owns advisory scaffold reporting (v0.7.1 PR 2), and "auditable before executable" is a
readiness property, which is the doctor's job. Output: per-finding `[ok ]`/`[warn]` lines + one
summary line; **exit code unchanged** (advisory — doctor failure remains reserved for required docs +
dangerous staged files). A missing block → one informational line ("no machine-readable preferences —
prose-only PREFERENCES.md is fine"). A malformed block → `[warn]` with a next step, never a crash,
never a failure.

**Q4 — Parser hardening / untrusted input?**
The block is treated as **untrusted input**: extracted by exact fence markers (```` ```json ````
… ```` ``` ````, first block only, at most one allowed); **≤ 4096 bytes** (over-budget → warn,
ignored); parsed with stdlib `json.loads` only (no hooks, no eval, no YAML, no TOML); top level must
be an object; **key allowlist** (schema v1 keys only); per-key strict type/value checks;
relative-path checks reject absolute paths, drive letters, and `..`; **empty arrays warn**
(`extra_sensitive_paths: []` is stated-but-inert — probably a mistake); an **unknown `schema`
version → warn + ignore the whole block** (schema v2, if ever, arrives via its own reviewed spec
release — no silent coexistence/migration); **file-level hardening:** before reading, resolve
`PREFERENCES.md` to its **real path** and require it to reside **inside the project root** — a
symlinked/out-of-root file → one generic warn that leaks no content or target path;
`UnicodeDecodeError`/unreadable file → clean warn, never a traceback. Any anomaly →
**fail-closed to "invalid, ignored"** (a warn + no parsed result). The validator never writes, never
reads `.council/profile.*`, and cannot execute anything by construction.

*Why JSON and not vendored TOML (`tomli`)* — made explicit rather than resolved-by-floor: the
`>=3.10` floor lacks stdlib `tomllib`, and the project's binding default is **stdlib-only / no new
runtime dependency** (stated in `PREFERENCES.md`); vendoring `tomli` is a dependency by another
name. Independently, JSON's strictness is a *feature* for an untrusted-input validator: one tiny,
battle-tested stdlib parser, no anchors/multi-doc/implicit typing. Comment ergonomics are handled by
the carrier (the block sits inside prose Markdown — commentary lives around it).

**Q5 — Guide/context export pointer-only?**
**Yes — and untouched in v0.8.x.** The v0.7.1 PR 3 invariant tests lock those sections byte-stable;
v0.8.x does not modify them at all (not even to mention the JSON block). Pointer-only means what the
brief defined: a path/reference the reader follows; never content ingestion.

**Q6 — `.council/profile.*` store?**
**Deferred (v0.9.x at the earliest)**, and only behind its own design + **full** review. When it
ships, the v0.7.1 redaction rule promotes WARNING → CRITICAL per its documented path. No narrow
exception is designed in v0.8.x — the schema block in committed `PREFERENCES.md` covers the
project-level need without a machine-local store.

**Q7 — Named hardcoded profiles?**
**Deferred to v0.9.x.** Selecting a profile that changes behavior *is* application; and as pure
validation targets they add nothing the schema doesn't already give. When v0.9.x designs application,
named profiles become curated *presets of schema values* — an instance of the model, not a separate
mechanism. That resolves the council's open sub-question.

**Q8 — Workbench UX?**
**Deferred out of v0.8.x; bugfixes only.** Rationale: it is the only candidate with trust-boundary
adjacency, the review cost is full/balanced per PR, and no concrete dogfood pain is currently named.
The v0.8.2 slot the council reserved stays reserved: if the maintainer pulls a Workbench-UX PR into
the line, it enters as its own **full-review** slice with invariant re-verification — but the default
plan ships without it.

**Q9 — Launcher/session command surface?**
**`vibe init-agent [--agent {claude|codex|fable}]... [--role <role>] [--write] [--yes]`** — a thin,
dry-run-first orchestrator over existing machinery:

- **Default (no `--write`): read-only report.** Doctor-style summary (vault/scaffold state via the
  existing checks) + exactly what `--write` *would* append and to which files. Writes nothing.
  This mirrors the executor's dry-run-first precedent.
- **`--write`: appends** the selected agents' guide sections via the **existing** append/marker-skip
  machinery to the **fixed per-topic default files** (`CLAUDE.md`/`AGENTS.md`/`FABLE.md` at the
  project root). **No user-supplied target path in v1** — that is the path-safety guarantee: with no
  path argument there is no traversal/injection surface at all. Re-runs are idempotent
  (marker-skip). `--yes` required in non-interactive use, matching house convention.
- **It does not**: create `.council/`, seed/overwrite vault files (missing vault → report + pointer
  to `docs/context/project/README.md`), touch the network, call a model, or accept arbitrary paths.
- `vibe guide`/`vibe context export` remain independent and unchanged; `init-agent` composes them.

**Q10 — Local same-machine notifications?**
**Deferred.** It belongs with the mobile/LAN/voice threat-model line (a bell is the thin end of a
notification transport); v0.8.x gains nothing from it that `workbench serve`'s visible queue doesn't
already give a single-operator workflow.

**Q11 — The "do not implement this slice" stop condition?**
A slice is **not implemented** (halted, brought back to planning) if, during implementation, any of
the following becomes true: it needs a **new dependency**; it needs a **new network surface** or
non-loopback bind; it must **modify trust/executor/importer/panel semantics** to work; its diff
**grows beyond its stated files** in a way review can't absorb; the validator (v0.8.2) turns out to
need **behavior** to be useful; or the launcher needs an **arbitrary path/command argument**. Any of
those is a design error in this plan — stop, report, re-plan; do not push through.

## 4. Architecture boundaries

- **Launcher/session boundary:** `init-agent` is a *composer* of existing generators. It may read
  repo state and append to the three fixed root guide files (with `--write` + confirmation), nothing
  else. It never executes, never networks, never touches `.council/`.
- **Project-vault boundary:** the vault stays curated committed Markdown; `summarize-history` writes
  its digest **local/gitignored by default** (committed only by explicit opt-in flag mirroring
  `context build --allow-docs` convention); `RELEASES.md` is a capped pointer file (one line per
  release), never a changelog duplicate. The pack never ingests the vault (21/21 is a hard wall).
- **Preference-model boundary:** the schema lives in committed `PREFERENCES.md` (single fenced JSON
  block); the only consumer in v0.8.x is the doctor's read-only validator. Nothing else may read the
  block until v0.9.x's reviewed application design.
- **Guide/context-export boundary:** byte-stable in v0.8.x; pointer-only forever (path/reference,
  never content ingestion); never read `.council/profile.*`.
- **Workbench-UX boundary:** frozen (bugfixes only) unless a slice is explicitly pulled in at full
  review with invariant re-verification.
- **Doctor/redaction boundary:** doctor stays read-only/advisory-vs-fail exactly as today (the new
  preferences section is advisory); redaction rules unchanged in v0.8.x (the `local-profile-path`
  promotion trigger is a store shipping, which v0.8.x does not do).
- **No-stage / local-private boundary:** unchanged — `.council/**` (including `profile.*`),
  runtime/payloads, generated packs/exports/digests, private plans, `.env`/`.venv/`/`data/`, secrets
  never staged; the two private plan files stay untracked.

## 5. Safety invariants (bind every v0.8.x PR)

All of [`03-security-invariants.md`](03-security-invariants.md), plus phase-specific:

- **Local-first:** nothing binds beyond `127.0.0.1`; a CI/test guard asserts it (v0.8.0 deliverable).
- **No arbitrary shell; no auto-execution; no Workbench trust-boundary loosening; no command
  allowlist expansion** without its own explicitly-scoped full review.
- **No hosted/SaaS, no vector DB, no PostgreSQL/Redis/Docker/Next.js/LangGraph, no secrets storage,
  no new runtime dependency** (stdlib-only default; exception needs reason + security review +
  pinning).
- **No preference parser that turns text into actions** — the v0.8.2 validator parses *into a
  report*, never into behavior; no code path may route a parsed preference into a command's logic
  in this line.
- **No `.council/profile.*` local store** (separately approved design only); export/guide **never
  inline profile content** — *pointer-only = path/reference/advice only, not content ingestion*.
  Concretely: `` `docs/context/project/PREFERENCES.md` — review-preset policy `` is a pointer ✓;
  "read `PROFILE.md` before coding" is a pointer ✓; quoting a preference's *value* ("this project's
  default preset is balanced") into generated output is **content ingestion** ✗ — even one field.
- **Preferences can tighten guidance only** — never loosen safety/security/no-stage/trust rules; the
  schema is constructed so loosening is inexpressible (§3 Q1).
- **Workbench invariants untouched:** browser/client never sends executable payload/argv;
  server-side ids/hashes stay server-side; raw payload lives only in local payload artifacts.

## 9. Fable-specific risk catches

- **Launcher → arbitrary shell creep:** the moment `init-agent` grows a "run setup command" or a
  free path/target argument, it is a different, dangerous feature. v1 has **no path and no exec
  vocabulary**; keep it that way (PR 1/2 stop conditions).
- **Preference model → hidden behavior:** the subtle failure is a helper that "just returns the
  parsed preset" and gets imported somewhere convenient. PR 8's **findings-only public API** (plus
  its test) is the structural block; report any import of `backend/preferences.py` outside the
  doctor as a review flag.
- **Guide/export accidentally inlining preferences:** the v0.7.1 byte-stability tests are the
  tripwire — if a v0.8.x diff needs to edit those tests, the diff is wrong.
- **Context-pack bloat:** the digest/RELEASES work exists to *reduce* pack pressure; if any v0.8.x
  change requires a new pack fallback step to stay 21/21, stop (that is the v0.5-era cliff
  returning).
- **Named profiles as policy override:** deferred, but watch for it re-entering as "just a
  convenience flag" on the validator — a profile *selector* is application; it waits for v0.9.x.
