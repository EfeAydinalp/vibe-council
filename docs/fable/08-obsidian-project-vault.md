# 08 — Obsidian-like project vault (v0.6.2)

## Goal

A durable, human-readable and agent-readable local project knowledge system, so agents don't start
from zero each session and users don't re-explain history. Think Obsidian-style Markdown notes —
but **committed, curated, and budget-aware**, not a sprawling personal wiki.

## Folder strategy: extend `docs/context/project/`, do NOT invent `.vibe/`

vibe-council already has `docs/context/project/STATUS.md` and `docs/context/agent-brief.md`, and
`vibe context build` consumes them. **Extend that.** A new top-level `.vibe/` competes with
`docs/context/` and splits the source of truth — that is the mistake to avoid. One canonical
location wins.

Recommended layout (committed unless noted):

```
docs/context/
  agent-brief.md          # narrative brief for agents (exists)
  project/
    STATUS.md             # current state, present tense only (exists)
    ROADMAP.md            # forward plan (mirror of docs/fable/04-roadmap.md's spirit, concise)
    RISKS.md              # known risks / gotchas
    WORKFLOWS.md          # how we actually work (review policy, PR flow, verification)
    AGENTS.md             # agent roles + rules (generated by `vibe guide --write`)
    RELEASES.md           # one line per release (or a pointer to CHANGELOG.md)
    DECISIONS.md          # INDEX/pointer into docs/decisions/*.md — never restate them
    IMPLEMENTATION_NOTES.md  # durable "why it's built this way" notes (curated, small)
```

## Committed vs. local-only

- **Committed, hand-maintained:** STATUS, ROADMAP, RISKS, WORKFLOWS.
- **Committed, generated (via `--write`):** AGENTS.md / CLAUDE.md / `.codex/instructions.md`.
- **Committed, pointer-only:** DECISIONS.md (index into `docs/decisions/`), RELEASES.md (or a
  pointer to CHANGELOG.md).
- **Local-only / gitignored:** context packs (`.council/context/`), history-summary scratch,
  draft decisions, any raw/generated artifact. Never commit generated packs/exports.

## Generated vs. hand-maintained

- Hand-maintained files are the **source**. Generated files (packs, exports, summaries) are
  **projections** of the source and stay local.
- Do **not** duplicate `docs/decisions/*.md` into the vault — DECISIONS.md points at the canonical
  store. Duplication rots.

## Summarizing history without bloat

The context pack is **already at budget** (trims to ~14000 chars with fallback steps to stay 21/21).
That is the binding constraint. Rules:

- `STATUS.md` = **present only**. Move history out of it.
- History lives in a **capped, append-then-summarize `RELEASES.md`** (one line per release) or points
  at CHANGELOG.md rather than duplicating it.
- A future `vibe summarize-history` ([09-cross-project-onboarding.md](09-cross-project-onboarding.md))
  compresses old status into a short digest, so the pack pulls the digest, not raw history.
- Keep each committed vault file small (the ~4–5 KB decision-record norm is a good yardstick).

## Connection to `vibe context build`

- `context build` gains the new committed vault files as inputs, under the **existing budget
  discipline**. Do not reopen the 21/21 fight — if adding a file pushes the pack over budget, the
  file is too big or belongs behind a pointer, not in the pack.
- The vault's size must never become the pack's size. The pack is a budgeted selection, not a dump.

## Connection to Workbench proposals

- A proposal's `summary` (and an agent's session context) can reference vault sections as human
  context. The vault is **read-only** to the bridge — a proposal never writes the vault as a side
  effect of import.
- The vault is where an agent reads "what is this project, what's decided, what's the current focus"
  before proposing, so proposals arrive already aligned with the roadmap and invariants.

## How this helps Fable / Claude continue across sessions

A new session reads the vault (via the context pack, the agent-brief, and — if generated — AGENTS.md)
and immediately knows the product, the invariants, the current focus, and the review policy. The
onboarding cost drops from "re-explain everything" to "read the committed vault." That is the entire
point of the phase.

## Non-goals (v0.6.2)

- No `.vibe/` folder competing with `docs/context/`. No context-pack bloat / reopening 21/21. No
  duplicating `docs/decisions/`. No committing generated packs/exports. No auto-writing the vault from
  a proposal import.
