## Version metadata: where it lives, what changes

A grep for version strings across the repo (excluding `.council/`) finds three real
package/tool sources plus many doc/historical mentions:

| Location | Current | Action |
|----------|---------|--------|
| `backend/__init__.py` → `__version__` | `0.1.0-dev` | **→ `0.1.0`** (the one real change) |
| `pyproject.toml` → `version` | `0.1.0` | already correct — no change |
| `uv.lock` → `llm-council` package `version` | `0.1.0` | already correct (mirrors pyproject) — no change |

`backend/cli.py` reads `__version__` from `backend/__init__.py`, so bumping that single
constant makes `vibe --version` print `vibe-council 0.1.0`. After the bump, **all three
sources agree at `0.1.0`** (the checklist's "two separate strings must agree" gate). The
`uv.lock` entry mirrors `pyproject.toml`; `uv sync --locked` (or a quick lockfile check)
confirms they stay consistent.

## Tag strategy (after merge — documented, not executed here)

**Verify *before* tagging** (council: don't make a broken build public, then check). The
tag is only created once verification on the merge commit passes:

1. Merge this PR to `master` (only with **all CI green**).
2. **Verify the merge commit first** — fresh clone of `master`, install, then confirm the
   **minimal smoke criteria** below all pass.
3. Only if step 2 passes: `git tag -a v0.1.0 -m "vibe-council v0.1.0"` on the merge commit.
4. `git push origin v0.1.0`.
5. Create the GitHub Release for `v0.1.0`, pasting `docs/releases/v0.1.0.md` /
   the `CHANGELOG.md` `[0.1.0]` section; mark "latest".

**Minimal post-install smoke criteria** (all must succeed, else don't tag / roll back):

- `vibe --version` prints `vibe-council 0.1.0`.
- `vibe status` runs without error.
- One real operation works — e.g. `vibe extract --preset cheap --file <plan> --usage`
  (cheapest model step), confirming the council path and `--usage` reporting work.

**If a critical issue surfaces after tagging:** follow the existing
**Rollback / abort procedure** already in `docs/release-checklist.md` (delete the
local+remote tag, draft/delete the Release, fix forward to `v0.1.1`). No new rollback
text is needed — it's already documented.

**No tag and no Release are created in this PR** (explicit rule).
