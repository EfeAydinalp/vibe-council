# Plan: v0.5 — exact allowlisted command execution (design only)

A **design/safety document**, not an implementation. `run_command` real execution has been
fail-closed since the guarded executor began (PR #73–#77): the deterministic trust boundary
(`backend/workbench_trust.py`) already evaluates `run_command` proposals (allowlist + no shell
metacharacters), but `backend/workbench_executor.py` rejects any real `run_command` execution with
`ExecutorError` regardless of what the invariant says. This plan designs the **narrow, optional**
next step — real execution of a small set of **exact, allowlisted, read-only project-verification
commands** — before any code changes it. **No executor, panel, or CLI change is added in this task.**

Review date: **2026-07-02**. Preset policy for any council runs on this doc: **`balanced` only**,
always `--usage`, raw `.council/` output stays local.

> **Scope guard.** This is command execution's **first and narrowest** possible form: verification
> commands only (tests/lints/status/context/MCP-health checks), never a general shell. Anything
> broader is a **stop condition** (§11). License/provenance remains the unresolved **"Question 0"**
> gate; unrelated here.

---

## 1. Purpose

Real `run_command` support should exist **only** to let an approved task run narrow, already-
allowlisted **project verification** commands — the same class of command a human would run by hand
to check their own work: test suites, lints, status/decision checks, context-pack checks, MCP health
checks. It is not a general-purpose command runner. If a proposed command isn't already on the
allowlist, it stays blocked — exactly as today.

## 2. Non-goals (permanent, not just "not yet")

Arbitrary shell · user-provided/free-form shell strings · shell metacharacters (`;`, `&&`, `|`, `>`,
`<`, backticks, `$(...)`) · package install · deploy · `git push`/any git mutation · credential/secret
access · network commands · provider/model/API calls · background daemons/long-running processes ·
destructive commands (delete/format/reset) · recursive/autonomous loops. Several of these are
permanent stop conditions (§11), not staged work for a later PR.

## 3. Allowlist model

**No shell, ever.** Real execution uses `subprocess.run(argv, shell=False, ...)` with a **fixed argv
list**, never a string handed to a shell. This closes the string-vs-argv ambiguity at the source: a
command is never "parsed" into argv at execution time (no `shlex.split` on user/approval-controlled
text) — each allowlist entry is a **pre-defined argv list**, chosen by the executor, keyed by a stable
**command id** the approval/action references.

- **Reuse, don't replace, the existing policy.** `workbench_trust.TrustPolicy.allowed_commands` /
  `is_command_allowed` already normalize a command string, reject any shell metacharacter
  (`_SHELL_META_RE`), and require an **exact** match against a small allowlist:
  ```
  git status --short
  git diff --stat
  git diff --cached --stat
  python -m unittest discover -s tests -t .
  vibe lint --redaction
  vibe decisions lint
  vibe context check
  vibe mcp inspect --context --health
  ```
  (`vibe context build` is a plausible future addition — not added by this doc; a real gap to close
  in the implementation PR, not here.)
- **The string is the label; the argv is looked up, never derived.** Keep the existing normalized
  strings as the human-readable label the trust boundary and panel already show, but resolve **each**
  allowlisted string to a **fixed, pre-built argv list** via a small `Dict[str, List[str]]` lookup —
  never by splitting the label string. E.g. `"vibe lint --redaction"` maps to
  `[sys.executable, "-m", "backend.cli", "lint", "--redaction"]` (see §12 on why not the `vibe`
  launcher script itself). No entry accepts extra/dynamic args; the argv is complete and fixed.
- **No pipes, redirects, glob expansion, or environment-variable interpolation** — impossible by
  construction once there is no shell and no string parsing.

## 4. Execution invariant (extends the existing one, doesn't replace it)

Real `run_command` execution may run only if **every** existing condition
(`workbench_executor.validate_execution_invariant`) holds — approval exists and is `approved`, action
exists and is `pending`, action/approval/task linkage matches, action scope matches the approved
scope, a **fresh** deterministic trust evaluation is not blocked — **plus**, specific to commands:

- the command **exactly** matches an allowlist entry (§3) — normalized-string match, same as today;
- the resolved **argv** for that entry is the fixed, pre-built list — never influenced by the action's
  own text beyond selecting *which* fixed entry to use;
- a **timeout** is set (§5) before the process starts;
- **cwd is the project root** — never a path derived from the action (§7);
- the **environment is sanitized** (§6) before the process starts;
- output capture is **bounded** (§5);
- an **audit/log record** is created for the attempt (§5), matching the file executor's existing
  `result_summary` style;
- as with file actions, **approving never starts a command** — execution is a separate, explicit step
  (unchanged panel/CLI invariant from PR #73–#77).

If any check fails: the action becomes `blocked` (or stays `pending`); no process is ever started;
the reason is visible; there is no partial/leftover process.

## 5. Timeout / output capture (conservative defaults, revisit with real data)

- **Timeout:** 60 seconds initially (`subprocess.run(..., timeout=60)`). On timeout: the process is
  killed (`subprocess.TimeoutExpired` → terminate/kill), the action is marked `failed` (not
  `completed`), and the timeout itself is recorded — never silently retried, never left running.
- **Output cap:** at most 64 KB captured combined stdout+stderr (truncate with a clear marker, e.g.
  `…(truncated, N bytes total)`); never streamed live to the panel in this phase.
- **Redact before storing/showing.** Captured output is run through the existing redaction guard
  (`backend/redaction.py`) before it is written to `result_summary` or shown in the panel/logs — a
  critical finding in output blocks the action (`blocked`, not `completed`) rather than exposing it.
- No output is ever written raw into `docs/` or any committed location — same rule as everywhere else
  in `.council/runtime/`.

## 6. Environment sanitization

The subprocess environment is **not** the caller's inherited `os.environ` by default:

- **no secret/API-key passthrough** (`OPENROUTER_API_KEY` and friends explicitly excluded);
- **no `.env` loading** by the executor (if a subcommand loads its own `.env`, that's that
  subcommand's existing behavior, not something the executor adds);
- a **minimal, explicit** environment: `PATH` (needed to resolve `git`/the Python interpreter),
  standard locale/encoding vars if needed for stable output, nothing else by default;
- **no shell startup files** are ever sourced, because there is no shell (§3).

## 7. Working directory

- **cwd is always the project root** (`Path(project_root)` / `Path.cwd()`, same resolution the file
  executor already uses) — never a path read from the action, approval, or any request.
- No path traversal is possible because cwd is fixed, not derived from input.
- Execution is refused (fail-closed, same as the file executor's path guard) if the resolved project
  root itself is under a denied location — though in practice the project root is always the repo
  root, so this is a defense-in-depth check, not an expected real case.
- Denied to run *from* — matching the file executor's existing denylist, applied to the *targets* any
  allowlisted command could plausibly touch, not to cwd selection itself (there is no cwd selection):
  `.council/`, `.venv/`, `data/`, any cloned/vendored repo, and any other path already denied by
  `TrustPolicy.denied_paths` / `denied_relpaths`.

## 8. Panel behavior (not built in this PR)

The panel must **not** auto-run any command. If/when PR #80/#81 build it:

1. a separate, explicit "Execute" affordance per pending `run_command` action (mirroring the
   PR #77 file-action button — approve and execute stay separate);
2. a **preview** of exactly which allowlisted command will run (the label, e.g. `vibe lint
   --redaction`) and its timeout;
3. an explicit note that **only the exact allowlisted command runs — no arbitrary input is
   accepted**;
4. the **redacted**, possibly-truncated captured output as the result, never raw/unbounded output.

## 9. CLI behavior (not built in this PR)

A plausible future explicit command: `vibe workbench execute <action-id>` (dry-run by default,
`--apply` to really run) — mirroring how file execution was deliberately kept off the CLI surface
until PR #77 proved the panel path first. Not implemented now; may not be needed if the panel path is
sufficient.

## 10. Tests required before any command-execution implementation PR

- an exact allowlisted command succeeds in a temp project (asserts stdout/exit captured, action
  → `completed`);
- a non-allowlisted command is blocked (no process started);
- an allowlisted command with one extra/different argument is blocked (exact match only — no
  "starts with" or prefix matching);
- a command string containing a shell metacharacter is blocked (existing `_SHELL_META_RE`, re-
  verified against the resolved argv path, not just the label string);
- **label-to-argv resolution is deterministic**: the same allowlist label always resolves to the same
  fixed argv, and no code path ever calls `shlex.split`/string-splits the label into argv;
- a command that exceeds the timeout is killed, the action → `failed`, no process left running
  (verified via a deliberately slow temp-project fixture, not a real long-running dependency);
- output over the cap is truncated, not silently dropped without a marker;
- environment sanitization: a canary secret set in the **test's** own `os.environ` is asserted absent
  from the subprocess's environment;
- missing/rejected/held approval blocks execution (no process started) — same invariant as file
  actions;
- action not `pending` (already `completed`/`failed`/`blocked`) blocks re-execution;
- a fresh trust-boundary block (e.g. command removed from the allowlist between approval and
  execution) blocks execution — stale approval/audit cannot authorize, same as file actions;
- cwd is asserted to be the project root regardless of any path-like text in the action;
- **no** provider/model/network call happens (existing test-suite guard pattern: no outbound
  connections);
- **no** package-install/deploy/`git push`-shaped command is ever on the allowlist or reachable;
- **no** background process survives the test (process group / explicit wait+kill verified);
- **no** raw secret value appears in `result_summary`/logs/captured-output display (redaction-guard
  assertion, matching the file executor's existing "no content in logs" test pattern).

## 11. Stop conditions (halt and ask for review / a council pass)

Stop implementation and request review if any of these arises:

- **arbitrary shell** becomes tempting for any reason (e.g. "just this once, allow a raw string");
- **package install or deploy** is requested for the allowlist;
- a **network-touching command** is requested (curl/wget/ping/anything provider-facing);
- the design would need to pass **secrets/credentials** into the subprocess environment;
- command **args become dynamic** (anything beyond selecting *which* fixed argv entry to run);
- a **long-running or background** process is requested (server start, watch mode, daemons);
- captured **output risks leaking secrets** in a way redaction can't reliably catch;
- **Windows/Linux behavior would differ unsafely** (e.g. an allowlist entry that resolves to a shell
  builtin on one OS and a real binary on another).

## 12. Windows / Linux compatibility

The project runs on Windows, macOS, and Linux (CI matrix; `scripts/vibe.ps1|.cmd|.sh` launchers
already differ per OS). Two concrete compatibility choices this design makes to avoid platform drift:

- **Allowlist by command id, not by shell text.** Each allowlisted label (§3) maps to a fixed argv
  *per platform* where needed — in practice almost all entries are OS-agnostic once shell is removed
  (`git status --short` is `["git", "status", "--short"]` everywhere), but Python invocations should
  resolve via `sys.executable` (the interpreter actually running the executor) rather than a bare
  `"python"`/`"python3"` string, which resolves differently — or not at all — depending on PATH/venv
  activation state per OS.
- **Bypass the `vibe` launcher scripts for real execution.** `vibe lint --redaction` is kept as the
  **display label** (matches what a human would type), but its argv resolves to
  `[sys.executable, "-m", "backend.cli", "lint", "--redaction"]` — i.e. the same module invocation
  CLAUDE.md already documents as the canonical way to run the backend — instead of shelling out to
  `vibe.ps1`/`.cmd`/`.sh`, which adds OS-specific indirection with no safety benefit and a real
  portability risk (the launcher might not be installed/on PATH in every environment this executor
  runs in).
- Tests for this feature must **not** rely on Unix-only commands (no bare `ls`, `cat`, POSIX-only
  flags); allowlist entries and their tests should pass identically on Windows and POSIX CI.

## 13. Recommended implementation sequence (after this docs PR)

- **PR #78 (this plan).** Docs-only design. No code.
- **PR #79.** Command allowlist → argv resolution model (`Dict[str, List[str]]`, §3) +
  dry-run/preview only for `run_command` (mirroring PR #73's file dry-run) — still no real execution.
  Proves the label→argv mapping and invariant extension (§4) are correct before anything can run.
- **PR #80.** Real execution for the allowlisted commands only, behind the full invariant (§4),
  timeout/output-capture/redaction (§5), and environment/cwd sanitization (§6–§7) — in a temp/test-
  safe scope first, matching how PR #74 proved bounded file execution before any panel wiring.
- **PR #81.** Panel display for command results (§8) — mirrors PR #77's file-action execute button,
  built only after PR #80 is proven.

(Adjust if a safer ordering emerges; keep each PR tiny and reversible, matching every prior executor
PR in this track.)

## 14. Acceptance criteria for this plan

- The allowlist model removes shell entirely and eliminates string→argv parsing ambiguity (§3).
- The execution invariant is stated as an **extension** of the existing one, not a parallel/duplicate
  check (§4).
- Timeout, output capture, redaction, environment, and cwd are each given conservative, concrete
  defaults (§5–§7).
- Panel/CLI wiring is explicitly deferred, matching the approve/execute separation already
  established (§8–§9).
- A concrete test list (§10) and stop-condition list (§11) exist before any implementation PR.
- Windows/Linux compatibility is addressed concretely, not just noted as a risk (§12).
- No execution, executor, panel, or CLI code changes in this PR.
